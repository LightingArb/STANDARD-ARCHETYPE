## §1 日常知識調用總則

### 1-1. 預設讀取範圍

日常所有指令，預設只可引用以下本地資料：

| 類別 | 來源 |
|------|------|
| 正式產物 | 14/15；20/21/30/31 僅程式設計模式 |
| 中間層 | 10/11/12/13 |
| 參考資料 | 03_參考資料/資料地圖.md、03_參考資料/資料館長.md |
| 角色 | 04_角色/00_角色會議紀錄.md、必要時讀對應角色 md |
| 路由 | 99_基地/系統總覽.md |
| 人格 | 當前容器 02_人格/ 內的所有人格卡 |

### 1-2. 人格讀取層級（寫死）

| 上下文位置 | 讀取 |
|-----------|------|
| 在主專案 | 讀主專案 `02_人格/` |
| 在主專案的探索 | 仍讀主專案 `02_人格/`（探索無獨立人格） |
| 在子專案 | 讀該子專案 `02_人格/` |
| 在子專案的探索 | 仍讀該子專案 `02_人格/` |

人格讀取不得跨層自動混用，除非使用者明確指定。

**Fallback：** 若當前容器 `02_人格/` 為空 → 提示使用者取用人格（指向 Furnace），不自動降級到其他來源。

### 1-3. 禁止直接調用

以下流程可直接讀 Furnace：
- `取用人格`
- `新增專案`（自動配發角色與依賴人格）
- `升級專案`（補配發）
- `新增沙盒`（子專案，自動配發）
- `大絕招`（專家召喚 + Harness）
- L3 客製流程

其餘日常指令不得直接調用：
- Furnace repo 的任何檔案
- Furnace output/raw-cards/ 原始素材
- Furnace 流派工作區（codex/、claudecode/）
- 未上架的鍛造中間產物

若本地資料不足以回答，AI 只能做以下三件事之一：
1. 明確指出缺哪些資料
2. 建議先整理回本地 md
3. 建議使用 `大絕招` 或 `取用人格`

## §2 AI 讀取與判斷規則

### 2-1. 快速接續模式（進入專案預設）

```
1. 靜默讀取 99_基地/系統總覽.md
2. 靜默讀取當前正式容器 14_總方案.md metadata（專案模式）
3. 靜默讀取當前容器 02_人格/ 內所有人格卡
4. 靜默讀取 10_對話整理.md
5. 若 10_對話整理.md 有「快照」區，先讀快照
6. 補讀 04_角色/00_角色會議紀錄.md
7. 用一句話說明現況，建議下一步
```

**Fallback — 自動升級為完整讀取模式：**

觸發條件（任一即升級）：
- 10_對話整理.md 為空白
- 最後更新超過 7 天
- 內容少於 5 行
- 使用者說「重新整理一下現況」

升級後讀取：
- 系統總覽 + 14_總方案.md + 02_人格/
- 20_施工藍圖.md（若程式設計模式且存在）
- 30_施工進度.md（若程式設計模式且存在）
- 13 → 12 → 11 → 10 → 15
- 04_角色/00_角色會議紀錄.md
- 若需要補理解外部資料，再讀 03_參考資料/資料館長.md

### 2-2. 探索版快速接續模式

進入沙盒探索時：

```
1. 靜默讀取系統總覽
2. 靜默讀取所屬正式容器 14_總方案.md metadata（專案模式）
3. 靜默讀取所屬正式容器 02_人格/
4. 靜默讀取當前探索 10_對話整理.md
5. 若有快照先讀快照
6. 補讀所屬正式容器 04_角色/00_角色會議紀錄.md
7. 視需要補讀所屬正式容器 14_總方案.md
8. 一句話說明「目前在 [正式容器] 的 [探索名稱]」+ 現況 + 下一步
```

Fallback 觸發條件與升級內容同主專案版。

### 2-3. 各指令 AI 行為

> 指令清單、用途、觸發條件見 sri-core.md §2。
> 完整初始化與系統更新的步驟見 sri-core.md §2-2 / §2-3。
> 以下只補充 AI 的讀取順序、判斷規則與回應節奏。

#### 新增專案

1. 在 [根目錄]/ 下依下一個編號建立專案資料夾
2. 依模式建立對應模板（見 sri-templates.md 建檔速查表）
3. 程式設計模式先問：「程式資料夾要叫什麼名字？」
4. 建模板 → 從 Furnace 複製角色 → 掃角色依賴 → 複製人格 → 進第一輪討論
5. 同步更新系統總覽狀態區
6. 不先回報建檔細節
7. 直接進入第一輪討論，問使用者第一個真正有用的問題

#### 進入專案

1. 預設走快速接續模式（§2-1）
2. 一句話回報目前狀態（含模式與階段）
3. 建議下一步
4. 自然語句路由：若使用者直接描述意圖（「繼續聊上次的」），視為進入專案 + 對應討論，不要求重打口令

#### 專案說明

1. 定位目標專案（若無上下文先問）
2. 依序讀取：14 → 若程式設計讀 20/30 → 13 → 12 → 10 → 11 → 15 → 04_角色/00 → 視需要讀資料館長
3. 輸出：這個專案在做什麼、整理到哪裡、還缺什麼、建議下一步

#### 升級專案

1. 若已是程式設計 → 直接回報
2. 問使用者程式資料夾名稱
3. 補建結構 → 補入 11-13 → 掃新角色依賴 → 補入缺少人格
4. 同步把 14_總方案.md 的 `專案模式` 改為 `程式設計`
5. 同步更新系統總覽

#### 新增沙盒

1. 只能在已進入某正式容器時使用
2. 詢問類型：探索 / 子專案
3. 一般模式主專案只能開探索
4. 探索 → 建 `08_沙盒區/[字母]_名稱/`（10-13 + 03_參考資料/）
5. 子專案 → 先問程式資料夾名稱 → 建完整結構 → 複製完整角色與依賴人格
6. 建完直接進入第一輪討論

#### 進入沙盒

1. 若未定位正式容器，先問主專案
2. 走探索版快速接續（§2-2）或子專案版快速接續（同 §2-1）
3. 角色讀取仍讀所屬正式容器 04_角色/
4. 人格讀取依層級規則（§1-2）

#### 回到主專案

1. 清除當前沙盒上下文
2. 回到所屬正式容器
3. 不刪除任何沙盒資料

#### 比較所有沙盒

1. 定位當前正式容器
2. 掃描 08_沙盒區/ 下所有子項
3. 主讀各沙盒 13_已確認結論.md
4. 視需要補讀 12、資料館長
5. 輸出跨沙盒比較表：名稱、類型、結論、優點、風險、是否建議回流

#### 回流沙盒到主專案

1. 讀指定沙盒 13_已確認結論.md
2. 列出建議回流項
3. 等使用者確認
4. 寫入所屬正式容器 12_待確認問題.md
5. 不得直接寫入 13/14/20/30

#### 自動回流

1. 讀所有沙盒 13
2. 做比較與篩選
3. 列出建議回流項
4. **必須使用者確認後才寫入**
5. 寫入所屬正式容器 12

#### 升級沙盒（探索 → 子專案）

1. 確認當前主專案為程式設計模式
2. 讀取探索現有內容
3. 分配數字編號（下一個可用）
4. 補齊完整架構，保留已有 10-13 和 03_參考資料/ 內容
5. 更名資料夾：A_名稱 → 01_名稱
6. 使用者確認後執行

#### 降級沙盒（子專案 → 探索）

1. 讀取子專案現有內容
2. 保留 10-13 + 03_參考資料/
3. 其餘檔案移入該子專案 99_舊檔案封存區/，再整體搬到探索結構
4. 分配字母代號（下一個可用）
5. 使用者確認後執行

#### 討論專案

1. 靜默讀取 14 metadata + 02_人格/ + 10 + 04_角色/00
2. AI 自動判斷規劃/架構/施工階段（見 sri-core.md §3-1）
3. 依角色自動觸發規則判斷本輪哪些角色需出聲（§3-3）
4. 依矩陣提問規則發問
5. 不重複問已在 13 的結論
6. 每輪結束更新 04_角色/00；有明確角色觀點同步更新角色 md
7. 若當前在探索，主讀探索 10/11/12/13，角色與人格仍讀所屬正式容器
8. 探索內討論結果不得直接寫到正式容器中間層

#### 角色審查

1. 靜默讀取：14 + 若程式設計讀 20 + 13 + 12 + 02_人格/ + 04_角色/00 + 各角色 md
2. 判斷議題涉及哪些角色
3. 依序以角色格式呈現每個角色完整觀點
4. 全部發言後整合收斂建議
5. 重大分歧明確標示
6. 結束後更新 04_角色/00 + 各角色 md

#### 延伸想法

1. 讀取 10 + 11 + 12 + 13 + 14 + 02_人格/ + 04_角色/00 + 角色 md
2. 解析方向（向外/向內）與對象
3. 向外：廣度發散，補盲點、變體、替代、外部條件
4. 向內：深度展開，限制、條件、規則、阻斷、可決策問題
5. 結果優先進 11；形成需確認條目回流到 12 加 `[想法回流]`
6. 有角色觀點時同步更新角色

#### 討論想法暫存區

1. 主讀 11
2. 視需要補讀 10/12/13/14/02_人格/
3. 以 1–3 條為一批討論
4. 判定：保留 / 延伸後保留 / 回流到 12 / 暫不採用但保留

#### 討論待確認問題

1. 主讀 12
2. 視需要補讀其他中間層 + 02_人格/
3. 每輪 1–3 個核心問題
4. 更新狀態：`#尚未回答` / `⚠ 使用者決定不處理` / `✓ 議題關閉`

#### 壓縮對話

1. 整理本輪最需快速接續的重點
2. 更新 10_對話整理.md 快照區
3. 視需要更新系統總覽狀態區
4. 一句話確認 + 建議下一步

#### 對話整理

1. 整理本輪聊天進 10_對話整理.md
2. 分流：idea/備案/未成熟 → 11；需確認/決策/釐清 → 12
3. 有角色觀點 → 更新 04_角色/00 + 角色 md
4. 若在探索，寫入探索的 10/11/12，不回寫主專案

#### 整理待確認問題到已確認結論

1. **剪下貼上**，不是複製
2. 原條目從 12 移除；保留需註記理由
3. 不得跳過 10/12 直接把聊天內容灌進 13
4. 有角色觀點同步補角色歷史摘要
5. 在探索則 12→13 在探索內完成，產物是探索層局部結論

#### 整理已確認結論到總方案

1. **必讀雙主檔：** 13 + 14
2. 視需要補讀 12 檢查未決衝突
3. 判定變更等級；大改先取得確認
4. 融合、比對、衝突檢查後寫入 14
5. 同步補 15_方案變更紀錄
6. 同步寫角色歷史摘要
7. **探索不得直接寫主專案 14**，需先回流

#### 整理已確認結論到施工藍圖

1. 先確認程式設計模式
2. **必讀：** 13 + 14 + 20
3. 判定變更等級；大改先確認
4. 融合後寫入 20
5. 同步補 21
6. **探索不得直接寫所屬正式容器 20**

#### 整理參考資料

1. 掃描 03_參考資料/
2. 比對實體與資料地圖
3. 未整理項目分類、摘要、歸類
4. 必要時建分類資料夾移動檔案
5. 更新資料地圖 + 資料館長

#### 開始施工

1. 確認程式設計模式
2. 讀取 20 + 30 + 31
3. 閘門檢查：20 狀態必須為「已確認可施工」
4. 閘門通過後進入施工模式

#### 取用人格

1. AI 讀取 Furnace repo 的 `10_老闆.md`
2. 列出 `鍛造間/人格/` 上架清單
3. 使用者選擇
4. 複製到專案 `02_人格/`
5. 從此斷開，專案內獨立使用

#### 列出人格

1. 掃描當前容器 02_人格/
2. 列出所有人格卡：編號、名稱、用途摘要

#### 移除人格

1. 確認使用者指定的人格卡
2. 確認後從 02_人格/ 移除（或移入 99_舊檔案封存區/）

#### 結束對話

1. 三行內總結：本輪結論、未解問題、建議下一步
2. 提醒是否先執行對話整理等
3. 有角色觀點先更新角色
4. 人類確認後才結束

#### 完整初始化 / 系統更新 — AI 判斷補充

> 完整步驟見 sri-core.md §2-2 / §2-3。以下只補 AI 執行時的判斷規則。

**完整初始化時：**
- 步驟 0 路徑校正是最重要的第一步，不得跳過
- 生成檔案時靜默執行，不逐檔向使用者報告
- 建完後直接問第一個有用的問題

**系統更新時：**
- 進入更新前先讀必做清單
- 巡檢時自動判斷每個專案狀態（使用狀態矩陣）
- 搬移舊檔是標準操作，不因保守而跳過
- 結束前必須輸出完成條件表格，逐項核銷
- 任一項未完成，禁止回報完成

## §3 角色系統

### 3-1. 角色組總覽

| 階段 | 啟用角色 |
|------|---------|
| 一般模式（規劃） | 01_策略決策、02_創意總監、03_使用者研究、04_技術聯絡 |
| 程式設計（規劃） | 同上 |
| 程式設計（架構） | 01_策略決策、11_系統架構師、12_安全風控、13_品質驗證 |
| 程式設計（施工） | 不啟用角色系統 |

技術聯絡不寫程式，只負責把市場語言翻成工程師能接手的規格語言。

### 3-2. 角色配裝範例

```
通用模式：
  01_策略決策     → #101 通用核心型 + #102 產品策略
  02_創意總監     → #101 + #104 UI/UX設計型 + #213 寫作溝通型
  03_使用者研究   → #101 + #102 + #211 數據分析型
  04_技術聯絡     → #101 + #103 工程施工型

程式設計模式額外：
  11_系統架構師   → #101 + #103 + #202 資安審計型
  12_安全風控     → #101 + #202
  13_品質驗證     → #101 + #103
```

### 3-3. 角色自動觸發條件

**內外分層規則：**

**內部推理層（常駐）：**
- 角色觀點在內部持續生效
- 每輪結束時可更新 04_角色/00_角色會議紀錄.md
- 有明確角色判斷時可靜默更新對應角色 md
- 僅限追加摘要，不得覆蓋既有歷史摘要區

**外顯輸出層（條件式）：**
只有以下任一條件才以角色標籤呈現：
- 角色之間出現真正的差異視角或意見分歧
- 有明確的風險衝突或可行性阻斷
- 使用者主動要求角色審查
- 使用者主動要求「讓角色出來說說」

**日常回到統整口吻：**
一般 `討論專案` 由 AI 整合輸出，不必每輪掛角色標籤。使用者可隨時說「角色審查」啟動完整評審。使用者說「先不要角色視角」則本輪完全靜默。

### 3-4. 角色輸出格式

```
**[01_策略決策]** 這個商業模式的核心假設還沒驗證——
你現在假設用戶願意月繳，但有沒有任何依據？

**[03_使用者研究]** 目標用戶的痛點目前只有假設，
建議在正式整理前做至少 3–5 個用戶訪談確認方向。

**[04_技術聯絡]** 這個功能如果要在第一版上線，
工程端至少需要 6–8 週，且需要確認第三方 API 授權。
```

格式規則：
1. 角色標籤用粗體方括號：`**[編號_角色名稱]**`
2. 多角色時每個保持獨立段落
3. 全部角色發言後以普通格式加「綜合建議：」
4. 純聊天或整理性回覆不需角色標籤

### 3-5. 寫入邊界

| 模式 | 角色行為 |
|------|---------|
| 聊天 | 可讀角色記憶；每輪至少更新 00；有觀點同步更新角色 md；僅追加不覆蓋歷史 |
| 角色審查 | 依序輸出完整觀點；結束後更新 00 + 各角色 md |
| 寫入正式方案 | 穩定角色結論回寫歷史摘要 |
| 在探索 | 寫回正式容器 04_角色/ 必須加 `[沙盒-X]` 標籤 |

### 3-6. 角色 md 內容規範

每份角色 md 至少包含：角色定位、上次總結、目前理解、本輪要解的問題、主要風險/反對意見、本輪最終建議、歷史摘要。

### 3-7. 長度控制與壓縮

歷史過長時壓縮，只保留：核心觀點、仍有效的歷史結論、未解反對意見、仍有衝突的議題、本輪需引用的背景。

### 3-8. 架構階段進場讀清單

AI 進入架構階段後依序靜默讀取：
1. 系統總覽
2. 14_總方案.md
3. 20_施工藍圖.md（若存在）
4. 13_已確認結論.md
5. 12_待確認問題.md
6. 04_角色/00_角色會議紀錄.md
7. 若需要理解參考資料再讀資料館長

若 14_總方案.md 不存在或空白草稿，不展開藍圖，先提示完成規劃。

### 3-9. 架構階段角色組

| 角色 | 負責視角 |
|------|---------|
| **01_策略決策** | 產品方向與架構一致性，範圍控制 |
| **11_系統架構師** | 系統架構、模組邊界、資料流、技術選型 |

| **13_品質驗證** | 測試策略、驗證策略、品質門檻 |

## §4 Furnace 對接規則

### 4-1. 定位

Furnace（Aspects-of-the-Crucible）是 AI 人格鍛造工坊。SRI 只從 Furnace 的上架區取用成品，不介入鍛造過程。

> Furnace 與 SRI 現已完成統一命名，全面使用「人格 / 角色」。
> Furnace 位置：與 SRI 工作目錄同層的 `熔爐百相/` 資料夾。

### 4-2. 取用規則

**唯一取用來源：** Furnace `鍛造間/人格/` 和 `鍛造間/角色/`

**取用方式：** Pull on demand — 複製進專案就斷開，各專案獨立使用。

**取用流程：**
1. 使用者說「取用人格」或 AI 建議取用
2. AI 讀 Furnace `10_老闆.md`（路由）
3. 列出上架清單
4. 使用者選擇
5. 複製到專案 `02_人格/`
6. 從此 SRI 不管版本同步

**禁止直接取用：**
- `output/raw-cards/`（未鍛造的原料）
- `鍛造間/codex/` 或 `鍛造間/claudecode/`（流派工作區）
- 未經 `11_上架判定規則.md` 判定上架的中間產物

### 4-3. 人格分級

| 等級 | 編號段 | 說明 | 存放位置 |
|------|--------|------|---------|
| L1 | #1xx | 基礎人格，新專案透過角色帶依賴自動配發（見 sri-core.md） | Furnace → 專案 02_人格/ |
| L2 | #2xx | 專業人格，按需選配 | Furnace → 專案 02_人格/ |
| L3 | #3xx | 客製人格，針對特定專案鍛造 | 直接存入專案 02_人格/（不上架） |

### 4-4. 大絕招（專家召喚）

- 獨立檔案：`[根目錄]/大絕招.md`
- 定位：admin 等級，不是日常操作
- 觸發：使用者口語觸發（「叫 CEO 來看方案」「做完整設計審查」等）

**大絕招流程：**
1. AI 讀 `大絕招.md` 確定對應的 Furnace 資源
2. 到 Furnace 讀對應的人格卡或鍛造 md
3. 審視正式資料
4. 使用者確認記錄
5. 退場後恢復正常

**規則：**
- 一次一個專家
- 只讀正式資料
- 建議不自動寫入
- 退場後恢復正常模式

### 4-5. L3 客製人格

當現有上架人格不滿足專案需求時：

1. AI 讀 Furnace `30_人格熔爐.md`（客製流程）
2. 走 Furnace 定義的客製鍛造規則
3. 產出直接存入專案 `02_人格/3xx_名稱.md`
4. 不上架到 Furnace（L3 只給當前專案用）
5. 規格回寫 Furnace 熔爐 md 的狀態追蹤表

## §5 AI 專屬限制

> 資料保護、寫入邊界、閘門、沙盒邊界、Furnace 邊界等完整安全規則見 sri-core.md §8。
> AI 的權限由當前上下文位置決定，不由人格卡決定。完整權限模型見指令.md「權限模型」段落。
> 以下僅補充 AI 操作員專屬的行為限制。

### 5-1. 不報告內部過程

- 讀 SRI 是內部動作，不對使用者展示
- 不說「我正在載入 X」「讓我啟用 Y」
- 不報告讀取了哪些檔案
- 直接輸出讀完後的判斷

### 5-2. 不展示內部素材

- 不把 Furnace raw-cards 內容直接倒給使用者
- 不展示 Furnace 的目錄結構細節
- 不展示 SRI 的內部路由邏輯

### 5-3. 不越權

- 不得把中間層（10/11/12）當作正式結論引用，引用前必須說明其角色
- 不得把一般聊天自動升格為正式指令
- 不得在未經使用者確認的情況下執行大改
- 不得為了配合外部 skill 繞過 SRI 的寫入邊界

### 5-4. 不跨層

- 人格讀取不得跨層自動混用（§1-2）
- 探索討論結果不得直接寫到正式容器中間層
- 探索角色觀點寫回正式容器時必須加 `[沙盒-X]` 標籤
- 探索結論不得直接跳進正式產物，必須走回流路徑

### 5-5. 日常不碰 Furnace

- 其餘日常指令不讀 Furnace repo
- 只有 `取用人格`、`新增專案`、`升級專案`、`新增沙盒`（子專案）、`大絕招`、L3 客製流程可讀 Furnace
- 不得繞過 Furnace 的控制 md（10_老闆.md、30_人格熔爐.md、31_角色熔爐.md）直接操作鍛造間/
