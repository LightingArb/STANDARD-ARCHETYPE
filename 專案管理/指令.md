## 常用指令

> 本檔是 SRI 三態中的**第二態入口**：指令驅動半自動。
> 第一態（純人工）：專案管理使用手冊.md
> 第三態（閘門式自主推進）：大絕招.md
> v4 起，10-15 會在背景持續自動更新；`對話整理` 仍保留為手動強制入口。

---

### 開始使用

- `完整初始化`：第一次建立整套環境。
- `系統更新`：全域總更新。同步 source / 衍生物、巡檢專案、補缺檔、整理路徑。

---

### 專案操作

- `新增一般專案 名稱`：建立一般模式專案。建立 10-15、`01_團隊/`、`02_參考資料/`、`03_文件產出/`、`05_沙盒區/`、`99_舊檔案封存區/`。
- `新增程式專案 名稱`：建立程式設計模式專案。除上述結構外，再建 20/21/30/31 與 `04_主專案程式/`。
- `進入專案 名稱或編號`：回到既有專案，快速接續。
- `專案說明 名稱` / `這個專案是做什麼的`：看目前方向、進度、缺口。
- `升級專案 程式設計`：一般模式升級成程式設計模式。補齊完整結構與團隊。

---

### 沙盒操作

- `新增沙盒 名稱`：AI 會問你要開**探索**還是**子專案**。
- `進入沙盒 名稱或代號`：切到某個沙盒。
- `回到主專案`：從沙盒回主線。
- `比較所有沙盒`：比較所有探索 / 子專案的結論與風險。
- `回流沙盒到主專案`：把沙盒結論整理回主線。仍需你確認才寫入。
- `自動回流`：AI 先幫你挑哪些值得回流，仍需你確認。
- `升級沙盒`：探索 → 子專案。
- `降級沙盒`：子專案 → 探索。

**v4 沙盒規則：**

- 沙盒固定放在 `05_沙盒區/`。
- 探索用字母前綴：`A_名稱`
- 子專案用數字前綴：`01_名稱`
- 一般模式只能開探索，不能開子專案。

---

### 討論與發想

- `討論專案`：AI 自動判斷你目前在規劃 / 架構 / 施工 / 運維哪個階段。
- `角色討論 主題`：多角色先發散再收斂。你看到的是結論與分歧點，完整過程記到 `01_團隊/角色會議紀錄.md`。
- `角色審查`：比 `角色討論` 更正式，適合重大決策、風險或技術分歧。
- `發散思考 主題`：多角色純發散，不急著下結論。
- `收斂思考`：把剛才的發散結果整理成可採取方案。
- `延伸想法`：針對單一主題往外擴展或往內深挖。
- `討論想法暫存區`：逐條處理 11_想法暫存區。
- `討論待確認問題`：逐條處理 12_待確認問題。

---

### 整理

- `壓縮對話`：更新 10_對話整理.md 快照區。
- `對話整理`：立即執行一次完整分流。
- `整理待確認問題到已確認結論`：12 → 13，採剪下貼上。
- `整理已確認結論到總方案`：13 + 14 融合後寫回 14。
- `整理已確認結論到施工藍圖`：13 + 14 + 20 融合後寫回 20。
- `整理參考資料`：整理 `02_參考資料/`，不刪原始素材。

**v4 自動更新：**

- 背景最小間隔 3 輪、最大間隔 10 輪。
- AI 會自動把想法 / 問題 / 結論 / 方向分流到 10-15。
- `對話整理` 不是唯一入口，只是手動強制執行。

---

### 施工與執行單元

- `開始施工`：僅程式設計模式。前提是 `20_施工藍圖.md` 已標記「已確認可施工」。
- `新增執行單元 名稱`：在 `04_主專案程式/` 內建立新的執行單元。
- `進入執行單元 編號或名稱`：切到指定執行單元，主讀其 README。
- `執行 編號或名稱`：依該單元 README 約定執行。
- `查看輸出 編號或名稱`：看該單元 `output/` 結果。
- `升級執行單元 編號`：探索版 → 子專案版。
- `降級執行單元 編號`：子專案版 → 探索版。
- `列出執行單元`：掃描 `04_主專案程式/`。

**v4 命名規則：**

- 探索版：`001_名稱__探索`
- 子專案版：`002_名稱__子專案`

**程式碼修改閘門（寫死）：**

改 `04_主專案程式/` 內任何非 `.md` 檔案之前，AI 必須：

1. 停下來。
2. 列出要改什麼、為什麼改。
3. 等你確認「可以改」。
4. 確認後才執行。

---

### 團隊操作

- `取用人格`：從 Furnace 取人格卡到當前容器 `01_團隊/`。
- `列出團隊`：列出當前容器已有的人格與角色。
- `移除人格`：從 `01_團隊/` 移除指定人格卡或封存。

**v4 團隊位置：**

- 人格與角色都併入 `01_團隊/`
- 人格檔名：`人格_101_通用核心型.md`
- 角色檔名：`角色_01_策略決策.md`
- 會議紀錄：`角色會議紀錄.md`

**自動配發：**

| 時機 | 自動帶入 |
|------|---------|
| 新增一般專案 | 角色 01-04 + 人格 #101 #102 #103 #104 #211 #213 |
| 新增程式專案 | 上述 + 角色 11-13 + 人格 #202 |
| 新增子專案 | 同程式專案 |
| 新增探索 | 不複製，繼承所屬正式容器 |

---

### MCP 操作

- `列出 MCP 伺服器`：讀 `99_基地/mcp/servers.json`，列出可用 server。
- `啟用 MCP 名稱`：啟用指定 MCP server。
- `停用 MCP 名稱`：停用指定 MCP server。

**MCP v4 原則：**

1. 所有 server 預設 `read-only`
2. 首次連線必須確認
3. 會寫外部系統的 server 永遠需要確認
4. 路由固定：
   - 原始輸入 → `02_參考資料/raw_input/`
   - 消化後分析 → `03_文件產出/`
   - 實際程式整合 → `04_主專案程式/`（先過程式碼閘門）

## 角色進場規則

| 階段 | 啟用角色 | 何時外顯 |
|------|---------|---------|
| 規劃 | 角色_01、02、03、04 | 有分歧、你要求、或 AI 建議開角色討論 |
| 架構 | 角色_01、11、12、13 | 重大技術決策、安全風險、藍圖分歧 |
| 施工 | 不常態啟用 | 需多角色深審時，先停或回架構 |
| 運維 | 不常態啟用 | 迭代 / 結構性問題時回上游階段 |

**你說 `先不要角色視角`：本輪完全靜默角色。**

## 寫入邊界速查

- 10/11/12 不是正式拍板依據。
- `02_參考資料/` 是素材，不是正式規格。
- `03_文件產出/` 是交付物，不是正式決策真值。
- 沙盒結論不能直接寫進主線 14/20/30。
- 探索若要影響主線，必須走：沙盒 13 → 主線 12 → 13 → 14 → 20 → 30。
