# 大絕招

> 本檔是 SRI 三態中的**第三態入口**：閘門式自主推進（Harness）。
> 第一態：專案管理使用手冊.md
> 第二態：指令.md
> 這不是新系統，而是 AI 自主駕馭同一套結構、指令與閘門。

---

## 啟動 Harness

1. 進入你要推進的專案：`進入專案 <名稱>`
2. 對 AI 說以下任一句：
   - `啟動 Harness`
   - `自主推進到 [目標]`
   - `自主完成這個專案`
3. AI 會先做目標解析（Step 0），列出推進計畫。
4. 你確認計畫後，Harness 才開始自主推進。
5. 你隨時可以說 `暫停` 或 `停止`。

---

## 這是什麼

第二態是「人類決定下一步，AI 執行」。

第三態是「AI 自己判斷現在該下哪個第二態指令，再自己執行」。

也就是說：

```text
第二態：你說「討論專案」→ AI 執行
第三態：AI 判斷現在該「討論專案」→ AI 自己執行
```

Harness 不發明新規則，不繞過現有規則，只是把決策權前移給 AI，但保留硬閘門。

---

## 什麼情況不建議啟動

以下情況較適合留在第二態：

- 目標本身還很模糊，連成功條件都說不清楚。
- 使用者明確表示只是想聊天探索，不想自動推進。
- 任務大量依賴登入、敏感權限或人類手動操作。
- 任務主要是美學偏好、創意風格、品牌感受。
- 只是單輪小修，不需要完整自主循環。

遇到這些情況，AI 應先做 Step 0 計畫，再由使用者決定要不要真的開啟 Harness。

---

## Step 0：目標解析

收到 Harness 指令後，AI 必須先做以下五件事：

1. 解析目標範圍
   - 目標是什麼
   - 起點在哪
   - 終點在哪
2. 定位當前上下文
   - 主專案 / 子專案 / 探索
   - 目前是否已在某個執行單元
   - 若上下文不明，必須先問
3. 判斷專案模式
   - 一般模式 or 程式設計模式
   - 若還沒有專案，先決定該建立哪一種
4. 盤點團隊與約束
   - `01_團隊/` 有哪些角色與人格
   - 是否缺卡
   - 是否已存在藍圖或施工進度
5. 提出推進計畫
   - 預計經過哪些階段
   - 預計會遇到哪些閘門
   - 已知風險與不確定

**Step 0 完成後一定要等使用者確認，不能直接開跑。**

---

## 自主推進循環

```text
while 目標未完成:
  1. 判斷當前階段
  2. 判斷當前該做什麼
  3. 判斷需不需要角色討論
  4. 檢查有沒有硬閘門
  5. 執行一個最小可驗證步驟
  6. 更新 10-15 / 團隊會議紀錄 / 系統總覽狀態
  7. 判斷是否該停下來報告
```

### 最小可驗證步驟

Harness 每輪只允許完成一個最小可驗證步驟，例如：

- 完成一次待確認問題收斂
- 完成一次已確認結論整理
- 完成一次角色討論
- 完成一次參考資料整理
- 完成一次交付文件產出
- 完成一次施工藍圖更新
- 完成一次進度報告

**禁止：**

- 同一輪同時做需求收斂、藍圖重寫、開始施工
- 同一輪跨越多個硬閘門
- 同一輪自行做多個重大方向決策

---

## Harness 可以自主做什麼

在 `.md` 檔案範圍內，Harness 具備高度自主權。

| 類型 | 可否自主 |
|------|---------|
| 10-15 自動分流與整理 | 可以 |
| `01_團隊/角色會議紀錄.md` 更新 | 可以 |
| `02_參考資料/` 整理與分類 | 可以 |
| `03_文件產出/` 產出新文件 | 可以 |
| 上網抓資料 | 可以 |
| 自主開角色討論 | 可以 |
| 決策、發散、收斂 | 可以 |
| 記下疑問先繼續推進 | 可以 |

---

## 硬閘門

以下情況一律必停，不能自主越過：

### 1. 程式碼修改閘門（寫死）

Harness 在 `.md` 檔案範圍內完全自主。

但在修改 `04_主專案程式/` 內任何非 `.md` 檔案之前，必須：

1. 停下來
2. 列出要改什麼、為什麼改
3. 等使用者確認 `可以改`
4. 確認後才執行

使用者確認後，AI 可在**該批次**內自主完成修改。下一批新的程式碼修改，要再次確認。

### 2. 大量檔案操作

以下動作必停確認：

- 大量刪除
- 大量搬移
- 大量覆蓋
- 會導致既有檔案大規模改名或結構重排的操作

### 3. MCP 首次連線 / 權限升級

以下情況必停：

- 第一次啟用某個 MCP server
- 權限從 read-only 升級到可寫
- 會寫入外部系統

## 角色討論

Harness 推進過程中，AI 可自主觸發角色討論。

適合觸發的情況：

- 遇到方向分歧
- 遇到技術可行性疑問
- 遇到安全或風險疑問
- 規劃階段重大決策前

規則：

- 完整過程寫入 `01_團隊/角色會議紀錄.md`
- 使用者看到的是進度報告中的結論與分歧點
- 若結果涉及硬閘門，仍要停下來

---

## 停止機制（v4）

v4 不再用舊版計數器式停止主機制。

Harness 的停止原則改為：

1. **AI 自主判斷**
   - AI 覺得「該停下來讓使用者看一下」時，就可以停。
2. **安全網上限**
   - 最多連續 10 輪自主推進，必須停一次做進度報告。
3. **硬閘門**
   - 改程式碼、大量檔案操作、MCP 首次連線或權限升級，無論第幾輪都必停。
4. **使用者中斷**
   - 使用者隨時可說 `暫停` 或 `停止`。

---

## 進度報告

Harness 停下來時，報告至少包含：

- 現在做到哪裡
- 為什麼停
- 已完成什麼
- 下一步打算做什麼
- 目前風險或待決事項

### 建議格式

```markdown
## Harness 進度報告

- 目標：
- 當前階段：
- 已完成：
- 目前卡點 / 停止原因：
- 下一步：
- 風險：
```

---

## 完成報告

達成目標後，輸出完成報告，至少包含：

- 原始目標
- 結果：完成 / 部分完成
- 經過的階段
- 產出清單
- 硬閘門確認紀錄
- 尚未解決事項
- 建議下一步

---

## 暫停、停止、繼續

- `暫停`：停在當前步驟，輸出進度報告。
- `停止`：結束 Harness，回到第二態。
- `繼續推進`：從上次暫停點繼續。
- `重新推進到 [目標]`：重跑 Step 0，重新定義範圍。

---

## MCP 在 Harness 中的使用

Harness 可以使用 MCP，但仍受同一套硬閘門管制。

路由固定如下：

- 原始輸入 → `02_參考資料/raw_input/`
- AI 消化後分析 → `03_文件產出/`
- 實際程式碼整合 → `04_主專案程式/`

第三項仍必須先過程式碼修改閘門。

---

## 原則

1. Harness 不發明新規則。
2. Harness 不繞過硬閘門。
3. Harness 不繞過變更分級。
4. Harness 可被隨時中斷。
5. Harness 每輪只做一個最小可驗證步驟。
6. Harness 在 `.md` 內高度自主，在程式碼與外部權限上高度保守。
