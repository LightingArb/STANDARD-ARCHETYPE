## 二、A 類：歷史預報快照來源（YES 方向核心資料）

### 2.1 Open-Meteo Historical Forecast API（第一優先）

**用途：** 取得「過去某天的預報，對未來某天最高溫的預測值」

**關鍵發現：**
- 查詢方式：`start_date` / `end_date` + `daily=temperature_2m_max`
- 座標直接傳入：`latitude=51.5048&longitude=0.0495`
- **可區分 lead time**：用 `_previous_dayN` 後綴，例如 `temperature_2m_max_previous_day2` = 2 天前的 run 對該日的預報
- lead time 最大約 day7（第 8 天以上回傳 null）
- 支援多模型：GFS、ECMWF IFS 0.25、JMA 等 40+ 種

**歷史深度：**
- GFS 系列：2022 年起（部分 2021 年 3 月）
- ECMWF IFS 0.25：2024 年初起

**限制：**
- 回傳 JSON 不含明確的 model run timestamp，需根據 `_previous_dayN` 自行推算
- 非商業免費，商業需付費
- 日限 10,000 次、月限 300,000 次

**範例查詢：**
```
https://historical-forecast-api.open-meteo.com/v1/forecast?latitude=51.5048&longitude=0.0495&daily=temperature_2m_max,temperature_2m_max_previous_day1,temperature_2m_max_previous_day2&timezone=Europe/London&start_date=2025-03-01&end_date=2025-03-31&models=gfs_seamless
```

### 2.2 Open-Meteo Previous Runs API（補充）

**用途：** 比較最近 ~7 天內各次模式運行的預報差異

**實測結果（GFS, 2026-04-02 12:00 London）：**

| 模式運行 | 預報值 |
|---------|--------|
| 最新 (day0) | 7.3°C |
| 1 天前 (day1) | 9.8°C |
| 2 天前 (day2) | 7.5°C |
| 3 天前 (day3) | 12.0°C |

**結論：** 只適合看近期預報差異，無法回補歷史。搭配 Historical Forecast API 使用。

### 2.3 NOAA GFS archive（第二優先）

**用途：** 回補 2022 年以前的歷史預報，或需要完全免費無商業限制時使用

**資料位置：**
- AWS S3: `noaa-gfs-bdp-pds`（免費、無需帳號）
- NCEI: HTTPS / THREDDS 存取

**關鍵發現：**
- 每天 4 次 run（00Z/06Z/12Z/18Z），每個 run 預報到 384 小時（16 天）
- 直接包含 `TMAX`（2m 最高溫）變數，每 3 小時一個值
- 取某日 8 個 TMAX 值的最大值 = 日最高溫預測
- 格式為 GRIB2，0.25° 格點（~28km），EGLC 附近有 4 個格點可做雙線性內插

**推薦工具鏈：** Python + Herbie（自動處理 AWS 路徑）+ xarray/cfgrib
- 用 `searchString` 參數可只下載 TMAX 變數，每個檔案從數百 MB 縮到數 KB

**限制：**
- 工程複雜度中~高（需處理 GRIB2 格式）
- 大量回溯時檔案數量大

### 2.4 ECMWF archive（第三優先）

**用途：** 最高品質的歷史預報，對倫敦預報精度優於 GFS

**關鍵發現：**
- ECMWF IFS 解析度 9km（GFS 為 13km），歐洲區域預報更準
- **但 Open Data 僅保留最近 2-3 天**，完整檔案需透過 MARS 系統
- MARS 需要會員國機構或簽訂 Service Agreement，有審核流程
- **最實際的做法：透過 Open-Meteo Historical Forecast API 取得 ECMWF IFS 預報**（2024 年起）

**結論：** 不建議直接對接 ECMWF，改用 Open-Meteo 作為中間層。

## 三、B 類：結算真值（Polymarket 定義）

### 3.1 Weather Underground EGLC 日歷史頁

**用途：** Polymarket 規則指定的最終結算真值

**頁面格式：**
```
https://www.wunderground.com/history/daily/gb/london/EGLC/date/2026-03-30
```

**可取得欄位：**
- 日摘要：最高溫、最低溫、平均溫度、降水、風速、氣壓等
- **日內每小時觀測表格：有**，包含溫度、露點、風速、降水等

**問題：**
- 使用 Angular SPA，直接 HTTP GET 拿不到資料，需 headless browser（Selenium/Playwright）
- 底層 API 是 `api.weather.com`（IBM），需 API key、付費
- **WU 的免費 API 已停止提供**

### 3.2 Iowa Environmental Mesonet (IEM) — 最佳替代方案

**這是本次研究最重要的發現之一。**

**用途：** 取得 EGLC 的日內每小時（實際每 30 分鐘）觀測溫度，資料來源與 WU 相同（都是 METAR）

**已驗證可用：**
- 站點：EGLC（London City Airport），`GB__ASOS` network
- 觀測頻率：每 30 分鐘（比每小時更密），一天約 48 筆
- 歷史深度：1988 年 1 月 29 日起
- 格式：CSV，可直接 HTTP GET 下載
- **完全免費、無需 API key、無速率限制**

**可用欄位：** 溫度(°F)、露點、相對濕度、風速/風向、降水、氣壓、能見度、雲量、原始 METAR

**範例查詢：**
```
https://mesonet.agron.iastate.edu/cgi-bin/request/asos.py?station=EGLC&data=tmpf&data=dwpf&year1=2026&month1=3&day1=30&year2=2026&month2=3&day2=30&tz=Etc%2FUTC&format=onlycomma
```

**注意：** 溫度單位是華氏度，需自行轉攝氏。

**為什麼 IEM 可以替代 WU 做觀測映射驗證：**
- IEM 和 WU 的 EGLC 資料都來自同一個 METAR 觀測源
- 數值應高度一致（可能完全相同）
- 但仍需實際比對確認，這也是映射品質驗證的一部分

## 四、C 類：實時快照系統（Forward Collection）

### 4.1 Met Office Weather DataHub（DataPoint 已於 2024 年退役）

**重要更新：DataPoint API 已於 2024 年 4 月停用，需改用 DataHub。**

**用途：** 從現在開始，每小時存一次倫敦的天氣預報快照

**查詢方式：**
- 不需要 station ID，直接傳座標：`latitude=51.5048&longitude=0.0495`
- API key 透過 DataHub 網站註冊取得

**可用端點：**

| 端點 | 預報範圍 | 用途 |
|------|---------|------|
| `/point/hourly` | 未來 48 小時 | 逐時預報快照 |
| `/point/three-hourly` | 未來 7 天 | 中期預報快照 |
| `/point/daily` | 未來 7 天 | 日預報快照（含日最高/最低溫） |

**觀測 API：**
- Land Observations：過去 48 小時逐時觀測
- 包含 `screenTemperature`
- 倫敦區域有觀測站覆蓋（需確認 EGLC 是否在 150 站名單中）

**免費額度：** 360 次/天（每小時抓 hourly + daily + observations = 72 次/天，綽綽有餘）

**建議快照架構：**
```
每小時 cron job:
  1. GET /point/hourly  → forecast_hourly_2026-04-01T1400Z.json
  2. GET /point/daily   → forecast_daily_2026-04-01T1400Z.json
  3. GET observations   → observation_2026-04-01T1400Z.json
```

## 五、補充來源

### 5.1 NOAA Integrated Surface Database (ISD)

- 全球 35,000+ 站點，含 EGLC
- 歷史回溯至 1901 年
- 每小時全球觀測
- AWS Open Data 可取得
- 免費但格式較原始（ASCII），需解析
- 可作為 IEM 的備援

### 5.2 Open-Meteo Historical Weather API（ERA5 再分析）

- 端點：`archive-api.open-meteo.com/v1/archive`
- 1940 年起逐小時資料
- 非站點觀測，是再分析模型值
- **不適合作為結算驗證**（與 WU 觀測值可能有差異）
- 可作為「觀測映射品質驗證」中的第三方參照

## 六、映射品質驗證計畫

目的：確認 Open-Meteo / GFS 等來源的預報映射到 WU 結算值是否可靠

**步驟：**
1. 從 IEM 取得 EGLC 的每日最高溫歷史（METAR 觀測，應與 WU 高度一致）
2. 從 Open-Meteo Historical Weather API 取得同期同座標的 ERA5 再分析日最高溫
3. 比對兩者差異，判斷是系統性偏移還是隨機亂跳
4. 若系統性偏移 → 建立校正公式
5. 若隨機亂跳 → 映射基礎不穩，需考慮其他方案

## 七、NO 方向資料缺口解決方案

NO 方向需要「日內每小時觀測溫度」來計算 `remaining_rise`。

**解法（優先順序）：**

1. **IEM EGLC**（首選）：每 30 分鐘 METAR 觀測，1988 年起，免費 CSV
2. **Met Office DataHub Land Observations**（forward collection 補充）：過去 48h 逐時，需每小時存
3. **WU 日歷史頁面**（備選）：有每小時觀測，但需 headless browser scrape
4. **NOAA ISD**（備援）：每小時觀測，格式較原始
