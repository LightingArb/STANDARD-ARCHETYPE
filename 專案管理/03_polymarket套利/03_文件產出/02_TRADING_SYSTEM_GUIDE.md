## 一、一句話說清楚這套系統在做什麼

**預報不等於真實值，但預報的「偏差習慣」是可以學的。學會之後你就比只看裸預報的市場多一層資訊，這層資訊就是你的 alpha。**

## 二、整條交易邏輯鏈

```
歷史誤差資料（forecast_temp vs observed_temp）
    ↓
數學模型學會「這個預報在這種條件下通常會偏多少」
    ↓
下次預報進來時，模型輸出「真實溫度的完整分布」（19 個分位點）
    ↓
對照合約條件，算出「事件機率 pₜ」
    ↓
拿 pₜ 和市場價格比 → 算出 edge 和 EV
    ↓
YES 有正 EV → 買 YES；NO 有正 EV → 買 NO；都沒有 → 不做
    ↓
有正 EV → 用 Kelly 算下多少 → 進場
    ↓
進場後，每小時重新算 pₜ → 比較「現在賣」vs「繼續抱」→ 決定離場
```

## 三、每一步的具體計算

### Step 1：建立誤差模型

你有一堆歷史資料，每筆包含：

```
模型 D1 在第 X 小時前，預測 Y 城市溫度 = Z1（forecast_temp = f）
結算時真實溫度 = Z2（observed_temp = y）
誤差 = Z2 − Z1 = e（error_temp = y − f）

三個都要存，不能只存差值。
```

把大量這種資料丟進數學模型，模型就會學到：

> 「D1 在 80 小時前預測時，誤差通常分布在 −3 到 +5 度之間，
>   中位數偏 +0.8 度，90% 的情況不超過 +4 度。」

---

### Step 2：下次預報進來，輸出溫度分布

D1 說：「80 小時後溫度是 20.5 度。」

你不是直接信 20.5，而是：真實溫度 = 20.5 + 誤差。
模型根據歷史誤差，輸出未來真實溫度的分布：

```
預報值 20.5 + 各分位點的誤差 = 溫度分布

  5%  分位點：20.5 + (−2.5) = 18.0°C
  10% 分位點：20.5 + (−1.8) = 18.7°C
  25% 分位點：20.5 + (−0.6) = 19.9°C
  50% 分位點：20.5 + (+0.4) = 20.9°C    ← 中位數
  75% 分位點：20.5 + (+1.5) = 22.0°C
  90% 分位點：20.5 + (+2.8) = 23.3°C
  95% 分位點：20.5 + (+3.6) = 24.1°C
```

這就是你對「未來真實溫度」的完整看法，不是一個點數字，而是一整條分布。

#### 分位數間隔怎麼選？

**第一版用 5% 間隔，共 19 個分位點：**

```
q05, q10, q15, q20, q25, q30, q35, q40, q45, q50,
q55, q60, q65, q70, q75, q80, q85, q90, q95
```

為什麼 5% 夠用：你最常交易的合約區間通常是 1-2 度寬（如 21-22°C）。
誤差分布的標準差大約 3-4 度時，5% 間隔對應約 0.5-1 度的溫度解析度，
用線性插值算區間機率已經足夠精確。

如果你交易極端合約（如「高於 35°C」），可在尾部加密：q01, q02, q03, q97, q98, q99。

#### 查表矩陣的結構

你可以把歷史誤差整理成一張查表矩陣。
但每一格不是一個數字，而是一組 19 個分位數：

```
查表 key = (城市, lead_hours, 月份, 時段)
查表 value = {q05: -2.5, q10: -1.8, ..., q95: +3.6}
```

查到之後：forecast_temp + 每個分位點 = 溫度分布 → 再算 pₜ。

---

### Step 3：對照合約條件，算出事件機率 pₜ

假設 Polymarket 合約是：**YES = 結算溫度落在 21°C ~ 22°C 之間**

```
pₜ = P(21 ≤ 真實溫度 ≤ 22)
   = CDF(22) − CDF(21)
   = 例如 0.15
   = 15%
```

其他合約類型也一樣：
- 「高於 30°C」→ pₜ = 1 − CDF(30)
- 「低於 0°C」 → pₜ = CDF(0)

**YES 和 NO 的關係**：你的 pₜ(YES) + pₜ(NO) = 100%，永遠互補。
但市場價格因為 spread 的關係，YES ask + NO ask 可能超過 $1.00。

---

### Step 4：兩邊都看，算 EV

拿到 pₜ 之後，YES 和 NO 兩邊都要算，看哪邊有正 edge：

```
YES 這邊：
  Edge = pₜ(YES) − YES_ask_price
  正 → 買 YES

NO 這邊：
  Edge = pₜ(NO) − NO_ask_price = (1 − pₜ) − NO_ask_price
  正 → 買 NO

都沒有正 edge → 不做
```

#### EV 計算範例

```
你的 pₜ(YES) = 15%，市場 YES ask = $0.10

  花 $0.10 買 1 股 YES
  15% 機率中 → 拿回 $1.00 → 淨賺 $0.90
  85% 機率不中 → 拿回 $0.00 → 淨虧 $0.10

  EV = 0.15 × 1.00 = $0.15
  花了 $0.10，期望拿回 $0.15
  EV / cost = 1.50 → 正 EV ✓
```

#### Edge 的三種表達方式

| 表達方式 | 公式 | 本例數值 | 意義 |
|----------|------|----------|------|
| 機率 edge | pₜ − market_price | 0.15 − 0.10 = 0.05 | 你比市場多看到 5% 的機率 |
| 每股 EV | pₜ − cost | $0.15 − $0.10 = $0.05 | 每股期望淨賺 5 分 |
| EV 倍數 | pₜ / cost | 0.15 / 0.10 = 1.50 | 每投入 1 元期望拿回 1.5 元 |

---

### Step 5：用 Kelly 算下多少

```
Cₜ_raw = pₜ = 0.15
Aₜ = ask + fee ≈ 0.103（含 taker fee）

full_kelly = (Cₜ_raw − Aₜ) / (1 − Aₜ) = (0.15 − 0.103) / (1 − 0.103) ≈ 5.2%

實際下注 = 5.2% × 0.25（1/4 Kelly）× $10,000 = $131
買入股數 = $131 / $0.103 ≈ 1,272 股
```

**Kelly 只管「下多少」，不管「什麼時候走」。**

---

### Step 6：進場後，每小時重新評估

```
每小時更新：
  1. 重新跑模型 → 新的 pₜ
  2. 查 orderbook → 算 Lₜ（現在真的賣得掉多少）
  3. 算 Cₜ = pₜ · exp(−rτ)（繼續抱到結算值多少，含時間折現）
  4. 比較：
     Lₜ ≥ Cₜ → 賣掉，拿錢去做下一筆
     Lₜ < Cₜ → 繼續抱，等下一小時再看
```

## 四、三個機率模型的比較

三個模型做的事情一樣：都是輸出一條溫度分布，最後都轉成 pₜ 再算 EV。
差別只是學誤差分布的方法不同，所以算出來的 pₜ 會不一樣。

### 模型 A：Empirical 分桶

**做法**：把歷史誤差按條件分桶（如「7 月 + 下午 + 80 小時」），直接把這個桶裡的誤差排序取分位點。

**優點**：零假設，不需要數學，看到什麼就是什麼。

**缺點**：桶分太細時樣本不夠。比如某個條件組合只剩 8 筆，分位數就很不穩定。

### 模型 B：分位數回歸

**做法**：不分桶，用一個函數去學「在任何條件下，各分位點應該是多少」。
輸入 (forecast_temp, month, hour, lead_hours)，分別學 q05 到 q95。

**優點**：不怕樣本少、可以處理連續變數、不假設誤差是常態分布。

**缺點**：需要選特徵、需要調模型。

### 模型 C：OU / AR(1)

**做法**：假設誤差有一個「正常水位」，不管偏多偏少最終都會回來。只學三個數字：長期平均偏差 μ、回彈速度 φ、波動大小 σ。然後假設誤差是常態分布。

**優點**：只要估三個參數，有解析公式，非常簡單。

**缺點**：強制假設常態分布。如果誤差分布偏斜或有厚尾，它會算錯。

### 同一情境三個模型的 pₜ 比較

```
合約：YES = 21-22°C，預報 20.5°C，80 小時 lead time

模型 A（Empirical）：pₜ = 22%
模型 B（分位數回歸）：pₜ = 24%
模型 C（OU/AR）：    pₜ = 18%

→ 不是選最帥的，而是用 Brier score 驗證哪個最準
```

### 模型選擇建議

第一版三個都做，用 Phase 0 的 Brier score 比較。不要一開始就只押一個。

如果一個明顯最好 → 直接用那個。
如果兩三個差不多好 → 之後再考慮加權合成（不是簡單平均）。
如果都不比市場好 → 先不交易，回去改模型。

## 五、Brier Score 與模型驗證

### 5.1 什麼是 Brier Score

一句話：**衡量你的機率預測和現實之間差多遠的評分。**

```
Brier score = 平均值 of (你的預測機率 − 真實結果)²

真實結果：發生 = 1，沒發生 = 0
分數範圍：0（完美）到 1（最爛）
越低越好。
```

它同時懲罰兩種錯：

```
你說 90% 會中，結果沒中 → (0.90 − 0)² = 0.81 ← 重罰
你說 10% 會中，結果中了 → (0.10 − 1)² = 0.81 ← 重罰
你說 90% 會中，結果中了 → (0.90 − 1)² = 0.01 ← 幾乎不扣分
你說 10% 會中，結果沒中 → (0.10 − 0)² = 0.01 ← 幾乎不扣分
```

### 5.2 怎麼做 Walk-Forward 驗證

**關鍵原則：不能用未來資料訓練，不能全樣本回填。**

```
假設你有 500 筆歷史資料，按時間排序：

  第一輪：用第 1-300 筆訓練，用第 301-350 筆測試
  第二輪：用第 1-350 筆訓練，用第 351-400 筆測試
  第三輪：用第 1-400 筆訓練，用第 401-450 筆測試
  第四輪：用第 1-450 筆訓練，用第 451-500 筆測試

每一輪的測試期間，模型只看到「過去的資料」，然後對未來做預測。
預測完之後再拿真實結果來比。
```

### 5.3 Phase 0：判斷你有沒有 Edge

Phase 0 分兩層：

**Phase 0A — Forecast Edge（你的模型比市場準嗎？）**

```
四個選手同時上場，200 筆測試資料：

模型 A（Empirical）：Brier = 0.19
模型 B（分位數回歸）：Brier = 0.17
模型 C（OU/AR）：    Brier = 0.21
市場 mid price：     Brier = 0.20

→ B 最好（0.17），而且比市場好（0.17 < 0.20）→ 有 forecast edge
→ C 比市場差（0.21 > 0.20）→ 不能用
```

**Phase 0B — Tradable Edge（扣掉費用還有正 EV 嗎？）**

```
Phase 0A 通過不代表 0B 一定通過。

有些情況：
  你比市場準一點點（Brier 只好 0.01）
  但 spread + fee + slippage 把這個優勢吃光了
  → 有 forecast edge，但沒有 tradable edge
  → 不要交易

檢查方式：用模型 B 的 pₜ 去跑回測，模擬實際進場（含 Aₜ）和結算，
看扣掉所有費用後 PnL 是否為正。
```

### 5.4 Brier Score 的限制

Brier score 只能評估「機率預測準不準」，不能直接告訴你「交易能不能賺錢」。
因為交易賺不賺錢還取決於：進場價格、費用、什麼時候離場、倉位大小。

所以驗證要分兩層：
- Brier / calibration → 機率準不準
- PnL / Sharpe → 交易賺不賺錢

## 六、Polymarket 天氣市場費率

### 6.1 核心規則

- **只有 Taker 付費，Maker 永遠免費。**
- Taker = 市價單吃別人的掛單。Maker = 掛限價單等別人來吃你。
- 費率自 2026 年 3 月 30 日起生效。
- **費率會變，不要硬編碼，用 API 動態查詢。**

### 6.2 天氣市場費率參數

| 項目 | 數值 |
|------|------|
| fee_rate | 0.025 |
| exponent | 0.5 |
| Peak effective rate（p=0.50） | **1.25%** |
| Maker rebate | **25%** |

### 6.3 費用公式

```
fee_per_share = p × fee_rate × (p × (1 - p))^exponent

天氣市場：fee_rate = 0.025, exponent = 0.5

白話：費用在 p = 0.50 時最高（1.25%），接近 0 或 1 時趨近於零。
```

### 6.4 費用速查表（每 100 股）

| 價格 | 費用 (USDC) | 有效費率 |
|------|-------------|----------|
| $0.05 | $0.03 | 0.54% |
| $0.10 | $0.08 | 0.75% |
| $0.20 | $0.20 | 1.00% |
| $0.30 | $0.34 | 1.15% |
| **$0.50** | **$0.62** | **1.25%** |
| $0.70 | $0.80 | 1.15% |
| $0.80 | $0.80 | 1.00% |
| $0.90 | $0.67 | 0.75% |
| $0.95 | $0.52 | 0.54% |

**重點**：在低價位（p < 0.15）買入時，費用影響很小（< 1%）。
在 p ≈ 0.50 附近，費用會明顯侵蝕 edge。

### 6.5 Maker vs Taker 策略

| 策略 | 費用 | 速度 | 適合場景 |
|------|------|------|----------|
| 市價單（Taker） | 付費 | 立即成交 | 急著進場、價格快速變動 |
| 限價單（Maker） | 免費 | 不一定成交 | 不急、想省費用 |

**回測一律用 Taker 費用（最保守）。實際交易可以嘗試 Maker。**

### 6.6 造市商激勵（Maker Rebates Program）

- Polymarket 把 taker fee 的 25% 每天分給造市商
- 每日以 USDC 結算，按你在該市場被成交的流動性比例分配
- 要成為 Maker：掛限價單 → 被別人吃到 → 自動收到 rebate

**注意：不要為了 rebate 改變交易策略。alpha 來自機率模型，不是做市。**

### 6.7 費率 API

```
GET https://clob.polymarket.com/fee-rate?token_id={token_id}

回傳非零 = 有費用，回傳 0 = 免費
也可在 market object 中檢查 feesEnabled 欄位
```

## 七、核心公式速查表

### 機率層

| 項目 | 公式 | 說明 |
|------|------|------|
| 誤差 | e = y − f | 真實值 − 預報值 |
| 溫度分布 | f + error_distribution | 預報值 + 誤差分布 |
| 事件機率 | pₜ = P(event \| distribution) | 對照合約條件算出 |

### 交易層

| 項目 | 公式 | 說明 |
|------|------|------|
| 機率 edge | pₜ − market_price | 你比市場多看到多少 |
| 每股 EV | pₜ − cost | 每股期望淨賺多少 |
| EV 倍數 | pₜ / cost | 每投入 1 元期望拿回多少 |
| 進場條件 | Cₜ − Aₜ ≥ min_edge | 持有價值 − 買入成本 ≥ 門檻 |
| Kelly sizing | f* = (pₜ − Aₜ) / (1 − Aₜ) | 用 raw pₜ，不含時間折現 |
| 實際下注 | f* × 0.25 × bankroll | 1/4 Kelly |

### 離場層

| 項目 | 公式 | 說明 |
|------|------|------|
| 持有價值（YES） | Cₜ = pₜ · exp(−rτ) | 含時間折現 |
| 持有價值（NO） | Cₜ = (1−pₜ) · exp(−rτ) | 含時間折現 |
| 可賣淨值 | Lₜ = bid − fee − slippage | 現在真的能拿多少 |
| 離場條件 | Lₜ ≥ Cₜ | 賣掉拿到的 ≥ 繼續抱的價值 |

### 費用層

| 項目 | 公式 | 說明 |
|------|------|------|
| 每股費用 | p × fee_rate × (p(1-p))^exponent | 天氣市場：fee_rate=0.025, exponent=0.5 |
| 買入淨成本 Aₜ | ask + fee_per_share | Taker 買入時 |
| 賣出淨值 Lₜ | bid − fee_per_share | Taker 賣出時 |
| Maker | fee = 0 | 掛限價單免費 |

### 單位定義

| 符號 | 意義 | 單位 |
|------|------|------|
| pₜ | 事件機率 | 0 到 1 |
| τ | 距結算剩餘時間 | 小時 |
| r | 資金再利用收益率 | per-hour |
| fee_rate | Polymarket 費率參數 | 無單位（天氣=0.025） |
| exponent | Polymarket 費率指數 | 無單位（天氣=0.5） |
| Lₜ, Aₜ, Cₜ | 每股價值 | USD per share |
| f | forecast_temp | 預測溫度 |
| y | observed_temp | 真實觀測溫度 |
| e | error_temp = y − f | 誤差 |

## 八、完整數值範例

### 情境設定

```
模型：D1
城市：倫敦
合約：YES = 結算時溫度落在 21°C ~ 22°C
結算時間：80 小時後
D1 預測：20.5°C
你的帳戶：$10,000
```

### Phase 0：你有 edge 嗎？

```
歷史比較（walk-forward，200 筆測試）：
  你的模型 Brier score = 0.18
  市場 mid price Brier score = 0.22

  0.18 < 0.22 → 你的模型比市場準 → Phase 0A 通過

回測 PnL（含所有費用）：
  模擬 200 筆交易，淨 PnL = +$430
  → Phase 0B 通過 → 可以開始交易
```

### 進場判斷

```
模型輸出溫度分布 → P(21 ≤ T ≤ 22) = 15%
pₜ = 0.15

市場 YES ask = $0.10（隱含機率 10%）

fee_per_share = 0.10 × 0.025 × (0.10 × 0.90)^0.5
              = 0.10 × 0.025 × 0.3
              = 0.00075

Aₜ = 0.10 + 0.00075 = $0.10075

Edge = pₜ − Aₜ = 0.15 − 0.10075 = 0.04925 > min_edge(0.03) → 進場 ✓

Kelly：
  full_kelly = (0.15 − 0.10075) / (1 − 0.10075) = 0.0549
  quarter_kelly = 0.0549 × 0.25 = 0.0137
  下注金額 = 0.0137 × $10,000 = $137
  買入股數 = $137 / $0.10075 ≈ 1,360 股
```

### 持有期間（每小時檢查）

```
=== 第 20 小時 ===（距結算還剩 60 小時）
新氣象資料進來 → pₜ = 0.18
Cₜ = 0.18 × exp(−0.0005 × 60) = 0.1747
Lₜ = best_bid − fee = 0.12 − 0.0053 = 0.1147
Lₜ(0.1147) < Cₜ(0.1747) → 繼續持有

=== 第 50 小時 ===（距結算還剩 30 小時）
天氣預報大幅更新 → pₜ = 0.22
Cₜ = 0.22 × exp(−0.0005 × 30) = 0.2167
Lₜ = 0.20 − 0.0080 = 0.1920
Lₜ(0.1920) < Cₜ(0.2167) → 繼續持有

=== 第 70 小時 ===（距結算還剩 10 小時）
bid 飆高 → pₜ = 0.25
Cₜ = 0.25 × exp(−0.0005 × 10) = 0.2488
Lₜ = 0.26 − 0.0097 = 0.2503
Lₜ(0.2503) ≥ Cₜ(0.2488) → 賣出！

買入成本 = 1,360 × $0.10075 = $137
賣出淨得 = 1,360 × $0.2503 = $340
實現利潤 = $340 − $137 = $203
ROI = 148%，持有 70 小時
```

### 如果抱到結算呢？

```
結算溫度 21.3°C（落在 21-22 之間）→ 拿回 1,360 × $1.00 = $1,360，淨賺 $1,223
結算溫度 22.8°C（沒落在區間內）→ 拿回 $0，淨虧 $137
```

**離場的價值**：在第 70 小時鎖定 148% 確定收益，不用賭最後 25% 的機率到底中不中。

## 九、最常見的判斷情境

### 情境 1：pₜ > market_price → 買 YES

```
你算出 pₜ(YES) = 0.15，市場 YES = 0.10 → 正 EV，買 YES
```

### 情境 2：pₜ < market_price → 看 NO 有沒有 edge

```
你算出 pₜ(YES) = 0.15，市場 YES = 0.30
→ YES 負 EV
→ NO：你認為 85%，市場 NO = 0.70
→ NO edge = 0.85 − 0.70 = +0.15 → 買 NO
```

### 情境 3：兩邊都沒 edge → 不做

```
你算出 pₜ(YES) = 0.10，市場 YES = 0.10
→ 兩邊 edge 都是零 → 不做
```

### 情境 4：市場 bid 飆高超過 Cₜ → 賣出

```
有人大量買入，bid = 0.30，Cₜ = 0.25
→ Lₜ ≥ Cₜ → 鎖定利潤
```

### 情境 5：pₜ 下降但 Lₜ < Cₜ → 繼續持有

```
pₜ 跌到 0.08，bid 跌到 0.06
→ Lₜ(0.06) < Cₜ(0.08) → 模型還是認為值 0.08 → 繼續抱
```

### 情境 6：接近結算 → 時間折現幾乎消失

```
τ = 2 小時
Cₜ = 0.15 × exp(−0.0005 × 2) = 0.1499
→ 幾乎等於裸 pₜ，主要看 pₜ 和 Lₜ 的直接比較
```

## 十、拉數據前必讀：10 個最容易踩的坑

### 坑 1（最致命）：結算定義沒對齊

Polymarket 上「倫敦最高溫」用的是哪個氣象站？哪個官方來源？幾點到幾點算一天？
你的預報模型對應的是同一個站點、同一個定義嗎？
差 1-2 度就能讓你的誤差模型從第一步偏掉。

**開始拉數據之前，先搞清楚每個合約的結算來源和定義。**

### 坑 2：時區和時間戳

所有 timestamp 統一 UTC。倫敦有 BST/GMT 切換，巴黎有 CET/CEST 切換。
沒標時區的 timestamp 之後對齊會變成災難。

### 坑 3：lead_hours 的定義

lead_hours = 預報發布時間到結算時間，不是你看到預報的時間到結算時間。
模型可能每 6 小時發布，你拿到的時間又晚 30 分鐘。定義不一致會混入不同新鮮度的預報。

### 坑 4：預報更新版本

同一模型一天出好幾版預報（00Z, 06Z, 12Z, 18Z）。
回測必須只用當時已發布的版本，不能事後用最新版回填。

### 坑 5：市場價格快照頻率

至少每小時存一次完整 orderbook snapshot。
存的是快照瞬間的 best bid/ask，不是一小時平均。

### 坑 6：歷史資料量

每個 (城市, lead_hours) 組合至少要 100 筆。不到 100 筆的先標記不可靠。

### 坑 7：結算結果的取得

每個合約都要存 settlement_outcome（YES=1 / NO=0）。沒有結果就算不了 Brier score。

### 坑 8：費率用真實交易驗證

實際下 2-3 筆小單，確認被收的費用跟公式一致。不要只靠文件。

### 坑 9：第一版只用一個模型

多模型合成是第二版的事。先確認一個模型整條 pipeline 跑得通。

### 坑 10：缺值處理

缺值標記 NaN，不做交易決策。不要用前一筆填補後假裝有資料。

## 十一、拉數據前的 Checklist

### 結算定義
- [ ] 每個合約的結算來源是哪個氣象站？
- [ ] 結算目標是某時點溫度、日最高溫、日最低溫、還是其他？
- [ ] 你的預報目標跟結算定義完全一致嗎？
- [ ] 「一天」的定義是 00:00-23:59 UTC？還是當地時間？

### 時間與版本
- [ ] 所有 timestamp 存 UTC
- [ ] 每筆預報有 forecast_issue_time
- [ ] lead_hours = target_time − forecast_issue_time
- [ ] 回測只用當時已發布的版本

### 氣象資料
- [ ] forecast_temp, observed_temp, error_temp 三欄位都有
- [ ] current_obs_temp（下預測當下的實況溫度）
- [ ] station_id
- [ ] 至少每小時或每次預報更新

### 市場資料
- [ ] market_id + 合約條件
- [ ] yes_price, no_price
- [ ] yes_best_bid, yes_best_ask, no_best_bid, no_best_ask
- [ ] yes_bid_depth_1, yes_ask_depth_1
- [ ] settlement_time, settlement_outcome
- [ ] 至少每小時快照

### 資料量
- [ ] 每個 (城市, lead_hours) 至少多少筆？
- [ ] 不到 100 筆的標記「樣本不足」
- [ ] 總覆蓋至少 3 個月

### 費率驗證
- [ ] 實際下幾筆小單確認費用公式正確
- [ ] 確認天氣市場 fee_rate = 0.025, exponent = 0.5

## 十二、一句話總結

**模型給你機率，市場給你價格。機率大於價格就是正 EV，用 Kelly 控制下多少，用 Lₜ ≥ Cₜ 控制什麼時候走。但在這一切之前，先用 Brier score 確認你的模型真的比市場準，再確認扣掉費用後 edge 還在。**
