# 天氣套利研究規劃

- 建立日期：2026-04-02
- 文件用途：把使用者需求、目前已知來源與下一步研究設計整理成可交接文件

## 研究背景

Polymarket 有一類市場會把某個城市或機場站點在指定日期的最高溫，切成整數攝氏區間來交易。這類市場的特徵是：

- 結算規則清楚
- bucket 是離散整數
- 市場會隨預報更新與日內觀測持續變動

對這種市場而言，真正的研究核心不是「今天天氣如何」，而是：

- 過去某個發布時刻，市場外部資料對未來最高溫怎麼預測
- 到結算時，這個預測與最終值的偏差通常多大

## 使用者需求翻譯

### 需求 1：找不到歷史預報快照就沒辦法做核心模型

使用者要的不是單純歷史實測，而是：

- `T-24h` 時，對 `T日最高溫` 的預測
- `T-12h` 時，對 `T日最高溫` 的預測
- `T-6h`、`T-3h`、`T-1h` 時的預測

如果沒有這種 issue-time forecast snapshot，就無法準確回答預報在不同 lead time 的歷史誤差。

### 需求 2：YES / NO 其實是兩條不同統計線

#### YES 線

- 問題：
  - 到第 N 小時時，當下預報對最終結算溫度的歷史準確率是否已足夠高
- 核心欄位：
  - `forecast_daily_high`
  - `final_daily_high`
  - `lead_hours_to_end_of_day`
- 核心指標：
  - `forecast_error = forecast_daily_high - final_daily_high`

#### NO 線

- 問題：
  - 到第 N 小時時，哪些較高溫度依歷史分布已經非常不合理
- 核心欄位：
  - `observed_temp_now`
  - `max_temp_so_far`
  - `final_daily_high`
- 核心指標：
  - `remaining_rise = final_daily_high - observed_temp_now`
  - 或
  - `remaining_rise_from_max_so_far = final_daily_high - max_temp_so_far`

## 第一版資料結構草案

| 欄位 | 說明 |
|------|------|
| `market_id` | Polymarket 市場識別 |
| `event_date_local` | 市場對應的當地日期 |
| `station_id` | 例如 `EGLC` |
| `snapshot_time_local` | 預報快照當下的當地時間 |
| `snapshot_time_utc` | 預報快照當下的 UTC |
| `forecast_source` | Open-Meteo / NOAA / ECMWF / Met Office 等 |
| `forecast_daily_high` | 當下預報的日最高溫 |
| `observed_temp_now` | 當下實測溫度 |
| `max_temp_so_far` | 截至快照時刻的當日最高溫 |
| `final_daily_high` | 結算真值 |
| `lead_hours_to_end_of_day` | 距離當地日界線的剩餘小時 |
| `forecast_error` | 預報誤差 |
| `remaining_rise` | 從當下溫度到最終高點的剩餘幅度 |
| `bucket_prices` | 各溫度 bucket 的市場價格快照 |

## 來源選型策略

### 方案 A：快速研究版

- 真值：
  - Wunderground `EGLC` 日歷史頁
- 預報快照：
  - Open-Meteo Previous Runs / Historical Forecast
- 適用：
  - 想先驗證研究方向是否成立

### 方案 B：官方 archive 版

- 真值：
  - Wunderground `EGLC` 日歷史頁
- 預報快照：
  - NOAA GFS archive
  - ECMWF historical archive
- 適用：
  - 需要更嚴格的 issue-time 官方 forecast 研究

### 方案 C：自建 forward collection 版

- 真值：
  - Wunderground `EGLC`
- 預報快照：
  - Met Office DataPoint / DataHub 自行定時抓取
- 適用：
  - 從現在開始建立可持續資料庫

## 目前建議順序

1. 固定真值定義與時區邊界。
2. 把第一版資料來源決策寫死，不再混用多條路線。
3. 先做一版 schema 與欄位定義。
4. 再決定程式端怎麼抓、怎麼存、怎麼回測。

## 現階段不做的事

- 不直接寫自動下單。
- 不把策略擴散到跨市場、跨主題、多事件套利。
- 不在還沒確認資料定義前就開始做大量程式施工。
