# 天氣套利資料源整理

- 整理日期：2026-04-02
- 整理目的：把「預報快照」和「結算真值」分開，避免資料層混淆

## 問題定義

這個專案要的不是一般歷史天氣，而是兩條可對齊的時間序列：

1. `歷史預報快照`
   - 某個發布時刻，對未來某一天最高溫的預測值
2. `最終結算真值`
   - Polymarket 規則最終採用的日最高溫

## 來源分級

| 來源 | 類型 | 最適合用途 | 不足之處 | 目前判斷 |
|------|------|------------|----------|----------|
| Polymarket 市場頁 | 規則 | 確認 bucket、站點、結算邏輯 | 不提供預報快照 | 必讀 |
| Wunderground `EGLC` 日歷史頁 | 真值 | 當最終結算真值 | 不是歷史預報 archive | 必用 |
| Wunderground About Data | 說明文件 | 了解其 forecast / observations 來源與更新頻率 | 未看到可回放舊版預報的公開 archive | 參考 |
| Met Office DataPoint | 官方 API | 自現在起存英國站點 forecast / observation | 不適合長期 historical backfill | 次要 |
| Met Office DataHub | 官方 API / FAQ | 確認觀測與 model 保留範圍 | observations 48h、model 2 days | 次要 |
| Open-Meteo Previous Runs | 歷史 forecast API | 快速取得過去 run 對未來的預測 | 不是 WU 原版 forecast | 第一優先 |
| Open-Meteo Historical Forecast | 歷史 forecast API | 建立 lead time 誤差資料集 | 來源是整合模型，不是結算來源 | 第一優先 |
| NOAA GFS archive | 官方模式 archive | 嚴格回測、長期歷史 | 需自己做格點處理與 daily max | 第二優先 |
| ECMWF historical archive | 官方模式 archive | 高品質歷史 forecast | 取得門檻較高 | 第二優先 |

## 對使用者需求的映射

### 使用者需求 A：哪些溫度區間在第 N 小時幾乎不可能

- 需要：
  - 當下觀測溫度
  - 當日截至目前最高溫
  - 最終日最高溫
- 可推導：
  - `remaining_rise = final_daily_max - current_or_max_so_far`
- 用途：
  - 做 `NO` 方向的風險上界與分位數分析

### 使用者需求 B：第 N 小時的預報是否已足夠準，可以偏向 YES

- 需要：
  - issue-time forecast snapshot
  - 最終日最高溫
- 可推導：
  - `forecast_error = forecast_daily_high - final_daily_high`
- 用途：
  - 做 `YES` 方向的預報準確率分析

## 目前最合理的研究搭配

### 快速研究版

- 真值：
  - Wunderground `EGLC` 日歷史頁
- 預報快照：
  - Open-Meteo Previous Runs / Historical Forecast
- 優點：
  - 快、可落地、方便先做統計
- 缺點：
  - 預報來源不是 Polymarket 規則直接採用的 Wunderground forecast

### 官方 archive 版

- 真值：
  - Wunderground `EGLC` 日歷史頁
- 預報快照：
  - NOAA GFS archive
  - ECMWF historical archive
- 優點：
  - 研究更嚴格，歷史 run 比較完整
- 缺點：
  - 整理與工程成本明顯更高

### forward collection 版

- 真值：
  - Wunderground `EGLC`
- 預報：
  - Met Office DataPoint / DataHub 自行每小時快照
- 優點：
  - 後續資料品質可自己掌控
- 缺點：
  - 無法立刻補回很久以前的歷史
