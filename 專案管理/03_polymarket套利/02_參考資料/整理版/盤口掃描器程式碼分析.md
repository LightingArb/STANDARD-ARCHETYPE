## 一、這個系統在做什麼（白話版）

整個系統就像一個自動巡邏員，每次執行做七件事：

1. **去 Polymarket 問：現在有哪些天氣題目？** → 拿到一大堆題目
2. **看懂每個題目在問什麼** → 「London 3/20 最高溫 14°C？」
3. **把同一天同一城市的所有格子放在一起** → 例如 London 3/20 有 11 格
4. **查出每個城市結算時看的是哪個氣象站** → London 看 EGLC
5. **去問天氣預報現在預測幾度** → 原本用 Open-Meteo + Visual Crossing，後來改成直接爬 Wunderground
6. **比較預報和盤口** → 找出價格可能有問題的格子
7. **把結果推到 Telegram** → 讓你在手機上看

## 二、每個程式檔案做什麼

### 值得提取的（3 個）

| 檔案 | 白話說明 | 為什麼值得 |
|------|---------|-----------|
| `polymarket_client.py` | 去 Polymarket 抓所有天氣題目 | 知道 API 怎麼呼叫、怎麼翻頁、怎麼過濾「highest temperature in」 |
| `market_parser.py` | 把英文題目拆成：城市、日期、溫度、類型 | 正規表達式已經處理了 4 種格式（精確、以上、以下、範圍），自己寫很容易漏 |
| `city_mapping.json` | 38 個城市的氣象站代碼、座標、時區 | 直接拿來用，不用自己查 |

### 可以參考但不直接用的（4 個）

| 檔案 | 白話說明 | 為什麼不直接用 |
|------|---------|---------------|
| `market_grouper.py` | 把格子按城市+日期分組 | 很簡單（45 行），邏輯可以直接抄，但未來可能要改分組方式 |
| `rules_enricher.py` | 查出每個城市的 Wunderground 結算網址 | 邏輯簡單，但我們可能要加更多欄位 |
| `analyzer.py` | 比較預報和盤口，還會查 Order Book 深度 | **Order Book 深度查詢是新發現**，我們之前沒考慮到，但分析邏輯太淺要重寫 |
| `output.py` | 格式化輸出到 Telegram | 格式以後再設計 |

### 有趣但需要 Playwright 的（1 個）

| 檔案 | 白話說明 | 注意事項 |
|------|---------|---------|
| `wunderground_client.py` | 用 headless browser 爬 Wunderground 的當下溫度和預測最高溫 | 需要裝 Playwright，而且 WU 頁面結構可能會變。但這是**唯一能拿到 WU 自己的預報值**的方式 |

## 三、新發現（之前不知道的）

### 3.1 Polymarket 盤口結構

從實際資料看到，每個城市每天的盤是這樣的：
- **11 個格子**：9 個精確值 + 1 個「以上」+ 1 個「以下」
- 精確值的範圍大約跨 9 度（例如 4°C ~ 12°C）
- 結算時間是 **UTC 12:00**（不是當地午夜！）

London 3/26 的例子：
> 3°C or below | 4°C | 5°C | 6°C | 7°C | 8°C | 9°C | 10°C | 11°C | 12°C | 13°C or higher

### 3.2 Order Book 深度查詢

`analyzer.py` 裡有一段會去查 Polymarket 的 Order Book：
- API：`POST https://clob.polymarket.com/books`
- 可以一次查 200 個 token 的掛單深度
- 回傳每個格子的 YES/NO 兩邊各有多少 USDC 掛單
- **這很重要**：知道深度才能判斷「有沒有足夠的量讓我下注」

### 3.3 天氣資料來源演變

這個系統經歷了兩個版本：
- **舊版**：用 Open-Meteo（免費）+ Visual Crossing（付費，你有 API key）
- **新版**：改成直接爬 Wunderground（用 Playwright）

改的原因可能是：Wunderground 是結算來源，直接用它的預報比用其他來源更準確。

### 3.4 Visual Crossing 是第三個預報來源

除了 Open-Meteo 和 Wunderground，你還註冊了 Visual Crossing。這是一個付費天氣 API，特點是：
- 需要 API key
- 回傳格式簡單（`tempmax` / `tempmin`）
- 可以作為第三方預報來對比

## 四、結算規則確認

從程式碼中確認的重要規則：

| 項目 | 內容 |
|------|------|
| 結算來源 | Weather Underground 歷史頁面 |
| 結算精度 | 整數 °C（四捨五入） |
| 結算時間 | **UTC 12:00**（從 preview 資料看到） |
| 站點 | ICAO 機場代碼（London = EGLC） |
| 格子數 | 每個城市每天 11 格（9 exact + 1 higher + 1 below） |

**重要：結算時間是 UTC 12:00，不是當地午夜。** 這跟我們之前假設的可能不同，需要確認。

## 五、建議提取順序

1. **立即提取**：`polymarket_client.py` + `market_parser.py` + `city_mapping.json`
   → 這三個合起來就能抓到所有天氣盤的即時盤口

2. **之後提取**：`analyzer.py` 裡的 Order Book 深度查詢邏輯
   → 等我們開始做真實交易判斷時再用

3. **備用**：`wunderground_client.py`（Playwright 爬蟲）
   → 如果需要拿 WU 自己的預報值時再用
