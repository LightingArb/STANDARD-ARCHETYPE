## 議題 1：WU 數據獲取

- WU 的 "Max Temperature" 跟逐時觀測表格的 max 在測試當天一致（55°F），但需更大樣本確認
- 系統架構師建議：攔截 WU 前端的 XHR request，找到底層 JSON endpoint，比爬 DOM 穩定

3. 合格標準：整數攝氏結算下不一致率 < 3%

## 議題 2：交易策略數學基礎

### 核心公式

```
EV = P_true - price_paid
```

EV > 0 等價於你的機率估計比市場準。

### 使用者的交易本質

不是方向交易，是波動率交易：
- 進場：結算前 80h，市場效率低
- 獲利：市場修正定價（價格回歸合理值）
- 退出：EV 上升時賣出獲利，或持有到結算
- 即使沒有 edge，也可在極端值提供流動性（Polymarket 政策對流動性供應友善）

### 4 種可能的 Edge

1. **時間 edge**（最可能）：80h 前市場效率低，做市商 bot 還沒關注
2. **校正 edge**：偏差校正做得比市場好
3. **資訊 edge**：19 模型 ensemble 的資訊量 > 市場
4. **速度 edge**：比市場更快反應新的預報 run

### Kelly Criterion

二元結果：`f* = (p × b - q) / b`

多 bucket 版本是凸優化問題，需 `scipy.optimize.minimize`。

建議用 **half-Kelly**（對沖模型不確定性）。

### PE 值定義

```
PE = price / P_true
PE < 1 → 市場低估 → 買入
PE > 1 → 市場高估 → 賣出
```

### 中途賣出決策

是最優停止問題（類似美式選擇權）：
- forward EV < 交易成本 → 賣出
- forward EV > 持有閾值 → 繼續持有

### 未解決

- 80h 前的定價效率到底比 24h 前差多少？（需歷史盤口資料）
- Polymarket 的交易成本（spread + fee）到底多少？
- 多結果 Kelly 的數值求解方案
- 11 個 bucket 的 YES 價格總和通常是多少？（市場效率指標）

## 議題 3：系統性偏差校正

### 3 層校正框架

| 層級 | 方法 | 參數量 | 何時使用 |
|------|------|--------|---------|
| Layer 1 | 全局偏差 | 19 | 必做 |
| Layer 2 | 按 lead time | 133 | 強烈建議 |
| Layer 3 | 按季節 × lead time | 532 | 需交叉驗證確認 |

### 模型合併方法

推薦 **Bayesian Model Averaging (BMA)** 或 **反方差加權**：
- 權重動態調整（近期表現好的模型權重更高）
- 先檢查模型間誤差相關矩陣（高相關的合併為一組）

### Overfitting 警告

- 19 模型 × 4 季節 × 7 lead time = 532 參數 → 幾乎一定 overfit
- 安全做法：限制複雜度，用 AIC/BIC 做模型選擇
- 奇偶年交叉驗證

### 偏差穩定性檢驗

- CUSUM test：偏差是否有結構性變化
- Kruskal-Wallis test：各月偏差是否相同
- Ljung-Box test：校正後殘差是否為白噪音
- 合格標準：校正後 RMSE < 校正前 80%

### 未解決

- 19 個模型的偏差自相關結構？
- 模型間誤差相關矩陣？
- BMA vs 反方差加權哪個更好？
- Open-Meteo 後處理 vs NOAA 官方 GFS 差異多大？

## 議題 4：滾動窗口 / 切線計算

### 核心公式

```
MSE(N) = (delta × N / 2)^2 + sigma^2 / N
最佳 N* = (2 × sigma^2 / delta^2)^(1/3)
```

### 建議

- **EWM（指數加權移動平均）優於 hard window**
- alpha=0.997 對應有效窗口 ~333 天
- 用 expanding window cross-validation 選最佳 alpha
- 模型升級是比氣候漂移更大的風險 → 用 change point detection（PELT 演算法）

### 收斂分析

- 有效樣本量需 > 225（考慮自相關後原始樣本需 ~1000）
- N=1000 大致在正確量級
- 溫度不適合取 log（可以是 0 或負數），直接比較 RMSE

### 未解決

- 每個模型的偏差自相關結構？
- 過去 5 年有哪些模型經歷重大升級？
- 統一 alpha vs 每模型獨立 alpha 哪個更好？

| 9 | 時間結構套利 | 中（連續兩天溫度高相關） | 做市商可能已矯正 |

### Burn Analysis + Bayesian 框架

```
P(bucket_k | forecast) ∝ P(forecast | bucket_k) × P(bucket_k)
P(bucket_k) = 歷史同期先驗
P(forecast | bucket_k) = 預報誤差分布
```

即使 API 全掛，還有先驗可用。

### Black-Scholes 類比

| Greeks | 天氣市場對應 | 意義 |
|--------|-----------|------|
| Delta | dP/dT_forecast | 預報變 1°C，機率變多少 |
| Theta | dP/dt | 時間過去，不確定性縮小 |
| Vega | dP/d_sigma | 波動率估計變了 |
| Gamma | d²P/dT² | bucket 邊界附近 Delta 最大 |

### ML 方法

**目前階段不建議使用。** 365 天 / 29 features = overfit。等累積 2 年 OOS 實績再考慮。

**唯一推薦：Isotonic Regression** 作為校準層（參數少、不易 overfit）。

## 議題 6：風險控制

### 硬限制（寫進程式碼）

```
單筆 < 5% 資金
日虧損 < 10% 資金
總暴露 < 30% 資金
最低 EV 閾值 > 2%
連虧 10 次 → 暫停
```

### 上線流程

| 階段 | 時長 | 資金 | 通過標準 |
|------|------|------|---------|
| Paper Trading | 30 天 | $0 | Brier Score < 0.20 |
| Phase 1 最小規模 | 60 天 | $500 | 無技術故障, ROI > -10% |
| Phase 2 正常規模 | 持續 | $3,000 | 月度審核 |
| Phase 3 全速 | Phase 2 成功 6 月後 | Kelly 決定 | — |

### 熔斷機制

- 連虧 10 次 → 暫停 24h
- 日虧損 > 10% → 暫停至次日
- Brier Score > 0.25 → 暫停至人工審核
- 資料源離線 > 2h → 暫停新交易

### Overfit 防範

- 訓練/驗證/測試集分割（2022-2024 / 2025H1 / 2025H2）
- 每 50 個交易做 calibration check
- 逐步加倉（$10 → $50 → $100 → Kelly）

### 對沖方式

- 跨相鄰 bucket 展倉（類似 straddle）
- 不是同時買 YES 和 NO（bucket 互斥，不是對沖）

### 使用者補充

- 沒有 edge 也可以在極端值提供流動性（期望值仍為正）
- Polymarket 政策越來越對流動性供應友善
- 不用擔心「沒有交易機會」的情況
