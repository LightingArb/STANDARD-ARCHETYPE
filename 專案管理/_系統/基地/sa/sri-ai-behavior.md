## §1 日常知識調用總則

### 1-1. 預設讀取範圍

日常所有指令，預設只可引用以下本地資料：

| 類別 | 來源 |
|------|------|
| 正式產物 | 14/15；20/21/30/31 僅程式設計模式 |
| 中間層 | 10/11/12/13 |
| 團隊 | `01_團隊/角色會議紀錄.md`、必要時讀對應 `角色_*.md` |
| 人格 | 當前容器 `01_團隊/人格_*.md` |
| 參考資料 | `02_參考資料/資料地圖.md`、`02_參考資料/資料館長.md` |
| 文件產出 | `03_文件產出/產出索引.md` 與必要的交付檔 |
| 路由 | `_系統/基地/系統總覽.md` |
| MCP 設定 | `_系統/基地/mcp/README.md`、`servers.json`、`policies.md` |

### 1-2. 人格讀取層級（寫死）

| 上下文位置 | 讀取 |
|-----------|------|
| 在主專案 | 讀主專案 `01_團隊/人格_*.md` |
| 在主專案的探索 | 仍讀主專案 `01_團隊/人格_*.md`（探索無獨立團隊） |
| 在子專案 | 讀該子專案 `01_團隊/人格_*.md` |
| 在子專案的探索 | 仍讀該子專案 `01_團隊/人格_*.md` |

人格讀取不得跨層自動混用，除非使用者明確指定。

**Fallback：** 若當前容器 `01_團隊/` 沒有人格卡，提示使用者走 Furnace 取用流程，不自動降級到其他容器。

### 1-3. 禁止直接調用

以下流程可直接讀 Furnace：

- `取用人格`
- `新增一般專案`
- `新增程式專案`
- `升級專案 程式設計`
- `新增沙盒`（子專案）
- `角色帶依賴自動配發`
- L3 客製流程

其餘日常指令不得直接調用：

- Furnace repo 的任何非上架檔
- Furnace `output/raw-cards/` 原始素材
- Furnace 流派工作區（`codex/`、`claudecode/`）
- 未上架的鍛造中間產物

若本地資料不足以回答，AI 只能做以下三件事之一：

1. 明確指出缺哪些資料。
2. 建議先整理回本地 md。
3. 建議使用 `取用人格`、`角色討論` 或進入 Harness。

## §2 自動更新與讀取模式

### 2-1. 10-15 自動更新機制

v4 開始，AI 不能再把 `對話整理` 視為唯一整理入口。日常對話中，AI 要在背景判斷何時自動分流。

| 規則 | 值 |
|------|---|
| 最小間隔 | 3 輪 |
| 最大間隔 | 10 輪 |
| 智慧觸發 | 偵測到重要結論、方向轉折、問題浮現時可提前 |
| 手動觸發 | 使用者說 `對話整理` 時立即執行 |

**自動分流對應：**

- 想法、靈感、備案 → `_系統/11_想法暫存區.md`
- 需回答或驗證的問題 → `_系統/12_待確認問題.md`
- 使用者明確確認的結論 → `_系統/13_已確認結論.md`
- 方向性決策、願景、成功條件 → `總方案.md`
- 本輪對話摘要 → `_系統/10_對話整理.md`
- 重大方案變更 → `_系統/15_方案變更紀錄.md`

**限制：**

1. 自動更新只做即時分流，不做跨層融合。
2. `整理已確認結論到總方案`、`整理已確認結論到施工藍圖` 仍需使用者顯式觸發。
3. 探索內的自動更新只寫探索本地，不回寫主線。

### 2-2. 快速接續模式（進入專案預設）

```
1. 靜默讀取 _系統/基地/系統總覽.md
2. 靜默讀取當前正式容器 總方案.md metadata（專案模式）
3. 靜默讀取當前容器 01_團隊/人格_*.md
4. 靜默讀取 _系統/10_對話整理.md
5. 若 10_有「快照」區，先讀快照
6. 補讀 01_團隊/角色會議紀錄.md
7. 用一句話說明現況，建議下一步
```

**Fallback：自動升級為完整讀取模式**

觸發條件（任一即升級）：

- _系統/10_對話整理.md 為空白
- 最後更新超過 7 天
- 內容少於 5 行
- 使用者說「重新整理一下現況」

升級後讀取：

- 系統總覽 + 總方案.md + `01_團隊/人格_*.md`
- _系統/20_施工藍圖.md（若程式設計模式且存在）
- _系統/30_施工進度.md（若程式設計模式且存在）
- 13 → 12 → 11 → 10 → 15
- `01_團隊/角色會議紀錄.md`
- 若需要補理解外部資料，再讀 `02_參考資料/資料館長.md`
- 若需要補理解交付物，再讀 `03_文件產出/產出索引.md`

### 2-3. 探索版快速接續模式

進入沙盒探索時：

```
1. 靜默讀取系統總覽
2. 靜默讀取所屬正式容器 總方案.md metadata（專案模式）
3. 靜默讀取所屬正式容器 01_團隊/人格_*.md
4. 靜默讀取當前探索 _系統/10_對話整理.md
5. 若有快照先讀快照
6. 補讀所屬正式容器 01_團隊/角色會議紀錄.md
7. 視需要補讀所屬正式容器 總方案.md
8. 一句話說明「目前在 [正式容器] 的 [探索名稱]」+ 現況 + 下一步
```

Fallback 觸發條件與升級內容同主專案版。

## §3 各指令 AI 行為

> 指令清單、用途與觸發條件見 `sri-core.md` §2。以下只補充 AI 的讀取順序、判斷規則與回應節奏。

### 3-1. 專案與沙盒操作

#### 新增一般專案

1. 在 `[根目錄]/` 下依下一個編號建立專案資料夾。
2. 建立 _系統/10-15、`01_團隊/`、`02_參考資料/`、`03_文件產出/`、`05_沙盒區/`、`_系統/99_舊檔案封存區/`。
3. 從 Furnace 複製規劃角色組到 `01_團隊/`，再掃角色依賴配發人格卡。
4. 建立空白 `01_團隊/角色會議紀錄.md`。
5. 同步更新系統總覽狀態區。
6. 不先回報建檔細節，直接進第一輪真正有用的討論。

#### 新增程式專案

1. 在一般專案基礎上，再建立 20/21/30/31 與 `04_主專案程式/`。
2. 配發完整角色組（01-04 + 11-13）與依賴人格卡到 `01_團隊/`。
3. 建立 `04_主專案程式/README.md`。
4. 同步更新系統總覽狀態區。
5. 直接進第一輪規劃討論，不先列目錄清單。

#### 進入專案

1. 預設走快速接續模式。
2. 一句話回報目前狀態（含模式與階段）。
3. 建議下一步。
4. 自然語句若已能定位到專案，視為進入專案，不要求重打口令。

#### 專案說明

依序讀取：14 → 若程式設計再讀 20/30 → 13 → 12 → 10 → 11 → 15 → 團隊會議紀錄 → 視需要讀資料館長與產出索引。

輸出至少包含：

- 這個專案在做什麼
- 整理到哪裡
- 目前還缺什麼
- 建議下一步

#### 升級專案 程式設計

1. 若已是程式設計模式，直接回報。
2. 補建 20/21/30/31、完整角色組與 `04_主專案程式/`。
3. 把 總方案.md 的 `專案模式` 改為 `程式設計`。
4. 掃新角色依賴，補入缺少人格卡到 `01_團隊/`。
5. 同步更新系統總覽。

#### 新增沙盒

1. 只能在已進入某正式容器時使用。
2. 詢問類型：探索 / 子專案。
3. 一般模式主專案只能開探索。
4. 探索：建 `05_沙盒區/[字母]_名稱/`（10-13 + `02_參考資料/`）。
5. 子專案：建完整結構並配發完整團隊。
6. 建完直接進入第一輪討論。

#### 進入沙盒

1. 若未定位正式容器，先問主專案。
2. 探索走探索版快速接續；子專案走主專案快速接續。
3. 探索無獨立團隊，角色與人格都讀所屬正式容器。
4. 子專案的團隊與人格只讀子專案本地 `01_團隊/`。

#### 回到主專案

1. 清除當前沙盒上下文。
2. 回到所屬正式容器。
3. 不刪除任何沙盒資料。

#### 比較所有沙盒

1. 定位當前正式容器。
2. 掃描 `05_沙盒區/`。
3. 主讀各沙盒 `_系統/13_已確認結論.md`。
4. 視需要補讀 `_系統/12_待確認問題.md` 與 `02_參考資料/資料館長.md`。
5. 輸出比較表：名稱、類型、結論、優點、風險、是否建議回流。

#### 回流沙盒到主專案 / 自動回流

1. 讀指定或全部沙盒的 13。
2. 列出建議回流項。
3. **必須等待使用者確認。**
4. 寫入所屬正式容器 `_系統/12_待確認問題.md`，不得直接寫入 13/14/20/30。
5. 回流成功後，在沙盒 13 補註記。

#### 升級沙盒 / 降級沙盒

1. 升級前確認主專案是程式設計模式。
2. 升級時保留既有 10-13 + `02_參考資料/` 內容，補齊完整結構。
3. 降級時保留 _系統/10-13 + `02_參考資料/`，其餘移入 `_系統/99_舊檔案封存區/`。
4. 兩者都必須先取得使用者確認。

### 3-2. 討論、整理與正式化

#### 討論專案

1. 靜默讀取 14 metadata + `01_團隊/人格_*.md` + 10 + 團隊會議紀錄。
2. AI 自動判斷規劃 / 架構 / 施工 / 運維階段。
3. 依矩陣提問規則發問。
4. 不重複問已在 13 的結論。
5. 若對話達到 3-10 輪觸發點，自動分流寫入 10-15。

#### 角色討論

1. 讀取 14 + 13 + 12 + 團隊會議紀錄 + 必要角色檔。
2. 觸發 AI 角色之間的發散與收斂討論。
3. 使用者看到：結論摘要 + 分歧點。
4. 完整過程記入 `01_團隊/角色會議紀錄.md`。
5. 半自動模式下，結論仍需使用者確認才生效。

#### 角色審查

1. 比 `角色討論` 更正式、更完整。
2. 依序輸出各角色觀點，再做總結。
3. 結束後更新會議紀錄與對應角色檔。

#### 發散思考

1. 讀取 10 + 11 + 12 + 13 + 14 + 團隊資料。
2. 以多角色視角擴散方向，不急著收斂。
3. 產物優先寫到 11；值得確認的條目回流到 12 並加 `[想法回流]`。

#### 收斂思考

1. 讀取最近一次發散產物（11 / 12 / 角色會議紀錄）。
2. 把發散結果收束成可採取的方案、風險清單與下一步。
3. 若已到正式化門檻，提示使用者進行 12→13 或 13→14。

#### 延伸想法

1. 仍保留，定位是局部向外擴展或向內深挖。
2. 與 `發散思考` 的差異在於範圍較小、可針對單一議題。

#### 討論想法暫存區 / 討論待確認問題

1. 以 1-3 條為一批。
2. `討論想法暫存區` 的輸出只能是：保留 / 延伸後保留 / 回流到 12 / 暫不採用但保留。
3. `討論待確認問題` 要更新每題狀態：`#尚未回答` / `⚠ 使用者決定不處理` / `✓ 議題關閉`。

#### 壓縮對話

1. 更新 _系統/10_對話整理.md 的快照區。
2. 不取代自動分流，只是壓短版接續內容。
3. 視需要更新系統總覽狀態區。

#### 對話整理

1. 立即執行一次完整分流。
2. 寫入 10，並把想法 / 問題 / 結論 / 方向分別寫入 11 / 12 / 13 / 14。
3. 若在探索，僅寫探索本地。

#### 整理待確認問題到已確認結論

1. **剪下貼上**，不是複製。
2. 不得跳過 12，直接把聊天內容灌進 13。
3. 探索內的 12→13 只在探索本地完成。

#### 整理已確認結論到總方案

1. **必讀雙主檔：** 13 + 14。
2. 視需要補讀 12 檢查未決衝突。
3. 判定變更等級；大改先取得確認。
4. 融合、比對、衝突檢查後寫入 14。
5. 同步補 15。
6. 探索不得直接寫主專案 14。

#### 整理已確認結論到施工藍圖

1. 先確認程式設計模式。
2. **必讀：** 13 + 14 + 20。
3. 判定變更等級；大改先確認。
4. 融合後寫入 20。
5. 同步補 21。
6. 探索不得直接寫所屬正式容器 20。

#### 整理參考資料

1. 掃描 `02_參考資料/`。
2. 比對實體與資料地圖。
3. 未整理項目分類、摘要、歸類。
4. 更新資料地圖 + 資料館長。
5. 若有可直接交付的成果，改寫入 `03_文件產出/` 而不是混在 02。

#### 自動歸檔行為

1. AI 產出或建立檔案時，先判斷應放哪個正式資料夾，以及是否需要建立子目錄。
2. 第一次遇到新類型檔案時，先向使用者確認歸檔位置與分類方式。
3. 使用者確認後，同類型檔案依既有慣例自動放置，不再重複詢問。
4. 系統更新時，定期回顧目前的歸檔慣例是否仍合理，必要時提出調整建議。

### 3-3. 施工、團隊與 MCP

#### 開始施工

1. 確認程式設計模式。
2. 讀取 20 + 30 + 31。
3. 閘門檢查：20 狀態必須為「已確認可施工」。
4. 若接下來會改 `04_主專案程式/` 內非 `.md` 檔案，先停下來列出修改清單。

#### 執行單元操作

- `新增執行單元`：依藍圖在 `04_主專案程式/` 內建立新執行單元。
- `進入執行單元`：主讀該單元 README.md。
- `執行` / `查看輸出`：依 README 約定執行並回報結果。
- `升級執行單元` / `降級執行單元`：必須先取得使用者確認。
- 執行單元間禁止跨單元 import，AI 不得自動建立耦合。

#### 取用人格 / 列出團隊 / 移除人格

- `取用人格`：讀 Furnace `10_老闆.md` → 列出上架清單 → 使用者選擇 → 複製到 `01_團隊/`。
- `列出團隊`：掃描當前容器 `01_團隊/`，同時檢視人格與角色。
- `移除人格`：確認使用者指定後，從 `01_團隊/` 移除或封存。

#### 列出 MCP 伺服器 / 啟用 MCP / 停用 MCP

1. 讀取 `_系統/基地/mcp/servers.json` 與 `policies.md`。
2. `列出 MCP 伺服器`：列出名稱、用途、權限等級、是否首次連線。
3. `啟用 MCP`：若為首次連線或權限升級，必須先停確認。
4. `停用 MCP`：只更新狀態，不刪除設定。
5. MCP 產物路由固定：
   - 原始輸入 → `02_參考資料/原始素材/`
   - AI 消化後分析 → `03_文件產出/`
   - 實際程式整合 → `04_主專案程式/`（需程式碼閘門）

#### 完整初始化 / 系統更新

完整步驟見 `sri-core.md`。AI 執行時的補充規則：

- 路徑校正是第一步，不得跳過。
- 系統更新必做清單必須先讀再執行。
- 補缺檔只補空白模板，不覆蓋既有正式內容。
- 舊路徑遷移採搬移封存，不採刪除。

## §4 角色系統與 Harness 邊界

### 4-1. 角色組總覽

| 階段 | 啟用角色 |
|------|---------|
| 一般模式（規劃） | 角色_01_策略決策、角色_02_創意總監、角色_03_使用者研究、角色_04_技術聯絡 |
| 程式設計（規劃） | 同上 |
| 程式設計（架構） | 角色_01_策略決策、角色_11_系統架構師、角色_12_安全風控、角色_13_品質驗證 |
| 程式設計（施工） | 不啟用角色系統；需多角色深審時回架構或先停確認 |

### 4-2. 角色討論機制

**內部推理層（常駐）：**

- 角色觀點在內部持續生效。
- AI 可主動建議開角色討論。
- 每輪結束可更新 `01_團隊/角色會議紀錄.md`。
- 有明確角色判斷時可靜默更新對應角色檔。

**外顯輸出層（條件式）：**

只有以下任一條件才以角色標籤呈現：

- 角色之間出現真正的差異視角或意見分歧
- 有明確風險衝突或可行性阻斷
- 使用者主動要求 `角色討論` 或 `角色審查`
- 使用者說「讓角色出來說說」

### 4-3. Harness 自主權邊界

| 動作 | 可自主 | 補充 |
|------|--------|------|
| 10-15 討論層讀寫 | 是 | 含自動更新 |
| `01_團隊/角色會議紀錄.md` 更新 | 是 | 含角色討論摘要 |
| `02_參考資料/` 整理與分類 | 是 | 不刪原始資料 |
| `03_文件產出/` 產出新文件 | 是 | 交付物可直接寫入 |
| 上網抓資料 | 是 | 仍需遵守高風險提醒 |
| 角色之間互相討論 | 是 | 可自主觸發 |
| 決策、發散、收斂 | 是 | 限 `.md` 範圍 |
| 改 `04_主專案程式/` 非 `.md` 檔案 | 否 | 必須先停確認 |
| 刪除 / 搬移 / 覆蓋大量檔案 | 否 | 必須先停確認 |
| MCP 首次連線或權限升級 | 否 | 必須先停確認 |

### 4-4. 程式碼修改閘門

修改 `04_主專案程式/` 內任何非 `.md` 檔案之前，AI 必須：

1. 停下來。
2. 列出要改什麼、為什麼改。
3. 等使用者確認「可以改」。
4. 確認後才執行。

同一批次確認只覆蓋當前那一批修改，下一批新的程式碼修改要重新確認。

## §5 Furnace 對接規則

### 5-1. 定位

Furnace（`熔爐百相/`）是 AI 人格鍛造工坊。SRI 只從 Furnace 的上架區取用成品，不介入鍛造過程。

### 5-2. 取用規則

**唯一取用來源：**

- `熔爐百相/鍛造間/人格/`
- `熔爐百相/鍛造間/角色/`

**取用方式：** Pull on demand。複製進專案 `01_團隊/` 後即與 Furnace 斷開同步。

### 5-3. 角色帶依賴自動配發

新增專案 / 升級專案 / 新增子專案時：

1. 依模式從 Furnace 複製角色卡到 `01_團隊/`（前綴 `角色_`）。
2. 掃描每張角色卡的預設人格。
3. 聯集去重所有需要的人格卡。
4. 複製到 `01_團隊/`（前綴 `人格_`）。
5. 建立空白 `01_團隊/角色會議紀錄.md`。

### 5-4. L3 客製人格

當現有上架人格無法覆蓋專案領域時：

1. AI 讀 Furnace `30_人格熔爐.md`。
2. 走客製鍛造流程。
3. 產出直接存入專案 `01_團隊/人格_3xx_名稱.md`。
4. 不上架 Furnace。

## §6 AI 專屬限制

### 6-1. 不報告內部過程

- 讀 SRI 是內部動作，不對使用者逐檔回報。
- 不說「我正在載入 X」。
- 直接輸出讀完後的判斷。

### 6-2. 不展示內部素材

- 不把 Furnace raw-cards 內容直接倒給使用者。
- 不展示 SRI 內部路由細節。
- 不展示 MCP 設定中的敏感細節。

### 6-3. 不越權

- 不得把 10/11/12/02/03 當作正式結論。
- 不得把一般聊天自動升格為正式整理動作。
- 不得在未經使用者確認的情況下執行大改。
- 不得繞過 SRI 的寫入邊界、程式碼閘門與 MCP 邊界。

### 6-4. 不跨層

- 人格讀取不得跨層自動混用。
- 探索討論結果不得直接寫到正式容器正式產物。
- 探索角色觀點寫回正式容器時必須加 `[沙盒-X]` 標籤。
- 探索結論不得直接跳進正式產物，必須走回流路徑。

### 6-5. 日常不碰 Furnace

- 只有 `取用人格`、自動配發、L3 客製流程可讀 Furnace。
- 不得繞過 Furnace 控制檔直接操作鍛造間。
