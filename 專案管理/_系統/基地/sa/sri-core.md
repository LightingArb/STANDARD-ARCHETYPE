## §1 系統定義

### 1-1. 語言規則

- 一律使用繁體中文
- 一律使用台灣用詞與中華民國用詞
- 對外文案、模板、指令說明、產物標題皆依此規則

### 1-2. 系統定位

**SRI 是專案管理 Harness，Furnace 是人格鍛造工坊。兩者獨立，只有取用邊界。**

```
SRI（專案管理）                Furnace（人格鍛造）
┌─────────────────┐           ┌─────────────────────────┐
│ 結構、流程、規則  │           │ 鍛造間/人格/（人格卡）    │
│ 閘門、階段、安全  │ ←─取用──  │ 鍛造間/角色/（角色卡）    │
│ 中間層、正式產物  │           │ Aspects-of-the-Crucible  │
└─────────────────┘           └─────────────────────────┘
       ↑ 複製進來就斷開
       │ 各專案獨立使用
```

- Furnace 位置：與 SRI 工作目錄同層的 `熔爐百相/` 資料夾
- 取用入口：`熔爐百相/10_老闆.md`
- 上架人格：`熔爐百相/鍛造間/人格/`
- 上架角色：`熔爐百相/鍛造間/角色/`
- L3 客製入口：`熔爐百相/30_人格熔爐.md`
- 角色鍛造入口：`熔爐百相/31_角色熔爐.md`

**Harness Engineering 核心原則（Anthropic + OpenAI 2025-2026）：**
1. Agent = Model + Harness — SRI 就是 Harness
2. 生成器和評估器要分開
3. 文件系統是最基礎的記憶原語
4. 文檔是給 Agent 看的，不只是給人看的
5. 人的角色是設計 Harness，不是寫內容

**三種使用模式：**

| 態 | 正式名稱 | 入口 | 核心行為 |
|----|---------|------|---------|
| 第一態 | 純人工 | `_系統/基地/專案管理使用手冊.md` | 人類按結構手動操作，AI 最多當顧問 |
| 第二態 | 指令驅動半自動 | 指令.md | 使用者下指令，AI 執行；大事小事都確認 |
| 第三態 | 閘門式自主推進（Harness） | 大絕招.md | AI 自主決策/發散/收斂；使用者只在硬閘門確認 |

### 1-5. 沙盒區結構

沙盒區（`05_沙盒區/`）統一管理所有分流。

**兩種沙盒類型：**

| 類型 | 前綴 | 結構 | 用途 |
|------|------|------|------|
| 探索 | 字母（A_、B_、C_） | 10-13 + 02_參考資料/ | 輕量比較、測試方向 |
| 子專案 | 數字（01_、02_） | 完整架構（跟主專案一樣） | 正式子系統開發 |

**轉換規則：**
- 探索 → 子專案（升級）：補齊完整架構
- 子專案 → 探索（降級）：多餘檔案移入 _系統/99_舊檔案封存區/
- AI 執行轉換前必須取得使用者確認

**子專案固定為程式設計模式。** 一般模式專案的沙盒區只能開探索，不能開子專案。只有程式設計模式主專案可以開子專案。

**可遞迴。** 子專案內可再開 05_沙盒區/，深度不建議超過 2 層。

### 1-6A. 04_交付與運行區

`04_主專案程式/` 是**交付與運行區**，不是單純的程式碼存放區。

- 04_ 內每個子資料夾都是一個執行單元
- 執行單元分兩種：探索版、子專案版
- 命名格式：`[三位數流水號]_[名稱]__探索` 或 `[三位數流水號]_[名稱]__子專案`
- 執行單元之間禁止跨單元 import；若需共用邏輯，應明確拆成獨立單元或在新版本整合

**程式碼修改閘門（寫死）：**
Harness（閘門式自主推進）在 .md 檔案範圍內完全自主。但在修改 `04_主專案程式/` 內的任何非 .md 檔案之前，必須：
1. 停下來
2. 列出要改什麼、為什麼改
3. 等使用者確認「可以改」
4. 確認後才執行
此規則無例外。

### 1-8. 系統總覽分區規則

`_系統/基地/系統總覽.md` 是**半生成、半狀態檔**。是路由，不是資料倉。

| 區塊 | 由誰產生 | 可否手改 | 系統更新行為 |
|------|---------|---------|------------|
| 結構區 | SRI 生成 | 禁止手改 | 差異化更新 |
| 狀態區 | AI 日常寫入 | AI 可寫 | 保留不覆蓋 |
| 版本一致性區 | 系統更新寫入 | 禁止手改 | 每次更新刷新 |
| 待決策衝突區 | 系統更新衝突時寫入 | 人類決策後可清理 | 有衝突時追加 |

**留痕觸發條件（寫死）：**
只有以下情況才更新系統總覽狀態區：
1. 寫入正式方案（整理到總方案／施工藍圖／施工進度）
2. 系統更新
3. 中改跨階段影響
4. 使用者明確要求

一般聊天不寫系統總覽。

### 1-9. 使用者體驗總原則

1. 人類看到的第一句話必須有用，不得先報內部操作
2. 建檔必須快，不預先塞長篇分析
3. 讀檔是內部動作，對外只回報讀完後的判斷
4. 確認類回覆不超過兩句；討論類依矩陣提問規則控制密度
5. 每個指令完成後，至少建議 1–2 個下一步
6. `進入專案` 優先走快速接續模式，不預設每次都全讀
7. `這個專案是做什麼的` 要能讓使用者不必先記住系統術語也能理解專案
8. 角色在內部持續生效並留下可追溯記錄；外顯輸出採條件式呈現
9. 讀 SRI 是內部動作，不對使用者展示
10. 共用規則只寫一次，不在多份檔案各自重寫
11. AI 自動判斷使用者當前所在模式與階段，不要求使用者主動切換
12. 使用者不需要理解內部系統結構就能操作
13. AI 每次建立或存放檔案時，自動判斷最佳放置位置與分類方式。第一次放新類型時先問使用者確認，之後同類型自動依慣例放置。

### 1-10. 10-15 自動更新機制

**v4 重大變更：取代 v3「使用者喊`對話整理`才整理」的被動模式。**

AI 邊聊邊自動判斷，把對話中的內容即時分流到對應檔案。

| 規則 | 值 | 說明 |
|------|---|------|
| 最小間隔 | 2 輪 | 防止太頻繁拖慢回覆 |
| 最大間隔 | 5 輪 | 超過必須觸發一次更新 |
| 智慧觸發 | AI 自主判斷 | 偵測到重要結論、方向轉折、問題浮現時可在 2-5 之間提前觸發 |
| 手動觸發 | 仍保留 | 使用者說「對話整理」仍可隨時手動觸發 |

**自動分流邏輯：**

| AI 偵測到的內容 | 自動寫入目標 |
|----------------|------------|
| 還沒確定的想法、靈感、假設 | → _系統/11_想法暫存區.md |
| 需要回答或驗證的問題 | → _系統/12_待確認問題.md |
| 使用者明確確認的結論 | → _系統/13_已確認結論.md |
| 方向性決策、願景、成功條件 | → 總方案.md |
| 對話重點摘要 | → _系統/10_對話整理.md（持續追加） |
| 方案的重大變更 | → _系統/15_方案變更紀錄.md |

**自動更新不取代正式整理動作。** `整理已確認結論到總方案` 等正式指令仍需使用者觸發。自動更新只做「即時分流」，不做「跨層融合」。

### 1-11. AI 角色討論機制

**v4 重大變更：角色討論從「一次性審查事件」升級為「持續性內部機制」。**

**半自動模式（指令.md）：**
- 使用者可主動觸發：`角色討論 <主題>`、`角色審查`、`讓角色出來說說`
- AI 也可主動建議：「這個方向有分歧，建議讓角色討論一下？」
- 使用者看到：結論 + 分歧點摘要
- 完整過程：記入 `01_團隊/角色會議紀錄.md`
- **結論需使用者確認才生效**

**Harness 模式（大絕招.md）：**
- AI 自主決定何時開啟角色討論
- AI 自主記錄、收斂、推進
- 使用者看到：進度報告中的結論 + 分歧點
- 完整過程：記入 `01_團隊/角色會議紀錄.md`
- **結論直接生效**（除非觸發硬閘門）

**討論場景觸發：**
- 遇到方向分歧 → 開角色會議
- 遇到技術可行性疑問 → 召集技術角色
- 遇到風險 → 召集安全角色
- 規劃階段重大決策前 → 自動觸發

### 1-12. MCP 工具層

MCP（Model Context Protocol）是 SRI 的工具層外掛，不是核心流程的一部分。

**設定位置：** `_系統/基地/mcp/`

| 檔案 | 用途 |
|------|------|
| README.md | MCP 策略總覽 |
| servers.json | 當前啟用的 server 清單 |
| policies.md | 安全政策 |

**安全規則：**
1. 所有 MCP server 預設為 read-only
2. 首次連線必須使用者確認
3. 會寫入外部系統的 server 永遠需要使用者確認
4. Harness 中的 MCP 呼叫遵守同樣的硬閘門

**MCP 產出的資料路由：**
- 原始設計檔、外部 API 回傳 → `02_參考資料/原始素材/`
- AI 消化後的分析 → `03_文件產出/`
- 實際程式碼整合 → `04_主專案程式/`（需確認閘門）

**系統更新絕對不得刪除以下內容：**

## §3 四階段共用規則

### 3-1. 四階段真值邊界

架構階段與施工階段僅適用程式設計模式。一般模式涵蓋規劃與運維；程式設計模式涵蓋規劃、架構、施工、運維。

#### 規劃階段的真值
項目願景、目標使用者、問題定義、成功條件、功能範圍、階段邊界、核心使用者流程、商業邏輯、需求優先序、探索問題是否納入正式範圍。

規劃階段**不負責**：寫程式、模組拆分、API 設計、技術選型、schema 設計、施工順序、測試實作。

#### 架構階段的真值
系統架構、模組邊界、資料流、控制流、API 契約、資料模型/schema、狀態模型、測試策略、驗證策略、施工順序、風險與回退方案、技術約束、開發階段切分。

架構階段**不負責**：寫程式、跑實際測試、決定產品願景、決定功能範圍、替規劃做需求裁切。

#### 施工階段的真值
實作進度、當前施工狀態、已完成模組/功能、已實作與藍圖差異、實際阻斷、測試結果、臨時 workaround、需回流的施工問題、下一步建議。

施工階段**不負責**：決定願景、決定功能範圍、決定正式架構藍圖、把 workaround 升格成正式藍圖。

#### 運維階段的真值
日常維護、監控與回報、問題排查、功能迭代、退役或移交的狀態與決策。

運維階段**不負責**：默默改動核心架構、跳過回流路徑直接改上游真值、把重大迭代偽裝成日常維護。

#### AI 自動階段判斷
- 每輪默默判斷當前屬於哪個階段
- 一般模式判斷規劃與運維；程式設計模式判斷四個階段
- 只在跨錯階段或準備寫正式檔案時才提醒
- 不顯示為標籤，不要求使用者操作

### 3-2. 變更分級

#### 小改：局部改動（可直接改）
- 只影響本階段或下游執行細節
- 不影響上游真值核心

處理：直接改 + 補變更紀錄。不需先問。

#### 中改：跨階段非核心改動（可直接改，但待複核）
- 影響其他階段，但未動到願景、範圍、核心功能、成功條件、安全邊界

處理：直接改 + 補變更紀錄 + 加 `待複核` 標記 + 明示受影響階段。

#### 大改：核心改動（必須先問使用者）
- 觸及真值核心

處理：先提醒 → 使用者確認 → 改正式產物 + 補變更紀錄。

### 3-3. 變更紀錄格式

```
## YYYY-MM-DD HH:mm｜變更等級：小改 / 中改 / 大改

- 修改對象：[哪份正式產物]
- 改了什麼：
- 為什麼改：
- 影響範圍：
- 狀態：已生效 / 待複核 / 已確認
```

### 3-4. 版本基底關聯

僅適用程式設計模式。

`_系統/20_施工藍圖.md` 必須標明：基於哪版 總方案.md、最後同步時間、若有差異需標註偏差原因。

`_系統/30_施工進度.md` 必須標明：基於哪版 _系統/20_施工藍圖.md、最後同步時間、一致性狀態。

### 3-5. 回流矩陣

AI 在日常對話中低成本背景判定是否跨錯階段。只在跨錯時提醒。

| 問題性質 | 建議優先處理階段 |
|---------|----------------|
| 只影響施工細節 | 施工 |
| 影響藍圖真值 | 架構 |
| 影響願景/範圍/核心功能/成功條件 | 規劃 |
| 無法判定 | 先標記，問使用者 |

### 3-6. 上游偏差處理

僅適用程式設計模式。

架構階段進場時比對 總方案.md 與藍圖版本基底 → 不一致先停。
施工階段進場時比對 _系統/20_施工藍圖.md 與施工進度版本基底 → 不一致先停。
不得默默沿用過期基底。

### 3-7. 矩陣提問規則

- 起始探索期：一次 3–5 題，按主題分組
- 細節收斂期：一次 1–2 題，逐步確認
- 通常不超過 5 題
- 風險高或上下文脆弱時，降到單題確認
- 每批之間必須等使用者回答完

### 3-8. 跨階段防呆規則

以下一律視為重大變更（必須人類確認）：
- 改模組主結構
- 刪功能 / 刪正式主檔
- 更換核心依賴
- 改 schema 主結構
- 影響超過 3 個正式檔案的重構
- 與既有規劃不一致的方向調整
- 改安全模型 / 權限模型 / 金流邏輯

## §4 規劃階段

### 4-1. 三個討論維度

#### 維度一：方向定位（願景層）

觸發：總方案.md 願景欄位為空或草稿，或 _系統/13_已確認結論.md 沒有項目願景。

態度：不急著談技術或細節，先把「這個專案到底要做什麼」講清楚。

產出路徑：聊天 → 自動分流到 10/11/12 → 穩定後整理進 13 → 正式化後進 14。

#### 維度二：問題處理（待確認問題層）

觸發：總方案.md 有穩定願景，且 _系統/12_待確認問題.md 還有 `#尚未回答` 的核心問題。

態度：找缺口、矛盾、未定義之處。1–3 題逐步確認。

#### 維度三：想法探索（想法層）

觸發：使用者明確說 `延伸想法`、`發散思考`、`討論想法暫存區`，或主要方向已收斂以探索為主。

態度：補盲點、變體、替代方案。探索進 11，值得確認的回流到 12 加 `[想法回流]` 標記。

### 4-2. 自動遞進邏輯

```
先讀 總方案.md
  ↓
願景有穩定內容？
  沒有 → 維度一
  有 → 讀 _系統/12_待確認問題.md
         有 #尚未回答 的核心問題？
           是 → 維度二
           否 → 維度三
```

人類覆蓋永遠優先：
- `延伸想法` / `發散思考` → 強制維度三
- `討論想法暫存區` → 強制維度三
- `討論待確認問題` → 強制維度二

## §5 架構階段

本部分僅適用程式設計模式。

### 5-1. 定位

把規劃已收斂的方案翻成可施工、可驗證、可交付的工程藍圖。

### 5-2. 預設模式

依矩陣提問規則，1–3 題直接收斂。聚焦藍圖、邊界、契約、測試、施工順序。

### 5-3. 升級模式（架構深審）

符合以下任一條件，AI 應主動建議啟動架構深審：
- 重大架構切換
- 不可逆技術選型
- 多模組邊界重切
- 共享 API 契約定版
- schema 主結構定版
- 安全/權限模型定版
- 部署/狀態管理主幹定版

使用者確認後才正式進深審。

### 5-4. 方向完整性校正

重大修改前必須先確認：依據的是哪份正式 md、基底是不是最新版、本輪目標是否與系統總覽一致、是否有新共識尚未回寫。

### 5-5. 完成條件

- `_系統/20_施工藍圖.md` 已更新且狀態為「已確認可施工」
- `_系統/21_藍圖變更紀錄.md` 已補變更摘要
- 版本基底已標記
- 施工階段可直接依此藍圖施工

**施工藍圖未標記「已確認可施工」前，不得進入施工階段。**

## §6 施工階段

本部分僅適用程式設計模式。

### 6-1. 定位

依已確認可施工的藍圖進行實際施工、測試、修正、交付與回報。

### 6-2. 已確認可施工閘門檢查

若 `_系統/20_施工藍圖.md` 不存在，或其狀態不是「已確認可施工」，不得進入正式施工。

### 6-3. 快速確認規則

#### 可本地解決（不升級）
條件：不改功能範圍、不改共享 API、不改安全模型、不改驗收標準、不改藍圖主結構。
處理：直接施工 + 施工日誌補 `[快速確認]`。

#### 應回架構
條件：藍圖細節不完整、技術不可行、API/schema/模組邊界需重定。
處理：施工日誌留痕，回架構階段。

#### 應回規劃
條件：功能範圍需刪減、核心流程需重定、使用者體驗要改、成功條件不合理。
處理：說明超出施工範圍，由使用者決定。

### 6-4. 審查深度三級

#### 第一級：日常模式
修 bug、補小功能、加註解、調樣式。不需獨立報告。

#### 第二級：深度審查
認證/授權、金錢、個資、外部 API、關鍵狀態流。簡短說明風險 + 跑必要測試。
不確定時預設第二級。

#### 第三級：全套審查
新增/刪除模組、重大重構、換核心依賴、改 schema、改部署主幹。完整 review + 人類確認。
判不清寧可升級。

## §7 中間層運作機制

### 7-1. 專案隔離規則

每個專案獨立一組中間層。**不同專案的中間層不得共用、不得混寫、不得互相引用作為正式前提。**

### 7-2. 結構化議題格式

#### 完整記錄格式（用於 10/11/12）

```markdown
### [議題標題]
- 類型：single_select / multi_select / rank_priorities / open_ended
- 問題：[完整問題文字]
- 選項：
  A) [選項 A]
  B) [選項 B]
  C) [選項 C]
- 使用者選擇：B
- AI 追問：[若有後續問題]
- 備註：[若有補充脈絡]
- 狀態：#尚未回答 / ✓ 議題關閉 / ⚠ 使用者決定不處理
```

#### 精簡記錄格式（用於 13）

```markdown
### [議題標題]
- 問題：[一句話版]
- 結論：[選項或摘要]
- 確認日期：YYYY-MM-DD
```

#### 格式規則
1. 12_待確認問題 使用完整格式，每題必須有狀態
2. 11_想法暫存區 使用完整格式，狀態可省略
3. 10_對話整理 使用完整格式記錄本輪議題
4. 13_已確認結論 使用精簡格式
5. 開放式問題：類型標 `open_ended`，選項寫「（開放式）」
6. 寫入 md 時不論對話中是否用了互動選項，都統一用上述格式

### 7-3. 三種關閉狀態

- `✓ 議題關閉`：已有明確結論
- `⚠ 使用者決定不處理`：已知但暫不決定
- `#尚未回答`：仍待回答

### 7-4. 已確認結論分區

```markdown
# 已確認結論

## 已確定區
（穩定、可作為寫入正式方案前提的結論）

## 不確定區
（已討論但仍有疑慮、待後續確認）
```

### 7-5. 想法暫存區

想法池，不等於正式採納。可放：暫時想法、未成熟 idea、備選方案、砍掉但值得保留的、可能回流到 12 的。

### 7-6. 對話整理

用於整理對話、保留脈絡與快速接續。不是正式決策檔。

**v4 變更：** 自動更新機制會在 2-5 輪間自動追加內容到 _系統/10_對話整理.md。手動觸發 `對話整理` 仍可用，效果為立即執行一次完整分流。

### 7-7. 參考資料規則

**02_參考資料/資料地圖.md** 是找檔入口、結構一致性檢查基準。

**02_參考資料/資料館長.md** 是內容理解入口，至少包含：整體總覽、分類分析、單檔分析、關聯與回流建議、缺口與待補。

**三種資料模式：**

| 模式 | 輸入範例 | AI 做什麼 | 產出 |
|------|---------|----------|------|
| 分析型 | 起訴書 PDF | OCR → 全文整理 → 摘要 | 資料館長更新 + 歸入 整理版/ |
| 資料型 | 台積電歷史報價 CSV | 不分析內容，只描述結構 | 資料夾 + metadata md |
| 快取型 | AI 抓的交易所首頁 | 分類存放 + 建索引 | 主題資料夾 + 資料地圖更新 |

### 7-8. 文件產出規則

**03_文件產出/** 是 AI 做完可直接交付使用者的成品存放區。

**03_文件產出/產出索引.md** 是目錄，列出所有產出（名稱、日期、用途）。

**分界線：02 是圖書館（input，不管生的熟的都是素材），03 是交付台（output，AI 做完可直接用的成品）。**

**寫入規則：**
1. AI 完成分析報告、回測結果、市場研究、正式文案等，直接寫入 03_文件產出/
2. 同步更新產出索引.md
3. 平層放置為預設；超過 10 個產出時可建子目錄分類
4. 文件產出不是專案決策真值，不得取代 總方案.md 的角色

### 7-9. 參考資料一致性規則

允許兩種狀態：
- **狀態 A：未整理輸入狀態** — 使用者可以直接丟東西進去
- **狀態 B：整理完成狀態** — 執行 `整理 參考資料` 或 `系統更新` 後的一致狀態

一致性檢查：
- 實體存在但未出現在資料地圖 → 錯誤
- 資料地圖有寫但實體不存在 → 錯誤

### 7-10. 中間層生命週期規則

- 新增專案時建立全套空白模板，不自動寫入內容
- 自動更新機制（§1-10）在 2-5 輪間自動分流；正式跨層融合仍需使用者觸發
- 中間層不可取代正式主檔
- 寫入正式方案完成後不清空；改在頂部標記「已整理歸檔」+ 日期 + 對應正式產物版本
- 下一輪需要新議題時，優先建立新檔（如 `12_待確認問題-02.md`）避免覆蓋

## §8 安全規則

以下安全規則適用於所有指令與所有情境。

### 8-1. 資料保護

1. **不得自動刪除**任何既有正式主檔、中間層檔案或 02_參考資料/ 原始資料
2. **不得自動刪除**任何 01_團隊/ 內既有角色記憶與人格卡
3. **不得自動刪除**任何 03_文件產出/ 內既有產出
4. 搬移舊檔到 _系統/99_舊檔案封存區/ 是封存，不是刪除，必須主動執行
5. 不得把 _系統/基地/系統總覽.md 當成唯一真值來源；正式決策以 總方案.md 為準
6. 不得把 02_參考資料/ 的內容直接視為正式規格或已確認結論

### 8-2. 寫入邊界

7. **不得把 10/11/12 視為正式拍板依據**
8. **不得把一般聊天內容自動升格為正式整理動作**（自動分流寫入 10-12 不等於正式整理）
9. **不得在未經使用者確認的情況下執行大改**
10. 平時未明確下 `整理 <資料夾>` 時，不得自動搬移或重組 `02_參考資料/`、`03_文件產出/`、`04_主專案程式/`、`05_沙盒區/` 內的實體資料（系統更新除外）
11. `整理 <資料夾>` 與 系統更新 可建資料夾、移動檔案與更新索引；若涉及 `04_主專案程式/` 內非 `.md` 檔案，仍須先停下來確認；任何情況都不得刪除既有資料
12. 系統更新可將舊版檔案移入 _系統/99_舊檔案封存區/；此行為是封存，不是刪除

### 8-3. 閘門與階段

### 8-4. 沙盒邊界
