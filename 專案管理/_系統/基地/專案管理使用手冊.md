# 專案管理使用手冊

> SRI v4.0.0
> 第一態：純人工
> 最後更新：2026-04-01

---

## 1. 這份手冊是做什麼的

這是 SRI 三態中的**第一態**：純人工。

它定義的是：

- 資料夾怎麼建
- 檔案怎麼放
- 哪些是正式真值
- 哪些只能當草稿或素材
- 哪些情況必須停

就算完全不用 AI，只靠這份手冊，也應該能手動從零跑完一個專案。

## 5. 編號規則

| 編號段 | 用途 | 類型 |
|--------|------|------|
| 01 | 團隊（人格 + 角色） | 資料夾 |
| 02 | 參考資料 | 資料夾 |
| 03 | 文件產出 | 資料夾 |
| 04 | 主專案程式 | 資料夾（僅程式設計模式） |
| 05 | 沙盒區 | 資料夾 |
| 10-15 | 討論與規劃 | 檔案 |
| 20-21 | 架構階段 | 檔案（僅程式設計模式） |
| 30-31 | 施工階段 | 檔案（僅程式設計模式） |
| 99 | 舊檔案封存區 | 資料夾 |

**原則：** 資料夾用個位數段，檔案用十位數段，永不撞號。

## 6. 專案模式

| 模式 | 適用 | 包含 |
|------|------|------|
| 一般 | 非程式專案 | 10-15 + 01/02/03/05/99 |
| 程式設計 | 軟體專案 | 10-31 + 01/02/03/04/05/99 |

模式判定寫死規則：

1. 唯一正式來源是本地 `總方案.md` 的 `專案模式` 欄位。
2. `系統總覽.md` 只做路由快取，不是正式判定來源。
3. 一般模式可升級為程式設計模式，不可反向降級。

---

## 7. 05_沙盒區

| 類型 | 前綴 | 結構 | 用途 |
|------|------|------|------|
| 探索 | `A_`、`B_` | 10-13 + `02_參考資料/` | 輕量比較、試方向 |
| 子專案 | `01_`、`02_` | 完整程式設計模式結構 | 正式子系統 |

沙盒規則：

1. 一般模式只能開探索。
2. 子專案固定為程式設計模式。
3. 探索不建立獨立 `01_團隊/`，直接繼承正式容器。
4. 探索結論不是正式真值，要影響主線必須回流。

## 8. 檔案角色與真值優先序

```text
總方案.md            → 規劃階段正式真值
_系統/15_方案變更紀錄.md      → 規劃階段正式留痕
_系統/20_施工藍圖.md          → 架構階段正式真值
_系統/21_藍圖變更紀錄.md      → 架構階段正式留痕
_系統/30_施工進度.md          → 施工階段正式真值
_系統/31_施工日誌.md          → 施工階段正式留痕
_系統/13_已確認結論.md        → 寫入正式真值前的穩定前提
_系統/12_待確認問題.md        → 待收斂
_系統/11_想法暫存區.md        → 想法池
_系統/10_對話整理.md          → 對話脈絡
02_參考資料/            → 素材與參考
03_文件產出/            → AI 做好的交付物
```

**重點：**

- `02_參考資料/` 不是正式規格。
- `03_文件產出/` 不是正式決策真值。
- 10 / 11 / 12 都不能當拍板依據。

## 9. 10-15 自動更新機制

v4 起，10-15 不再只靠手動整理。

| 規則 | 值 |
|------|---|
| 最小間隔 | 2 輪 |
| 最大間隔 | 5 輪 |
| 智慧觸發 | AI 可提前分流 |
| 手動觸發 | `對話整理` 仍保留 |

自動分流對應：

- 想法 → 11
- 問題 → 12
- 已確認結論 → 13
- 方向性決策 → 14
- 對話摘要 → 10
- 重大變更 → 15

**注意：** 自動更新不等於正式整理。`整理已確認結論到總方案` 仍是獨立動作。

## 10. 02_參考資料 與 03_文件產出

### 02_參考資料

定位：素材庫、圖書館、原始輸入層。

固定入口：

- `資料地圖.md`：找檔入口
- `資料館長.md`：內容理解入口
- `原始素材/`：原始素材
- `整理版/`：整理後素材

### 03_文件產出

定位：AI 已完成、可直接交付的成品。

固定入口：

- `產出索引.md`

分工邊界：

- 02 是 input
- 03 是 output
- 14 / 20 / 30 才是正式真值

## 11. 04_主專案程式

`04_主專案程式/` 是交付與運行區，不是單純程式碼倉庫。

### 執行單元命名

- `001_名稱__探索`
- `002_名稱__子專案`

### 規則

1. 每個執行單元都要有自己的 README。
2. 執行單元之間禁止跨單元 import。
3. 若需共用邏輯，拆成新的執行單元或在新版整合。
4. 修改此區任何非 `.md` 檔案前，必須先停確認。

## 13. _系統/基地/mcp

v4 把 MCP 正式移入 current。

```text
_系統/基地/mcp/
├── README.md
├── servers.json
└── policies.md
```

核心規則：

1. 所有 server 預設 read-only。
2. 首次連線必須確認。
3. 會寫外部系統的 server 永遠需要確認。
4. MCP 路由固定到 02 / 03 / 04。

---

## 14. 安全規則

1. 不得自動刪除正式主檔、中間層檔案、原始素材或既有交付物。
2. 不得把 10 / 11 / 12 / 02 / 03 當成正式拍板依據。
3. 不得在未確認下執行大改。
4. 探索結論不得直接寫進主線 14 / 20 / 30。
5. 程式碼修改閘門無例外。
6. MCP 首次連線或權限升級無例外。

## 16. 從零到完成

1. 建立環境：確認 `專案管理/_系統/基地/sa/` 與 6 份 source 存在。
2. 新增專案：選一般或程式設計模式。
3. 規劃：先收斂 14，再整理 15。
4. 架構：僅程式設計模式，整理到 20 / 21。
5. 施工：僅程式設計模式，確認 20 已可施工後才進 30 / 31 與 `04_主專案程式/`。
6. 運維：日常維護、迭代與回流。

---

## 17. 一句話總結

v4 的核心不是多加幾份檔案，而是把結構、團隊、交付物、MCP、Harness 邊界一次寫死，讓人類與 AI 都只能沿著同一條路工作。
