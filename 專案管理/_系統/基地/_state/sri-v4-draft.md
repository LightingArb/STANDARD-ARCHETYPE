## 三、行為模型（v4.0.0 新規則）

### 3-1. 三態定義

| 態 | 正式名稱 | 入口 | 核心行為 |
|----|---------|------|---------|
| 第一態 | 純人工 | 專案管理使用手冊.md | 人類按結構手動操作，AI 最多當顧問 |
| 第二態 | 指令驅動半自動 | 指令.md | 使用者下指令，AI 執行；大事小事都確認；AI 可發起角色討論但結論需使用者確認 |
| 第三態 | 閘門式自主推進（Harness） | 大絕招.md | AI 自主決策/發散/收斂/角色討論；使用者只在硬閘門處確認 |

### 3-2. 10-15 自動更新機制

**取代舊版的「使用者喊`對話整理`才整理」模式。**

| 規則 | 值 | 說明 |
|------|---|------|
| 最小間隔 | 3 輪 | 防止每輪都寫入拖慢回覆 |
| 最大間隔 | 10 輪 | 超過必須觸發一次更新 |
| 智慧觸發 | AI 自主判斷 | 偵測到重要結論、方向轉折、問題浮現時可在 3-10 之間提前觸發 |
| 手動觸發 | 仍保留 | 使用者說「對話整理」仍可隨時手動觸發 |

**自動分流邏輯：**

| AI 偵測到的內容 | 自動寫入目標 |
|----------------|------------|
| 還沒確定的想法、靈感、假設 | → 11_想法暫存區.md |
| 需要回答或驗證的問題 | → 12_待確認問題.md |
| 使用者明確確認的結論 | → 13_已確認結論.md |
| 方向性決策、願景、成功條件 | → 14_總方案.md |
| 對話重點摘要 | → 10_對話整理.md（持續追加） |
| 方案的重大變更 | → 15_方案變更紀錄.md |

### 3-3. AI 角色討論機制

**核心概念：角色討論從「一次性審查事件」升級為「持續性內部機制」。**

**半自動模式（指令.md）：**
- 使用者可主動觸發：「你們討論一下」「角色審查」「讓角色出來說說」
- AI 也可主動建議：「這個方向有分歧，建議讓角色討論一下？」
- 使用者看到：結論 + 分歧點摘要
- 完整過程：記入 `01_團隊/角色會議紀錄.md`
- **結論需使用者確認才生效**

**Harness 模式（大絕招.md）：**
- AI 自主決定何時開啟角色討論
- AI 自主記錄、收斂、推進
- 使用者看到：進度報告中的結論 + 分歧點
- 完整過程：記入 `01_團隊/角色會議紀錄.md`
- **結論直接生效**（除非觸發硬閘門）

### 3-4. Harness 自主權邊界（v4.0.0）

**核心原則：.md 內完全自主，改程式碼必停確認。**

| 動作 | 可自主 | 說明 |
|------|--------|------|
| 10-15 討論層讀寫 | ✓ | 包含自動更新 |
| 01_團隊/ 會議紀錄更新 | ✓ | 角色討論結果 |
| 02_參考資料/ 整理與分類 | ✓ | 素材管理 |
| 03_文件產出/ 產出新文件 | ✓ | AI 交付物 |
| 上網抓資料、下載資料 | ✓ | 不受限 |
| 角色之間互相討論 | ✓ | 自主觸發 |
| 決策、發散、收斂 | ✓ | 在 .md 範圍內 |
| 記下疑問繼續推進 | ✓ | AI 自行判斷何時停 |
| **改 04_主專案程式/ 內的程式碼** | **✗ 必停** | 確認後才放手 |
| **刪除/搬移/覆蓋大量檔案** | **✗ 必停** | 確認後才放手 |
| **MCP 首次連線或權限升級** | **✗ 必停** | 確認後才放手 |

### 3-5. Harness 停止機制（v4.0.0）

**取代 v3 的雙計數器機制。**

| 機制 | 規則 |
|------|------|
| AI 自主判斷 | AI 認為「該停下來讓使用者看一下」時即可停 |
| 安全網上限 | 最多連續 10 輪自主推進，必須停一次報告 |
| 硬閘門 | 改程式碼、大量檔案操作、MCP 權限 → 無論第幾輪都必停 |
| 使用者中斷 | 使用者隨時可說「暫停」或「停止」 |

## 四、指令體系（v4.0.0）

### 4-1. 指令.md 核心改動

**命令改名：**

| v3 | v4 | 原因 |
|----|----|------|
| `新增專案 <名稱>` | `新增一般專案 <名稱>` | 明確專案類型，不靠尾綴參數 |
| `新增專案 <名稱> 程式設計` | `新增程式專案 <名稱>` | 同上 |

**新增指令：**

| 指令 | 用途 | 模式 |
|------|------|------|
| `角色討論 <主題>` | 觸發 AI 角色之間的發散+收斂討論 | 半自動 |
| `發散思考 <主題>` | AI 多角色發散，不收斂 | 半自動 |
| `收斂思考` | 把發散結果收束成可行結論 | 半自動 |
| `列出 MCP 伺服器` | 查看可用 MCP | 半自動 |
| `啟用 MCP <名稱>` | 啟用指定 MCP server | 半自動 |

**保留但調整的指令：**

`對話整理` → 仍保留為手動觸發入口，但 10-15 自動更新機制會在背景持續運作，使用者不喊也會自動整理。

### 4-2. 大絕招.md 核心改動

**啟動方式（放在文件最前面）：**

```
## 啟動 Harness

1. 進入你要推進的專案：`進入專案 <名稱>`
2. 對 AI 說以下任一句：
   - 「啟動 Harness」
   - 「自主推進到 [目標]」
   - 「自主完成這個專案」
3. AI 會先做目標解析（Step 0），列出推進計畫
4. 你確認計畫 → Harness 開始自主推進
5. 你隨時可以說「暫停」或「停止」
```

**Harness 內的角色討論（新增段落）：**

Harness 推進過程中，AI 可自主觸發角色討論：
- 遇到方向分歧 → 開角色會議，記錄結論和分歧
- 遇到技術可行性疑問 → 召集技術角色討論
- 遇到風險 → 召集安全角色評估
- 討論結果寫入 `01_團隊/角色會議紀錄.md`
- 使用者在進度報告中看到結論摘要

**改程式碼的硬閘門（新增段落）：**

```
## 程式碼修改閘門（寫死）

Harness 在 .md 檔案範圍內完全自主。
但在修改 04_主專案程式/ 內的任何非 .md 檔案之前，必須：

1. 停下來
2. 列出要改什麼、為什麼改
3. 等使用者確認「可以改」
4. 確認後才執行

使用者確認後，AI 可在該批次內自主完成修改。
下一批新的程式碼修改，需要再次確認。

此規則無例外，即使 Harness 處於端到端自主推進模式。
```

## 五、MCP 整合規格

### 架構

```
99_基地/mcp/
├── README.md              ← MCP 策略總覽
├── servers.json           ← 當前啟用的 server 清單
└── policies.md            ← 安全政策
```

### servers.json 格式

```json
{
  "servers": [
    {
      "name": "stitch",
      "description": "Google Stitch — AI UI 設計拉取",
      "permission": "read-only",
      "first_connect": "require-confirmation",
      "config": {
        "command": "npx",
        "args": ["@_davideast/stitch-mcp", "proxy"]
      },
      "output_routing": {
        "raw_design_files": "02_參考資料/raw_input/",
        "processed_analysis": "03_文件產出/",
        "code_integration": "04_主專案程式/"
      }
    }
  ]
}
```

### 安全政策（policies.md 核心規則）

1. 所有 MCP server 預設為 read-only
2. 首次連線必須使用者確認
3. 會寫入外部系統的 server 永遠需要使用者確認
4. Harness 中的 MCP 呼叫遵守同樣的硬閘門
