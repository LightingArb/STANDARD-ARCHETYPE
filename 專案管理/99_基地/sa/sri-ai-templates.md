## S-03. 指令.md

※ 人類入口，**不加 generated header**。第一行必須直接是 `## 常用指令`。

````markdown
## 常用指令

> 本檔是 SRI 三層架構的**第二層入口**——人＋AI 協作流程。
> 第一層（結構規範）：專案管理使用手冊.md
> 第三層（AI 自主推進）：大絕招.md

---

### 開始使用

- `完整初始化` — 第一次建立整套環境。
- `系統更新` — 全域總更新。同步生成物，巡檢所有專案與沙盒。

### 專案操作

- `新增專案 名稱` — 建立**一般模式**專案。自動從 Furnace 複製角色（01-04）+ 角色依賴的人格卡（聯集去重）。直接開始第一輪討論。
- `新增專案 名稱 程式設計` — 建立**程式設計模式**專案。自動從 Furnace 複製完整角色（01-04 + 11-13）+ 角色依賴的人格卡（聯集去重）。多建 20/21/30/31 和 07_主專案程式/。
- `進入專案 名稱或編號` — 回到既有專案，快速接續。
- `專案說明 名稱` / `這個專案是做什麼的` — 看進度與缺什麼。
- `升級專案 程式設計` — 一般 → 程式設計（不可逆）。補建 20/21/30/31、補入缺少的角色（11-13）+ 掃描新角色依賴的人格卡 → 補入缺少的人格。

### 沙盒操作

- `新增沙盒 名稱` — 開分流。AI 會問：
  - **探索**（輕量，測方向）→ 字母前綴 `A_名稱`。不建立獨立 02_人格/，繼承所屬正式容器的人格集。
  - **子專案**（完整架構）→ 數字前綴 `01_名稱`（僅程式設計模式可開）。自動從 Furnace 複製完整角色 + 依賴人格卡。
- `進入沙盒 名稱或代號` — 切到某個沙盒。
- `回到主專案` — 從沙盒切回主線。
- `比較所有沙盒` — 跨沙盒比較結論。
- `回流沙盒到主專案` — 把沙盒結論整理回主線（你確認後才寫入）。
- `自動回流` — AI 判斷哪些該回流（仍需你確認）。
- `升級沙盒` — 探索 → 子專案（補齊完整架構）。
- `降級沙盒` — 子專案 → 探索（精簡回輕量版）。

### 討論與發想

- `討論專案` — AI 自動判斷你在規劃、架構、施工或運維階段，繼續討論。
- `角色審查` — 多角色完整評審（見下方「角色進場規則」）。
- `延伸想法` — 支援句型：`（向外／向內）延伸（想法／問題）（主題）`
- `討論想法暫存區` / `討論待確認問題` — 以對應檔案為主面板。

### 整理

- `壓縮對話` — 壓成短版快照。
- `對話整理` — 整理進 10_對話整理.md 並自動分流到 11（想法）和 12（待確認）。
- `整理待確認問題到已確認結論` — 12 → 13（剪下貼上）。
- `整理已確認結論到總方案` — 13 + 14 融合後寫入 14。
- `整理已確認結論到施工藍圖` — 13 + 14 + 20 融合後寫入 20（僅程式設計模式）。
- `整理參考資料` — 整理 03_參考資料/ 成結構一致狀態。

### 施工

- `開始施工` — 僅程式設計模式，藍圖已確認可施工後才能用。

### 執行單元操作（07_主專案程式/）

- `新增執行單元 名稱` — 在 07_ 內建立新的執行單元。AI 會問：
  - **探索版**（試可行性，輕量骨架）
  - **專案版**（正式系統，完整架構）
- `進入執行單元 編號或名稱` — 切到指定執行單元，主讀該單元 README.md。
- `執行 編號或名稱` — 依 README 約定的入口執行程式。
- `查看輸出 編號或名稱` — 查看該執行單元 output/ 內容。
- `升級執行單元 編號` — 探索版 → 專案版（補齊 src/output/tests/）。
- `降級執行單元 編號` — 專案版 → 探索版。
- `列出執行單元` — 掃描 07_ 內所有執行單元。

### 人格操作

- `取用人格` — 從 Furnace 取人格卡到專案 02_人格/。AI 會列出上架清單讓你選。
- `列出人格` — 看當前專案有哪些人格卡。
- `移除人格` — 從 02_人格/ 移除指定人格卡。

### 收束

- `結束對話` — 總結本輪、建議下一步。

### 備用強制口令

- `進入設計師`（規劃）
- `進入架構師`（架構；僅程式設計模式）
- `進入工班`（施工；僅程式設計模式）

## 角色進場規則

### 誰在什麼時候出聲

> 新增專案/升級/子專案時，系統依角色依賴聯集自動配發對應角色與人格，不再以 L1/L2 單獨區分。

| 階段 | 啟用角色 | 觸發 |
|------|---------|------|
| 規劃 | 01_策略決策、02_創意總監、03_使用者研究、04_技術聯絡 | `討論專案`、`角色審查` |
| 架構 | 01_策略決策、11_系統架構師、12_安全風控、13_品質驗證 | `討論專案`（架構階段）、`角色審查` |
| 施工 | 不啟用角色系統 | — |
| 運維 | 不啟用角色系統。議題需深審時回規劃或架構角色 | — |

### 角色什麼時候外顯

**預設：** AI 內部用角色思維判斷，但輸出用統整口吻，不掛角色標籤。

**以下情況才外顯角色標籤：**
- 角色之間出現真正的差異視角或意見分歧
- 有明確的風險衝突或可行性阻斷
- 你說「角色審查」
- 你說「讓角色出來說說」

**你說「先不要角色視角」→ 本輪完全靜默角色。**

### 角色配裝（預設人格卡映射）

| 角色 | 預設人格卡 | 說明 |
|------|-----------|------|
| 01_策略決策 | #101 通用核心 + #102 產品策略 | 商業模式、ROI、範圍控制 |
| 02_創意總監 | #101 + #104 UI/UX設計 + #213 寫作溝通 | 品牌調性、視覺方向 |
| 03_使用者研究 | #101 + #102 + #211 數據分析 | 受眾分析、痛點驗證 |
| 04_技術聯絡 | #101 + #103 工程施工 | 技術可行性、工期預估 |
| 11_系統架構師 | #101 + #103 + #202 資安審計 | 架構設計、技術選型 |
| 12_安全風控 | #101 + #202 | 安全模型、威脅建模 |
| 13_品質驗證 | #101 + #103 | 測試策略、品質門檻 |

### 缺卡時的處理

1. AI 發現當前指令需要某張人格卡，但 02_人格/ 裡沒有
2. AI 提示：「這個操作建議使用 #103 工程施工型，要現在取用嗎？」
3. 你確認 → AI 走 Furnace 取用流程
4. 你拒絕 → AI 用 #101 通用核心型降級處理

## L3 客製人格觸發條件

AI 在以下情況會建議走 Furnace L3 客製流程：

| 情況 | 範例 | AI 建議 |
|------|------|---------|
| 現有人格卡無法涵蓋專案領域 | 中醫診斷系統、農業物聯網 | 「建議客製一張領域專家人格」 |
| 現有角色配裝不足 | 需要法規合規視角但沒有對應卡 | 「建議客製 L3 法規顧問人格」 |
| 使用者明確要求 | 「我想做一個專門懂加密貨幣的人格」 | 走 Furnace 30_人格熔爐.md |

**L3 客製流程：**
1. AI 讀 Furnace `30_人格熔爐.md`
2. 走客製鍛造規則
3. 產出存入專案 `02_人格/3xx_名稱.md`
4. 不上架 Furnace（L3 只給當前專案用）

> 取用人格 用於手動補充額外人格；基礎角色與依賴人格由自動配發規則處理。

## 角色帶依賴自動配發

新增專案時，AI 不是只帶幾張人格卡，而是：

1. 依專案模式從 Furnace `鍛造間/角色/` 複製對應角色卡到 `04_角色/`
2. 掃描每張角色卡的「預設人格」欄位
3. 聯集去重所有需要的人格卡
4. 從 Furnace `鍛造間/人格/` 複製到專案 `02_人格/`

> Furnace 位置：與 SRI 工作目錄同層的 `熔爐百相/` 資料夾。

### 一般模式自動帶入

角色：01_策略決策、02_創意總監、03_使用者研究、04_技術聯絡
→ 聯集人格：#101, #102, #103, #104, #211, #213（6 張）

### 程式設計模式自動帶入

角色：01-04 + 11_系統架構師、12_安全風控、13_品質驗證
→ 聯集人格：#101, #102, #103, #104, #202, #211, #213（7 張）

### 子專案

同程式設計模式。

### 探索

不複製角色和人格。繼承所屬正式容器的 02_人格/ 和 04_角色/。

### 升級專案（一般→程式設計）

補入缺少的角色（11-13）→ 掃描新角色依賴 → 補入缺少的人格卡。

### 缺卡處理

若 AI 在執行中發現某個角色需要的人格卡不在 02_人格/ 裡：
1. AI 提示：「角色 XX 需要 #YYY，但專案裡沒有。要從 Furnace 取用嗎？」
2. 你確認 → AI 去 Furnace 鍛造間/人格/ 複製過來
3. 你拒絕 → AI 用 #101 通用核心型降級處理

## 變更分級速查

| 等級 | 你需要做什麼 |
|------|-------------|
| **小改** | 不用管，AI 自己改 + 補紀錄 |
| **中改** | 不用管，但 AI 會標 `待複核`，你之後可以看 |
| **大改** | **AI 會先問你**，你確認後才改 |

> 什麼算大改：改模組主結構、刪功能、換核心依賴、改 schema、影響 3+ 個正式檔案的重構、改安全/權限/金流。

---

## 權限模型

### 核心原則

**人格不決定權限，權限由 AI 當前所在位置決定。**

- 人格 = 能力（會什麼、怎麼想、用什麼框架）
- 權限 = 上下文邊界（在哪裡、能做到哪裡）

**上下文由最後一個「進入」指令決定。** 站在專案層（`進入專案`）適用專案容器規則；站在執行單元層（`進入執行單元`）適用執行單元邊界規則。

### 三層權限

**第一層：網路**
- 自由讀取，不用問
- 但涉及登入、敏感權限、外部帳密 → 要先問

**第二層：當前專案容器（主專案 / 子專案）**
- 本容器內 CRUD → 不用問
- 大改仍依變更分級先問

**第三層：執行單元工作區（07_ 內的子資料夾）**
- 本執行單元內 CRUD → 不用問
- 離開本執行單元 → ⚠️ 先問使用者
- 離開本專案 → ⚠️ 先問使用者
- 確認後跨域 → **僅限 Read**
- 跨域永遠 **禁止** Create / Update / Delete

### 速查表

| AI 要做的事 | 權限 | 要不要問 |
|------------|------|---------|
| 在本執行單元寫程式 | CRUD | 不用問 |
| 在本專案改 14_總方案 | CRUD | 不用問（大改要問） |
| 上網查技術文件 | 讀取 | 不用問 |
| 想看同專案其他執行單元的做法 | Read | **要先問** |
| 想看隔壁專案的 14_總方案 | Read | **要先問** |
| 想改隔壁專案的檔案 | **禁止** | — |
| 想改隔壁執行單元的程式 | **禁止** | — |
| 用外部帳密登入服務 | 視情況 | **要先問** |
````

## 不建議啟動 Harness 的情況

以下情況，AI 應建議改用第二層（指令模式）：

- 需求本身仍非常模糊，連目標都講不清楚
- 使用者明確表示只是想聊天探索，不想自動推進
- 缺少大量關鍵資料，無法形成可執行計畫
- 任務高度依賴外部登入、敏感權限或人類手動操作
- 任務主要是美學判斷、風格取捨、創意偏好
- 只是單輪簡單修改，不需要完整自主循環
- 使用者正在逐步微調，不希望 AI 自動連續推進

**處理方式：** Harness 不直接啟動，而是回應：「這個任務目前較適合用指令模式逐步推進。若你要，我可以先做 Step 0 計畫，再由你決定是否開啟自主模式。」

## Harness 啟動流程

### Step 0：目標解析

AI 收到 Harness 觸發後，先做目標解析：

```
1. 解析目標範圍
   - 目標是什麼（做一個爬蟲？完成施工藍圖？從零到部署？）
   - 起點在哪（全新專案？既有專案中途接手？）
   - 終點在哪（到哪個階段算完？）

2. 定位當前上下文
   - 目前在哪個正式容器（主專案 / 子專案）
   - 目前是否位於沙盒（探索 / 子專案分流）
   - 目前是否已進入某個執行單元
   - 若使用者未明確指定 → 依最近一次「進入」記錄判定
   - 若有多個候選或名稱歧義 → 暫停詢問使用者
   - ⚠️ 上下文定位失敗 → 不得推進，必須先問

3. 判斷專案模式
   - 一般模式 or 程式設計模式
   - 若還沒有專案 → 先建專案

4. 盤點角色與人格
   - 掃描 04_角色/ 有哪些角色
   - 掃描 02_人格/ 有哪些人格卡
   - 判斷當前角色依賴的人格是否齊全
   - 缺卡 → 暫停，建議從 Furnace 取用或 L3 客製

5. 提出推進計畫
   - 列出預計經過的階段
   - 列出預計會觸發的閘門
   - 列出已知的風險或不確定
```

**Step 0 完成後，必須讓使用者確認計畫才開始推進。**

### Step 1～N：自主推進循環

```
while 目標未完成:

    ① 判斷當前階段
       讀 14_總方案.md / 20_施工藍圖.md / 30_施工進度.md
       判斷：規劃 / 架構 / 施工 / 運維

    ② 判斷當前該做什麼
       讀當前階段的中間層（12_待確認問題 / 13_已確認結論）
       決定下一步對應指令.md 的哪一條指令

    ③ 判斷需要哪個角色
       依指令.md 的角色配裝表，確認對應人格卡

    ④ 檢查是否需要停下來（見「人類介入點」）
       需要 → 暫停，報告現況，等使用者確認
       不需要 → 繼續

    ⑤ 執行指令
       依第二層規則執行
       寫入對應產物

    ⑥ 更新進度
       更新 10_對話整理.md（快照區）
       若有角色觀點，更新 04_角色/
       若階段切換，更新系統總覽狀態區

    ⑦ 判斷是否達到目標終點
       是 → 產出完成報告，結束 Harness
       否 → 回到 ①
```

### 每輪推進粒度規則

Harness 每輪自主推進，**只允許完成一個最小可驗證步驟**。不得在同一輪內同時做多個跨階段決策。

**最小可驗證步驟範例：**
- 完成一次待確認問題收斂
- 完成一次已確認結論整理
- 完成一份施工藍圖草稿更新
- 完成一個執行單元骨架建立
- 完成一次角色審查
- 完成一次參考資料整理
- 完成一次程式執行與結果寫入

**禁止：**
- 同一輪同時做「需求收斂 + 藍圖重寫 + 開始施工」
- 同一輪跨越多個閘門
- 同一輪自行決定多個重大方向

## 人類介入點（寫死）

### 必須停下來問

| 情況 | 為什麼 |
|------|--------|
| **大改** | 觸及真值核心，不能自主決定 |
| **閘門：架構→施工** | 施工藍圖必須人類確認「可施工」 |
| **閘門：最終交付** | 完成產出後必須人類確認 |
| **金錢 / 安全 / 法律決策** | 風險太高 |
| **沙盒回流** | 結論進主線必須人類確認 |
| **跨專案 / 跨執行單元讀取** | 權限模型要求先問 |
| **缺人格卡** | 需要使用者決定取用或客製 |
| **連續 3 次推進失敗** | 避免 AI 在錯誤方向上空轉 |
| **遇到矛盾或不合理** | 安全規則要求先停 |

### 可以自主繼續

| 情況 | 對應指令 |
|------|---------|
| 小改 | 直接改 + 補紀錄 |
| 中改 | 直接改 + 標 `待複核` |
| 中間層寫入（10/11/12/13） | 對話整理、分流、收斂 |
| 角色更新 | 更新 04_角色/00 和角色 md |
| 參考資料整理 | 整理 03_參考資料/ |
| 壓縮對話 | 更新快照 |
| 上網查資料 | 不受限 |

## 階段轉換規則

Harness 在每個階段轉換點都要做一次檢查：

### 規劃 → 架構

```
前置檢查：
□ 14_總方案.md 願景、目標使用者、問題定義、成功條件已填寫
□ 12_待確認問題.md 無阻擋性的 #尚未回答
□ 14_總方案.md 的「工程交接所需關鍵約束」已填寫
□ 專案模式 = 程式設計（一般模式無架構階段）

全部通過 → 自主進入架構
任一未通過 → 暫停，列出缺項，問使用者
```

### 架構 → 施工

```
前置檢查：
□ 20_施工藍圖.md 狀態 = 「已確認可施工」

⚠️ 這是硬閘門。即使其他檢查全通過，此項未通過就不能進。
Harness 必須暫停，讓使用者確認藍圖。
```

### 施工 → 運維

```
前置檢查：
□ 30_施工進度.md 施工目標全部完成
□ 31_施工日誌.md 已更新
□ 執行單元 README.md 已寫好
□ 無未處理的回流需求

全部通過 → 暫停，產出完成報告，等使用者確認交付
（施工→運維不自動跳，一律等人確認）
```

## 進度報告

Harness 在以下時機產出進度報告：

| 時機 | 報告內容 |
|------|---------|
| 每次暫停（閘門/大改/問使用者） | 做到哪了、為什麼停、下一步建議 |
| 階段轉換時 | 本階段摘要、下一階段計畫 |
| 目標完成時 | 完成報告（見下方格式） |
| 連續自主推進超過 5 輪 | 中間報告，避免失控 |

### 進度報告格式

```
## Harness 進度報告

- 模式：限定範圍自主推進 / 端到端自主推進
- 目標：[原始目標]
- 當前階段：規劃 / 架構 / 施工 / 運維
- 已完成：
  - [列出已完成的關鍵步驟]
- 當前狀態：
  - [正在做什麼 / 為什麼暫停]
- 待決事項：
  - [需要使用者決定的事]
- 下一步：
  - [Harness 接下來打算做什麼]
- 風險：
  - [已知風險或不確定]
```

### 完成報告格式

```
## Harness 完成報告

- 目標：[原始目標]
- 結果：✓ 完成 / △ 部分完成（附原因）
- 經過的階段：[規劃 → 架構 → 施工]
- 產出清單：
  - 14_總方案.md — [摘要]
  - 20_施工藍圖.md — [摘要]
  - 30_施工進度.md — [摘要]
  - 07_主專案程式/XXX — [摘要]
- 閘門確認紀錄：
  - [列出每次暫停和使用者確認的摘要]
- 未解決事項：
  - [若有]
- 建議下一步：
  - [運維建議 / 迭代建議]
```

## 可中斷 / 可恢復

### 隨時可中斷

使用者隨時可以說：

- `暫停` → Harness 停在當前步驟，產出進度報告
- `停止` → Harness 結束，切回第二層（指令.md）模式
- 直接下第二層指令 → Harness 自動讓位，切回半自動

### 可恢復

- `繼續推進` → Harness 從上次暫停點繼續
- `重新推進到 [目標]` → Harness 重新做 Step 0 目標解析

### 降級

Harness 有兩個獨立計數器：

**A. 推進失敗計數器**（方向錯、方案撞牆、資訊矛盾、操作失敗且無法自修）
→ 連續 3 次 → **必須停下來，報告原因，等使用者決定**

**B. 介入停頓計數器**（因大改/閘門/跨域/缺卡/上下文不明而暫停）
→ 連續 3 次 → **自動建議降級回指令模式**

規則：
- 兩個計數器分開計，不混用
- 使用者確認並明確指示後，可重置介入停頓計數器
- 使用者未回覆不算新的一次，只算等待中

## Harness 的權限

Harness 遵守指令.md 的權限模型，無例外。

- 本專案 / 本執行單元：CRUD
- 離開前：先問
- 跨域：Read only
- 上網：不受限
- 上下文由最後一個「進入」決定
- **當多個邊界同時成立時，以最窄的當前邊界為準。** 例如：位於執行單元內時，執行單元邊界優先於專案容器邊界。

**Harness 不因為是自動模式就獲得更高權限。**

## Harness 寫入邊界

### 可自主寫入

## 安全規則

1. **Harness 不發明新規則** — 只調用第二層指令，不繞過
2. **Harness 不繞過閘門** — 閘門是寫死的，Harness 也要停
3. **Harness 不繞過變更分級** — 大改一律問
4. **Harness 可被隨時中斷** — 使用者說停就停
5. **Harness 留完整日誌** — 每個自主決策記錄在 10_對話整理.md
6. **每輪只做一個最小可驗證步驟** — 不跨階段連做
7. **連續自主推進不超過 5 輪不報告** — 避免失控
8. **推進失敗計數器 3 次必停** — 避免空轉
9. **介入停頓計數器 3 次建議降級** — 避免 Harness 不適合當前任務

## 附加能力：專家召喚

專家召喚不是獨立推進模式，而是 Harness 在自主推進過程中，為了深度審查、專業判斷或單次分析而呼叫的附加能力。

**觸發方式：**
- 使用者口語指定（「叫 CEO 來看方案」）
- Harness 判斷當前步驟需要深度審查時主動建議
- 使用者確認後觸發

**特性：**
- 單次介入，完成即退場
- 不改變當前 Harness 主目標
- 不獲得額外權限
- Harness 推進中觸發後，完成即繼續推進

**也可以獨立使用：** 不開 Harness，直接說「叫 CEO 來看方案」，走單次專家介入→退場。

# SRI 系統說明

SRI（Standard Archetype）是一套 AI 專案管理系統。它定義了專案的結構、流程、閘門與角色，讓 AI 能有紀律地與人類協作。

## 核心概念

- **SRI** = 專案管理 Harness（結構、規則、流程）
- **Furnace** = 人格鍛造工坊（AI 能力卡與角色卡的供應端）
- 兩者獨立，SRI 只從 Furnace 取用成品

## S-07. 人格配發規格

> 本節不是模板檔案，而是描述專案 `02_人格/` 的配發邏輯。
> 供完整初始化、新增專案、系統更新時參照。

### 配發規則

| 時機 | 行為 |
|------|------|
| 新增專案 | 依模式複製角色 → 掃描角色依賴 → 聯集去重 → 複製人格 |
| 升級專案 | 補入新角色 → 掃新角色依賴 → 補入缺少人格 |
| 新增子專案 | 同程式設計模式 |
| 新增探索 | 不複製，繼承所屬正式容器 |
| 使用者說「取用人格」 | 走 Furnace 取用流程（手動補充額外人格） |
| 需要客製 | 走 Furnace L3 客製流程，產出存入 02_人格/ |

### 自動配發結果

| 模式 | 角色 | 人格（聯集去重） |
|------|------|----------------|
| 一般 | 01-04（4 張） | #101, #102, #103, #104, #211, #213（6 張） |
| 程式設計 | 01-04 + 11-13（7 張） | #101, #102, #103, #104, #202, #211, #213（7 張） |

### 02_人格/ 內容規範

- 檔名格式：`[三位數編號]_[名稱].md`
- L3 客製人格編號從 301 起
- SRI 不主動同步 Furnace 更新
- AI 日常讀取 02_人格/ 內所有卡作為思維框架
