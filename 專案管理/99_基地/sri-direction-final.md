# SRI v4.0.0 當前方向總結

> 日期：2026-04-01
> 狀態：v4 定版施工完成，可進入整批替換

---

## 一、v4 的核心改動

v4 不是小修，而是主版本升級。核心改動只有六件事：

1. `01_團隊/` 合併人格與角色。
2. `02_參考資料/` 與 `03_文件產出/` 分清 input / output。
3. `04_主專案程式/` 正式成為交付與運行區。
4. `05_沙盒區/` 取代舊沙盒編號。
5. `99_基地/mcp/` 從 future 移入 current。
6. 10-15 自動更新、持續性角色討論、程式碼修改閘門寫死。

## 三、6 份 source 的定位

| 檔案 | 定位 |
|------|------|
| `sri-core.md` | 結構、編號、指令總表、階段與安全規則 |
| `sri-templates.md` | 專案容器模板 |
| `sri-ai-behavior.md` | AI 讀取邏輯、角色機制、Furnace / MCP 邊界 |
| `sri-ai-templates.md` | 系統層與衍生物模板規格 |
| `sri-history.md` | 歷史與退役對照 |
| `sri-future.md` | 仍未落地的 future 議題 |

**規則：永遠先改 source，再改衍生物。**

## 五、v4 已定案的硬規則

1. 內文正式路徑以 `sri-core-v4.md` 為唯一參考。
2. 10-15 會背景自動更新，`對話整理` 不再是唯一入口。
3. 角色討論是持續性內部機制，不再只是一次性審查。
4. `04_主專案程式/` 內任何非 `.md` 修改前必停確認。
5. MCP 首次連線或權限升級必停確認。
6. 探索結論不得直接跳進 14 / 20 / 30。


```text

  → 主線 13
  → 主線 14

  → 30
```

## 八、替換原則

v4 採平行建置策略：

1. 先建立 `-v4` 檔案。
2. 內容直接寫 v4 正式路徑，不寫暫存路徑。
3. 驗證完成後，再整批替換正式檔名。
4. `sri-history.md` 保留舊路徑對照，其餘 active docs 盡量不殘留 v3 名稱。
