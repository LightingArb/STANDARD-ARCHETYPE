## 1. 本手冊定位

本手冊是 SRI 三層操作架構中的**第一層**——專案管理使用手冊。

```
第一層：本手冊         → 資料夾怎麼建、檔案怎麼放、編號怎麼排
第二層：指令.md        → 人＋AI 協作流程（誰什麼時候進場、怎麼推進）
第三層：大絕招.md      → AI 自主推進（人類確認關鍵節點）
```

**本手冊的驗證標準：** 一個不懂 SRI 的人，只靠這份手冊，能手動從零跑完一個專案。不需要 AI，不需要第二層或第三層。

**適用對象：** 所有專案參與者。無論後續是否使用 AI 協作，本手冊定義的結構與規則都是基礎。

## 5. 專案模式

每個專案在建立時選定模式，記錄在 `14_總方案.md` 的 metadata 區。

| 模式 | 適用 | 包含階段 | 包含檔案 |
|------|------|---------|---------|
| **一般** | 非程式專案（策略、研究、規劃） | 規劃 → 運維 | 10-15 + 02/03/04/08/99 |
| **程式設計** | 軟體專案 | 規劃 → 架構 → 施工 → 運維 | 10-31 + 02/03/04/07/08/99 |

**模式判定規則（寫死）：**
1. 唯一正式來源：該容器 `14_總方案.md` 的 `專案模式` 欄位
2. `系統總覽.md` 的模式欄位只做路由快取，不是正式判定來源
3. 一般模式可升級為程式設計模式（不可逆）
4. 升級時補建：20/21/30/31、程式設計角色組（11-13）、07_主專案程式/，並依新增角色的預設人格補入缺少的人格卡。

## 6. 沙盒區

沙盒區（`08_沙盒區/`）統一管理所有分流。

### 6.1 兩種沙盒

| 類型 | 前綴 | 結構 | 用途 | 誰能開 |
|------|------|------|------|--------|
| **探索** | 字母（A_、B_、C_） | 10-13 + 03_參考資料/ | 輕量比較、測試方向 | 所有專案 |
| **子專案** | 數字（01_、02_） | 完整架構（同程式設計主專案） | 正式子系統開發 | 僅程式設計模式主專案 |

### 6.2 轉換規則

- **探索 → 子專案（升級）：** 補齊完整架構，保留已有內容
- **子專案 → 探索（降級）：** 多餘檔案移入 `99_舊檔案封存區/`
- 轉換前必須取得使用者確認

### 6.3 沙盒硬規則

- **子專案固定為程式設計模式。** 一般模式專案只能開探索。
- **可遞迴。** 子專案內可再開 `08_沙盒區/`，深度不建議超過 2 層。
- **探索結論不是正式真值。** 要影響主線，必須走回流路徑（見 §9.6）。

### 7.1 真值優先序（由高到低）

```
14_總方案.md            → 規劃階段正式真值
15_方案變更紀錄.md      → 規劃階段正式留痕
20_施工藍圖.md          → 架構階段正式真值（僅程式設計模式）

30_施工進度.md          → 施工階段正式真值（僅程式設計模式）
31_施工日誌.md          → 施工階段正式留痕（僅程式設計模式）
13_已確認結論.md        → 寫入正式方案的穩定前提
────── 以下不是正式真值 ──────
12_待確認問題.md        → 待收斂
```

**禁止：** 把 10/11/12 或 03_參考資料/ 的內容當作正式結論引用。引用前必須先說明其非正式角色。

### 7.2 檔案可寫性總覽

| 檔案 | 維護者 | 可寫性 |
|------|--------|--------|
| `99_基地/sa/sri-*.md` | 使用者透過 git | 使用者唯一 |

| `03_參考資料/` | 依整理規則維護 | 可寫 |

### 7.3 版本基底關聯（僅程式設計模式）

- `20_施工藍圖.md` 必須標明：基於哪版 `14_總方案.md`、最後同步時間

- 發現上游已更新但下游未同步時，先停下來確認，不得默默沿用過期基底

## 8. 四個階段

### 8.0 階段總覽

```
一般模式：     規劃 ──────────────────────────→ 運維
程式設計模式：  規劃 ──→ 架構 ──→ 施工 ──→ 運維
```

每個階段有明確的**真值邊界**——負責什麼、不負責什麼。跨階段時需確認上游真值是否最新。

### 8.1 規劃階段

**目標：** 把「這個專案到底要做什麼」講清楚。

**真值範圍：** 願景、目標使用者、問題定義、成功條件、功能範圍、階段邊界、核心使用者流程、商業邏輯、需求優先序。

**不負責：** 寫程式、模組拆分、API 設計、技術選型、schema 設計、施工順序、測試實作。

**正式產物：**
- `14_總方案.md`（正式真值）
- `15_方案變更紀錄.md`（正式留痕）

**三個討論維度：**

| 維度 | 觸發條件 | 態度 |
|------|---------|------|
| 方向定位 | 14 願景欄位空白或草稿 | 先把大方向講清楚，不急著談細節 |
| 問題處理 | 14 有穩定願景，12 還有未回答的核心問題 | 找缺口、矛盾、未定義 |
| 想法探索 | 主要方向已收斂，以探索為主 | 補盲點、變體、替代方案 |

**遞進邏輯：** 先確認願景 → 再處理待確認問題 → 最後探索延伸。

**規劃階段完成條件：**
- `14_總方案.md` 願景、目標使用者、問題定義、成功條件、功能範圍已填寫且穩定
- 無 `#尚未回答` 的核心問題阻擋下一階段
- 若為程式設計模式：`14_總方案.md` 的「工程交接所需關鍵約束」已填寫

### 8.2 架構階段（僅程式設計模式）

**目標：** 把已收斂的規劃方案翻成可施工、可驗證、可交付的工程藍圖。

**真值範圍：** 系統架構、模組邊界、資料流、控制流、API 契約、資料模型/schema、狀態模型、測試策略、驗證策略、施工順序、風險與回退方案、技術約束、開發階段切分。

**不負責：** 寫程式、跑實際測試、決定產品願景、決定功能範圍、替規劃做需求裁切。

**正式產物：**
- `20_施工藍圖.md`（正式真值）
- `21_藍圖變更紀錄.md`（正式留痕）

**進場前置：** 比對 `14_總方案.md` 與藍圖版本基底 → 不一致先停。

**架構深審觸發條件（任一即建議啟動）：**
- 重大架構切換
- 不可逆技術選型
- 多模組邊界重切
- 共享 API 契約定版
- schema 主結構定版
- 安全/權限模型定版
- 部署/狀態管理主幹定版

**架構階段完成條件：**
- `20_施工藍圖.md` 狀態為「**已確認可施工**」
- `21_藍圖變更紀錄.md` 已補變更摘要
- 版本基底已標記

### 8.3 施工階段（僅程式設計模式）

**目標：** 依已確認可施工的藍圖進行實際施工、測試、修正、交付與回報。

**真值範圍：** 實作進度、已完成模組/功能、已實作與藍圖差異、實際阻斷、測試結果、臨時 workaround、需回流的施工問題。

**不負責：** 決定願景、決定功能範圍、決定正式架構藍圖、把 workaround 升格成正式藍圖。

**正式產物：**
- `30_施工進度.md`（正式真值）
- `31_施工日誌.md`（正式留痕）

**閘門（寫死）：** `20_施工藍圖.md` 狀態不是「已確認可施工」→ 不得進入施工。

**進場前置：** 比對 `20_施工藍圖.md` 與施工進度版本基底 → 不一致先停。

**施工中遇到問題的分流：**

| 情況 | 處理 |
|------|------|
| 不改功能範圍、API、安全模型、藍圖主結構 | 本地解決，施工日誌補 `[快速確認]` |
| 藍圖細節不完整、技術不可行、邊界需重定 | 回架構階段 |
| 功能範圍需刪減、核心流程需重定 | 回規劃階段 |

**審查深度三級：**

| 等級 | 適用場景 | 處理 |
|------|---------|------|
| 第一級：日常 | 修 bug、補小功能、加註解、調樣式 | 不需獨立報告 |
| 第二級：深度 | 認證/授權、金錢、個資、外部 API、關鍵狀態流 | 簡短說明風險 + 跑必要測試 |
| 第三級：全套 | 新增/刪除模組、重大重構、換核心依賴、改 schema | 完整 review + 人類確認 |

判不清時預設第二級，寧可升級不降級。

**施工日誌格式：**

```
## YYYY-MM-DD HH:mm｜施工階段：X
- 類型：[正常] / [快速確認] / [重大阻斷]
- 當前施工目標：
- 本次完成：
- 阻斷 / 異常：
- 是否偏離藍圖：否 / 是（簡述）
- 是否需要回流：否 / 架構 / 規劃
- 下一步：
```

**施工階段完成條件：**
- 本輪施工目標已完成
- `31_施工日誌.md` 已更新
- `30_施工進度.md` 已更新
- 偏差已標記
- 回流需求已說明

### 8.4 運維階段

**目標：** 施工完成後的持續維護、監控、修正與迭代。

**何時進入：**
- 一般模式：規劃完成、方案執行中或執行後
- 程式設計模式：施工目標全部完成，系統已部署或交付

**運維的五種活動：**

| 活動 | 說明 | 回流到哪個階段 |
|------|------|--------------|
| 日常維護 | 小 bug 修復、文件更新、依賴升級 | 不回流，直接處理 + 日誌留痕 |
| 監控與回報 | 系統狀態、效能指標、使用者回饋收集 | 不回流，記錄到 `10_對話整理.md` |
| 問題排查 | 非預期行為、效能異常、安全事件 | 視嚴重度：輕→本地解決；重→回施工或架構 |
| 功能迭代 | 新需求、功能擴充、使用者回饋驅動的改善 | 回規劃（新一輪 12→13→14） |
| 退役或移交 | 系統下線、交接給其他團隊 | 不回流，產出移交文件 |

**運維階段的檔案使用：**
- `30_施工進度.md`：更新為運維狀態（標記「已上線」或「已交付」）
- `31_施工日誌.md`：繼續追加運維紀錄，類型標 `[運維]`
- `10_對話整理.md`：記錄運維期間的討論與決策
- `12_待確認問題.md`：收集運維中浮現的新需求或問題

**運維日誌格式：**

```
## YYYY-MM-DD HH:mm｜運維
- 類型：[日常維護] / [問題排查] / [功能迭代] / [監控回報]
- 內容摘要：
- 影響範圍：
- 是否需要回流：否 / 施工 / 架構 / 規劃
- 處理狀態：已解決 / 追蹤中 / 已建立新需求
- 下一步：
```

**運維 → 新一輪迭代的判定：**
- 只影響既有程式碼的小修 → 留在運維
- 需要改 API、schema、模組邊界 → 回架構（視為新一輪）
- 需要改功能範圍、核心流程、成功條件 → 回規劃（視為新一輪）
- 新一輪迭代可在沙盒中進行，不影響線上版本

**一般模式的運維：**
- 一般模式無施工日誌，運維紀錄寫在 `10_對話整理.md`
- 方案執行狀況追蹤在 `14_總方案.md` 底部加「執行追蹤」區塊
- 新需求回流到 `12_待確認問題.md`

## 9. 中間層運作機制

### 9.1 專案隔離

每個專案獨立一組中間層。**不同專案的中間層不得共用、不得混寫、不得互相引用作為正式前提。**

### 9.2 結構化議題格式

#### 完整記錄格式（用於 10/11/12）

```markdown
### [議題標題]
- 類型：single_select / multi_select / rank_priorities / open_ended
- 問題：[完整問題文字]
- 選項：
  A) [選項 A]
  B) [選項 B]
  C) [選項 C]
- 使用者選擇：B
- AI 追問：[若有後續問題]
- 備註：[若有補充脈絡]
- 狀態：#尚未回答 / ✓ 議題關閉 / ⚠ 使用者決定不處理
```

#### 精簡記錄格式（用於 13）

```markdown
### [議題標題]
- 問題：[一句話版]
- 結論：[選項或摘要]
- 確認日期：YYYY-MM-DD
```

#### 格式規則

1. `12_待確認問題` 使用完整格式，每題必須有狀態
2. `11_想法暫存區` 使用完整格式，狀態可省略
3. `10_對話整理` 使用完整格式記錄本輪議題
4. `13_已確認結論` 使用精簡格式
5. 開放式問題：類型標 `open_ended`，選項寫「（開放式）」

### 9.3 三種關閉狀態

- `✓ 議題關閉`：已有明確結論
- `⚠ 使用者決定不處理`：已知但暫不決定
- `#尚未回答`：仍待回答

### 9.4 已確認結論分區

```markdown
# 已確認結論

## 已確定區
（穩定、可作為寫入正式方案前提的結論）

## 不確定區
（已討論但仍有疑慮、待後續確認）
```

### 9.5 中間層生命週期

- 新增專案時建立全套空白模板，不自動寫入內容
- 中間層寫入只在明確下達對應指令時才發生
- 中間層不可取代正式主檔
- 寫入正式方案完成後不清空；在頂部標記「已整理歸檔」+ 日期 + 對應正式產物版本
- 下一輪需要新議題時，優先建立新檔（如 `12_待確認問題-02.md`）避免覆蓋

### 9.6 回流路徑（寫死）

沙盒結論要影響主線，必須走以下路徑：

```
沙盒 13_已確認結論
  → 所屬正式容器 12_待確認問題
  → 所屬正式容器 13_已確認結論
  → 所屬正式容器 14_總方案
  →（若程式設計模式）20_施工藍圖
  →（若程式設計模式）30_施工進度
```

**禁止：** 沙盒結論直接跳進正式容器正式產物。

結論成功回流後，在沙盒 `13_已確認結論.md` 補標記「已回流 YYYY-MM-DD → 正式容器 12」。

### 9.7 探索的角色共用規則

- 探索共用所屬正式容器的 `04_角色/`，不另建角色
- 探索內寫回正式容器 `04_角色/` 時，必須加 `[沙盒-X]` 標籤
- `[沙盒-X]` 的內容視為支線觀察，不是正式結論
- 完成回流後，才可整理為主線歷史摘要（移除標籤）

## 10. 變更管理

### 10.1 變更分級

| 等級 | 定義 | 處理 |
|------|------|------|
| **小改** | 只影響本階段或下游執行細節，不影響上游真值核心 | 直接改 + 補變更紀錄 |
| **中改** | 影響其他階段，但未動到願景、範圍、核心功能、成功條件、安全邊界 | 直接改 + 補變更紀錄 + 加 `待複核` 標記 + 明示受影響階段 |
| **大改** | 觸及真值核心 | **必須先問使用者** → 確認後改正式產物 + 補變更紀錄 |

### 10.2 變更紀錄格式

```
## YYYY-MM-DD HH:mm｜變更等級：小改 / 中改 / 大改

- 修改對象：[哪份正式產物]
- 改了什麼：
- 為什麼改：
- 影響範圍：
- 狀態：已生效 / 待複核 / 已確認
```

### 10.3 跨階段防呆（以下一律視為大改）

- 改模組主結構
- 刪功能 / 刪正式主檔
- 更換核心依賴
- 改 schema 主結構
- 影響超過 3 個正式檔案的重構
- 與既有規劃不一致的方向調整
- 改安全模型 / 權限模型 / 金流邏輯

## 11. 參考資料管理

### 11.1 雙入口結構

- **`03_參考資料/資料地圖.md`** — 找檔入口。記錄所有資料的路徑、類型、來源、主題分類。
- **`03_參考資料/資料館長.md`** — 內容理解入口。包含：整體總覽、分類分析、單檔分析、關聯與回流建議、缺口與待補。

### 11.2 三種資料模式

| 模式 | 輸入範例 | 處理方式 | 產出 |
|------|---------|---------|------|
| 分析型 | 起訴書 PDF | 全文整理 → 摘要 | 資料館長更新 + 歸入 finished_analysis/ |
| 資料型 | 歷史報價 CSV | 不分析內容，只描述結構 | 資料夾 + metadata md |
| 快取型 | 網頁快取 | 分類存放 + 建索引 | 主題資料夾 + 資料地圖更新 |

### 11.3 一致性規則

允許兩種狀態交替：
- **狀態 A：未整理輸入狀態** — 使用者可直接丟東西進去
- **狀態 B：整理完成狀態** — 執行整理後的一致狀態

一致性檢查：
- 實體存在但未出現在資料地圖 → 錯誤
- 資料地圖有寫但實體不存在 → 錯誤

## 12. 07_主專案程式規範

> 本節僅適用程式設計模式。

### 12.1 定位

`07_主專案程式/` 是**交付與運行區**——所有可執行、可輸出、可 review、可部署的實體單元都在這裡。

```
外層（10-31, 02-04, 08, 99）= AI 管理世界（討論、決策、審查）
07_主專案程式/               = 實際運行世界（程式碼、輸出資料、部署來源）
```

- 工程師 code review → 進 07_ 看
- AI 執行程式 → 進 07_ 跑
- 客戶要拿正式輸出 → 進 07_ 拿
- 部署到雲端 → 從 07_ 整包拉走

### 12.2 執行單元

07_ 裡的每個子資料夾稱為一個**執行單元**。每個執行單元自成一體，獨立可運行。

**命名格式：** `[三位數流水號]_[名稱](探索)` 或 `[三位數流水號]_[名稱](專案)`

```
07_主專案程式/
├── 001_trivago-scraper(探索)/       ← 探索版：先抓資料看可行性
├── 002_booking-scraper(探索)/       ← 探索版：試另一個來源
├── 003_multi-platform-price(專案)/  ← 專案版：正式多平台系統
└── 004_data-pipeline(專案)/         ← 專案版：資料處理管線
```

### 12.3 兩種執行單元

| | 探索版 | 專案版 |
|---|--------|--------|
| **用途** | 試可行性、抓資料看看、跑分析 | 正式可部署的系統 |
| **README.md** | ✓ 必備 | ✓ 必備 |
| **內容** | 自由（md、資料、簡單腳本） | 完整程式架構 |
| **可部署** | 不要求 | 必須可部署 |
| **可 review** | 不要求 | 必須可 review |
| **output/** | 建議有 | 必備 |
| **src/** | 非必備 | 必備 |
| **tests/** | 非必備 | 建議必備 |

**探索版最低骨架：**

```
001_trivago-scraper(探索)/
├── README.md              ← 必備：這個探索在做什麼
├── output/                ← 建議：輸出資料放這裡
└── （其他自由安排：md 筆記、抓下來的資料、簡單腳本）
```

**專案版最低骨架：**

```
003_multi-platform-price(專案)/
├── README.md              ← 必備：操作手冊 + AI 執行入口
├── src/                   ← 必備：原始碼
├── output/                ← 必備：執行輸出
├── tests/                 ← 建議：測試
├── config/                ← 選用：設定檔
└── docs/                  ← 選用：技術文件
```

### 12.4 轉換規則

- **探索 → 專案（升級）：** 補齊 src/、output/、tests/ 等結構，更新 README，改括號標記
- **專案 → 探索（降級）：** 多餘結構移入 99_舊檔案封存區/，改括號標記
- 轉換前必須取得使用者確認

### 12.5 與 08_沙盒區的差異

兩邊都有探索和正式版，但**不是同一回事**：

| | 08_沙盒區 | 07_主專案程式 |
|---|-----------|-------------|
| **世界** | AI 管理世界（討論/決策） | 實際運行世界（執行/交付） |
| **探索的意思** | 方案探索：「這個方向值不值得做？」 | 執行探索：「這個技術可不可行？」 |
| **正式版的意思** | 子專案：完整的 SRI 架構與討論 | 專案版：可部署的正式系統 |
| **內容** | 10-13、03_參考資料、角色 | 程式碼、輸出資料、操作手冊 |
| **回流** | 沙盒結論 → 主線 12 → 13 → 14 | 無回流概念，直接是交付物 |

### 12.6 隔離規則

**執行單元之間禁止 import。** 每個執行單元是獨立的，不得直接引用其他單元的程式碼或資料。

若需要共用邏輯，有兩種做法：
1. 提取成獨立的共用執行單元（如 `005_shared-utils(專案)/`）
2. 在進化版中整合（如從 001 + 002 進化成 003）

README 不只是安裝說明，而是**執行入口契約**——AI 和人類都靠這份理解這個單元。

### 12.8 架構師佈局規則

1. `07_主專案程式/` 下要開幾個執行單元、各自職責
2. 哪些是探索版、哪些是專案版
3. 每個專案版執行單元的內部目錄結構

### 12.9 執行單元進化範例

## 13. 人格與角色

### 13.1 人格（能力卡）

**來源：** Furnace（Aspects-of-the-Crucible）。SRI 只取用成品，不介入鍛造過程。

**取用流程：** 從 Furnace 複製到專案 `02_人格/` → 從此斷開、獨立使用。

**分級：**

| 等級 | 編號段 | 說明 |
|------|--------|------|
| L1 | #1xx | 基礎人格（#101 通用核心、#102 產品策略、#103 工程施工、#104 UI/UX 設計） |
| L2 | #2xx | 專業人格（行銷、資安、數據分析、PM、寫作溝通等） |
| L3 | #3xx | 客製人格（針對特定專案鍛造，不上架） |

**`02_人格/` 規範：**
- 檔名格式：`[三位數編號]_[名稱].md`（如 `101_通用核心型.md`）
- 新專案建立時自動配發（見 §13.3）
- SRI 不主動同步 Furnace 更新

### 13.2 角色

角色是專案中的討論視角，用於多面向審視方案。

**來源：** Furnace（Aspects-of-the-Crucible）`鍛造間/角色/`。複製到專案 `04_角色/` 後獨立累積記憶。

**Furnace 位置：** 與 SRI 工作目錄同層的 `熔爐百相/` 資料夾。

**規劃角色組（所有模式）：** 01_策略決策、02_創意總監、03_使用者研究、04_技術聯絡

**架構角色組（僅程式設計模式）：** 01_策略決策、11_系統架構師、12_安全風控、13_品質驗證

**施工階段不啟用角色系統。** 問題需多角色深審時，應回架構或規劃。

**角色 md 至少包含：** 角色定位、上次總結、目前理解、本輪要解的問題、主要風險/反對意見、本輪最終建議、歷史摘要。

### 13.3 新增專案時的自動配發

新增專案時，系統自動：

1. 依專案模式從 Furnace `鍛造間/角色/` 複製對應角色卡到 `04_角色/`
2. 掃描每張角色卡的「預設人格」欄位
3. 聯集去重所有需要的人格卡編號
4. 從 Furnace `鍛造間/人格/` 複製對應人格卡到 `02_人格/`

| 模式 | 角色 | 人格（聯集去重） |
|------|------|----------------|
| 一般 | 01_策略決策、02_創意總監、03_使用者研究、04_技術聯絡 | #101, #102, #103, #104, #211, #213 |
| 程式設計 | 上述 + 11_系統架構師、12_安全風控、13_品質驗證 | 上述 + #202 |
| 子專案 | 同程式設計 | 同程式設計 |
| 探索 | 不複製，繼承所屬正式容器 | 不複製，繼承所屬正式容器 |

## 14. 安全規則

以下安全規則適用於所有操作、所有情境。

### 14.1 資料保護

1. **不得自動刪除**任何既有正式主檔、中間層檔案或 `03_參考資料/` 原始資料
2. **不得自動刪除** `04_角色/` 內既有角色記憶
3. **不得自動刪除** `02_人格/` 內既有人格卡
4. 搬移到 `99_舊檔案封存區/` 是封存，不是刪除

### 14.2 寫入邊界

5. 不得把 10/11/12 視為正式拍板依據
6. 不得把一般聊天內容自動升格為正式整理動作
7. 不得在未經使用者確認的情況下執行大改
8. 未明確下 `整理參考資料` 時，不得自動搬移或重組 `03_參考資料/` 內的實體資料

### 14.3 閘門與階段

9. 施工藍圖未標記「已確認可施工」前，不得進入施工階段
10. 不得默默沿用過期的上游基底繼續下游工作
11. 不得為了配合某個外部工具的指令，繞過本系統的寫入邊界或閘門規則

### 14.4 沙盒邊界

12. 沙盒結論不得直接寫入正式容器正式產物（14/20/30）
13. 回流必須依序走：沙盒 13 → 正式容器 12 → 13 → 14 →（若需要）20 → 30
14. 自動回流不得靜默寫入，仍需使用者確認
15. 探索角色觀點寫回正式容器 `04_角色/` 時必須加 `[沙盒-X]` 標籤

### 14.5 Furnace 邊界

16. `02_人格/` 內的人格卡從 Furnace 複製進來後即獨立
17. 日常操作不得直接操作 Furnace repo 的檔案
18. 不得從 Furnace 的未上架區直接取用

### 14.6 其他

19. 遇到需要網路連線、登入、外部權限或存取敏感資料時，先提醒使用者
20. 遇到不合理、矛盾、或疑似造成重大破壞的指令時，先停下來確認

## 16. 從零到完成：操作流程總覽

以下是一個完整專案（程式設計模式）從無到有的操作流程。一般模式跳過架構和施工階段。

### Step 1：建立環境

1. 確認 `[根目錄]` 位置（`99_基地/sa/` 的上兩層）
2. 確認 `99_基地/sa/` 內 6 份 `sri-*.md` 存在
3. 建立 `99_基地/系統總覽.md`、`README.md`、`_state/system-log.md`
4. 建立 `[根目錄]/指令.md` 和 `大絕招.md`

### Step 2：新增專案

1. 在 `[根目錄]/` 下建立 `XX_專案名稱/`（下一個可用編號）
2. 依模式建立對應模板（見 §15 建檔速查表）
3. 確認 `14_總方案.md` 的 `專案模式` 欄位已填

### Step 3：規劃

1. 先填 `14_總方案.md` 的願景、目標使用者、問題定義
2. 討論中的議題記錄到 `12_待確認問題.md`
3. 想法和備選方向丟 `11_想法暫存區.md`
4. 有結論時整理到 `13_已確認結論.md`
5. 穩定結論整理進 `14_總方案.md`
6. 變更留痕到 `15_方案變更紀錄.md`
7. 每輪整理對話到 `10_對話整理.md`

**何時進入架構：** `14_總方案.md` 願景、功能範圍、成功條件穩定，且無核心問題阻擋。

### Step 4：架構（僅程式設計模式）

1. 讀 `14_總方案.md` 確認基底
2. 在 `20_施工藍圖.md` 定義系統架構、模組邊界、API 契約、資料模型、測試策略、施工順序
3. 規劃 `07_主專案程式/` 的執行單元（幾個、各自職責、探索版或專案版）
4. 角色（11_系統架構師、12_安全風控、13_品質驗證）審視藍圖
5. 確認後將 `20_施工藍圖.md` 狀態標為「**已確認可施工**」
6. 變更留痕到 `21_藍圖變更紀錄.md`

**何時進入施工：** `20_施工藍圖.md` 狀態為「已確認可施工」。

### Step 5：施工（僅程式設計模式）

1. 確認 `20_施工藍圖.md` 閘門通過
2. 依藍圖在 `07_主專案程式/` 建立執行單元（探索版或專案版）
3. 每個執行單元建立 `README.md`
4. 依施工順序實作程式碼
5. 每個施工單元完成後更新 `30_施工進度.md` 和 `31_施工日誌.md`
6. 遇到問題判斷：本地解決 / 回架構 / 回規劃
7. 完成後確認 `README.md` 包含自然語言操作範例與輸出位置說明

**何時進入運維：** 施工目標全部完成。

### Step 6：運維

1. 更新 `30_施工進度.md` 標記「已上線」或「已交付」
2. 日常維護紀錄追加到 `31_施工日誌.md`
3. 新需求回流到 `12_待確認問題.md`
4. 重大迭代開新沙盒或新一輪規劃
