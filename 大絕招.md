# 大絕招

> 本檔是 SRI 三層架構的**第三層入口**——AI 自主推進規則。
> 第一層（結構規範）：專案管理使用手冊.md
> 第二層（協作流程）：指令.md
> Furnace 位置：與 SRI 工作目錄同層的 `熔爐百相/` 資料夾
> 取用入口：熔爐百相/10_老闆.md

---

## 這是什麼

大絕招是 SRI 的**自動駕駛模式**。

它不是新系統，不發明新規則。它用的是同一套專案結構（第一層）和同一套協作流程（第二層），只是把操作者從「人類下指令」換成「AI 自主判斷該下什麼指令」。

```
第二層（指令.md）：人類說「討論專案」→ AI 執行
第三層（大絕招）：AI 自己判斷現在該「討論專案」→ 自己執行
```

---

## 兩種觸發模式

### 模式 1：限定範圍自主推進

```
你：「自主推進到施工藍圖完成」
AI：在規劃→架構範圍內自主推進，到藍圖完成時停下來報告。
```

### 模式 2：端到端自主推進

```
你：「自主完成這個專案」
AI：從規劃推到施工完成，每個閘門停一次讓你確認。
```

---

## 不建議啟動 Harness 的情況

以下情況，AI 應建議改用第二層（指令模式）：

- 需求本身仍非常模糊，連目標都講不清楚
- 使用者明確表示只是想聊天探索，不想自動推進
- 缺少大量關鍵資料，無法形成可執行計畫
- 任務高度依賴外部登入、敏感權限或人類手動操作
- 任務主要是美學判斷、風格取捨、創意偏好
- 只是單輪簡單修改，不需要完整自主循環
- 使用者正在逐步微調，不希望 AI 自動連續推進

**處理方式：** Harness 不直接啟動，而是回應：「這個任務目前較適合用指令模式逐步推進。若你要，我可以先做 Step 0 計畫，再由你決定是否開啟自主模式。」

## Harness 啟動流程

### Step 0：目標解析

AI 收到 Harness 觸發後，先做目標解析：

```
1. 解析目標範圍
   - 目標是什麼（做一個爬蟲？完成施工藍圖？從零到部署？）
   - 起點在哪（全新專案？既有專案中途接手？）
   - 終點在哪（到哪個階段算完？）

2. 定位當前上下文
   - 目前在哪個正式容器（主專案 / 子專案）
   - 目前是否位於沙盒（探索 / 子專案分流）
   - 目前是否已進入某個執行單元
   - 若使用者未明確指定 → 依最近一次「進入」記錄判定
   - 若有多個候選或名稱歧義 → 暫停詢問使用者
   - ⚠️ 上下文定位失敗 → 不得推進，必須先問

3. 判斷專案模式
   - 一般模式 or 程式設計模式
   - 若還沒有專案 → 先建專案

4. 盤點角色與人格
   - 掃描 04_角色/ 有哪些角色
   - 掃描 02_人格/ 有哪些人格卡
   - 判斷當前角色依賴的人格是否齊全
   - 缺卡 → 暫停，建議從 Furnace 取用或 L3 客製

5. 提出推進計畫
   - 列出預計經過的階段
   - 列出預計會觸發的閘門
   - 列出已知的風險或不確定
```

**Step 0 完成後，必須讓使用者確認計畫才開始推進。**

### Step 1～N：自主推進循環

```
while 目標未完成:

    ① 判斷當前階段
       讀 14_總方案.md / 20_施工藍圖.md / 30_施工進度.md
       判斷：規劃 / 架構 / 施工 / 運維

    ② 判斷當前該做什麼
       讀當前階段的中間層（12_待確認問題 / 13_已確認結論）
       決定下一步對應指令.md 的哪一條指令

    ③ 判斷需要哪個角色
       依指令.md 的角色配裝表，確認對應人格卡

    ④ 檢查是否需要停下來（見「人類介入點」）
       需要 → 暫停，報告現況，等使用者確認
       不需要 → 繼續

    ⑤ 執行指令
       依第二層規則執行
       寫入對應產物

    ⑥ 更新進度
       更新 10_對話整理.md（快照區）
       若有角色觀點，更新 04_角色/
       若階段切換，更新系統總覽狀態區

    ⑦ 判斷是否達到目標終點
       是 → 產出完成報告，結束 Harness
       否 → 回到 ①
```

### 每輪推進粒度規則

Harness 每輪自主推進，**只允許完成一個最小可驗證步驟**。不得在同一輪內同時做多個跨階段決策。

**最小可驗證步驟範例：**
- 完成一次待確認問題收斂
- 完成一次已確認結論整理
- 完成一份施工藍圖草稿更新
- 完成一個執行單元骨架建立
- 完成一次角色審查
- 完成一次參考資料整理
- 完成一次程式執行與結果寫入

**禁止：**
- 同一輪同時做「需求收斂 + 藍圖重寫 + 開始施工」
- 同一輪跨越多個閘門
- 同一輪自行決定多個重大方向

## 人類介入點（寫死）

### 必須停下來問

| 情況 | 為什麼 |
|------|--------|
| **大改** | 觸及真值核心，不能自主決定 |
| **閘門：架構→施工** | 施工藍圖必須人類確認「可施工」 |
| **閘門：最終交付** | 完成產出後必須人類確認 |
| **金錢 / 安全 / 法律決策** | 風險太高 |
| **沙盒回流** | 結論進主線必須人類確認 |
| **跨專案 / 跨執行單元讀取** | 權限模型要求先問 |
| **缺人格卡** | 需要使用者決定取用或客製 |
| **連續 3 次推進失敗** | 避免 AI 在錯誤方向上空轉 |
| **遇到矛盾或不合理** | 安全規則要求先停 |

### 可以自主繼續

| 情況 | 對應指令 |
|------|---------|
| 小改 | 直接改 + 補紀錄 |
| 中改 | 直接改 + 標 `待複核` |
| 中間層寫入（10/11/12/13） | 對話整理、分流、收斂 |
| 角色更新 | 更新 04_角色/00 和角色 md |
| 參考資料整理 | 整理 03_參考資料/ |
| 壓縮對話 | 更新快照 |
| 上網查資料 | 不受限 |

## 階段轉換規則

Harness 在每個階段轉換點都要做一次檢查：

### 規劃 → 架構

```
前置檢查：
□ 14_總方案.md 願景、目標使用者、問題定義、成功條件已填寫
□ 12_待確認問題.md 無阻擋性的 #尚未回答
□ 14_總方案.md 的「工程交接所需關鍵約束」已填寫
□ 專案模式 = 程式設計（一般模式無架構階段）

全部通過 → 自主進入架構
任一未通過 → 暫停，列出缺項，問使用者
```

### 架構 → 施工

```
前置檢查：
□ 20_施工藍圖.md 狀態 = 「已確認可施工」

⚠️ 這是硬閘門。即使其他檢查全通過，此項未通過就不能進。
Harness 必須暫停，讓使用者確認藍圖。
```

### 施工 → 運維

```
前置檢查：
□ 30_施工進度.md 施工目標全部完成
□ 31_施工日誌.md 已更新
□ 執行單元 README.md 已寫好
□ 無未處理的回流需求

全部通過 → 暫停，產出完成報告，等使用者確認交付
（施工→運維不自動跳，一律等人確認）
```

## 進度報告

Harness 在以下時機產出進度報告：

| 時機 | 報告內容 |
|------|---------|
| 每次暫停（閘門/大改/問使用者） | 做到哪了、為什麼停、下一步建議 |
| 階段轉換時 | 本階段摘要、下一階段計畫 |
| 目標完成時 | 完成報告（見下方格式） |
| 連續自主推進超過 5 輪 | 中間報告，避免失控 |

### 進度報告格式

```
## Harness 進度報告

- 模式：限定範圍自主推進 / 端到端自主推進
- 目標：[原始目標]
- 當前階段：規劃 / 架構 / 施工 / 運維
- 已完成：
  - [列出已完成的關鍵步驟]
- 當前狀態：
  - [正在做什麼 / 為什麼暫停]
- 待決事項：
  - [需要使用者決定的事]
- 下一步：
  - [Harness 接下來打算做什麼]
- 風險：
  - [已知風險或不確定]
```

### 完成報告格式

```
## Harness 完成報告

- 目標：[原始目標]
- 結果：✓ 完成 / △ 部分完成（附原因）
- 經過的階段：[規劃 → 架構 → 施工]
- 產出清單：
  - 14_總方案.md — [摘要]
  - 20_施工藍圖.md — [摘要]
  - 30_施工進度.md — [摘要]
  - 07_主專案程式/XXX — [摘要]
- 閘門確認紀錄：
  - [列出每次暫停和使用者確認的摘要]
- 未解決事項：
  - [若有]
- 建議下一步：
  - [運維建議 / 迭代建議]
```

## 可中斷 / 可恢復

### 隨時可中斷

使用者隨時可以說：

- `暫停` → Harness 停在當前步驟，產出進度報告
- `停止` → Harness 結束，切回第二層（指令.md）模式
- 直接下第二層指令 → Harness 自動讓位，切回半自動

### 可恢復

- `繼續推進` → Harness 從上次暫停點繼續
- `重新推進到 [目標]` → Harness 重新做 Step 0 目標解析

### 降級

Harness 有兩個獨立計數器：

**A. 推進失敗計數器**（方向錯、方案撞牆、資訊矛盾、操作失敗且無法自修）
→ 連續 3 次 → **必須停下來，報告原因，等使用者決定**

**B. 介入停頓計數器**（因大改/閘門/跨域/缺卡/上下文不明而暫停）
→ 連續 3 次 → **自動建議降級回指令模式**

規則：
- 兩個計數器分開計，不混用
- 使用者確認並明確指示後，可重置介入停頓計數器
- 使用者未回覆不算新的一次，只算等待中

## Harness 的人格調用

```
Harness 判斷當前步驟
  ↓
查看 02_人格/ 有什麼人格卡
  ├── 有需要的 → 直接使用
  └── 沒有 → 暫停，建議取用或 L3 客製
  ↓
以對應人格卡的思維框架執行
  ↓
需要深度審查？
  ├── 是 → 觸發附加能力：專家召喚，完成後繼續推進
  └── 否 → 繼續自主循環
```

## Harness 的權限

Harness 遵守指令.md 的權限模型，無例外。

- 本專案 / 本執行單元：CRUD
- 離開前：先問
- 跨域：Read only
- 上網：不受限
- 上下文由最後一個「進入」決定
- **當多個邊界同時成立時，以最窄的當前邊界為準。** 例如：位於執行單元內時，執行單元邊界優先於專案容器邊界。

**Harness 不因為是自動模式就獲得更高權限。**

## Harness 寫入邊界

### 可自主寫入

在當前上下文權限允許下：

- `10_對話整理.md`
- `11_想法暫存區.md`
- `12_待確認問題.md`
- `13_已確認結論.md`（小改/中改）
- `04_角色/00_角色會議紀錄.md`
- `04_角色/*.md`
- `03_參考資料/` 的整理性內容
- `07_主專案程式/` 當前執行單元內的本地檔案
- 執行單元 `output/` 內的輸出結果

### 必須停下來確認後才可寫入

- `14_總方案.md` 的大改
- `20_施工藍圖.md` 的大改
- `30_施工進度.md` 的正式交付狀態切換
- 沙盒回流到主線
- 任何跨專案 / 跨執行單元的寫入
- 最終交付相關狀態切換

### 禁止

- 自主跨域修改其他專案
- 自主跨域修改其他執行單元
- 繞過第二層的變更分級與閘門

## 安全規則

1. **Harness 不發明新規則** — 只調用第二層指令，不繞過
2. **Harness 不繞過閘門** — 閘門是寫死的，Harness 也要停
3. **Harness 不繞過變更分級** — 大改一律問
4. **Harness 可被隨時中斷** — 使用者說停就停
5. **Harness 留完整日誌** — 每個自主決策記錄在 10_對話整理.md
6. **每輪只做一個最小可驗證步驟** — 不跨階段連做
7. **連續自主推進不超過 5 輪不報告** — 避免失控
8. **推進失敗計數器 3 次必停** — 避免空轉
9. **介入停頓計數器 3 次建議降級** — 避免 Harness 不適合當前任務

## 附加能力：專家召喚

專家召喚不是獨立推進模式，而是 Harness 在自主推進過程中，為了深度審查、專業判斷或單次分析而呼叫的附加能力。

**觸發方式：**
- 使用者口語指定（「叫 CEO 來看方案」）
- Harness 判斷當前步驟需要深度審查時主動建議
- 使用者確認後觸發

**特性：**
- 單次介入，完成即退場
- 不改變當前 Harness 主目標
- 不獲得額外權限
- Harness 推進中觸發後，完成即繼續推進

**也可以獨立使用：** 不開 Harness，直接說「叫 CEO 來看方案」，走單次專家介入→退場。
