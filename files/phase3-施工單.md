## 執行流程

### 階段 1：全量複盤（唯讀）

執行觸發口令：`sudo 系統修復`

對所有專案（含沙盒），逐容器檢查以下項目：

```
a. 01_團隊/ 人格卡是否為 placeholder 或 < 1000 bytes
b. 01_團隊/ 角色卡是否有空白欄位或 < 500 bytes
c. 人格卡是否含必備章節（metadata / 基底區 / 專案累積區）
d. 角色卡是否含必備章節（角色定位 / 職責 / 預設人格 / 本輪最終建議）
e. 正式專案檔是否包含 template_only 或 demo_only 標記
f. 資料夾命名是否符合 NN_ 前綴規則
g. 必備檔案是否齊全
   一般模式：10~15, 總方案, 資料地圖, 資料館長, 產出索引, 角色會議紀錄
   程式設計模式：上述 + 20/21/30/31, 04_主專案程式/
h. 總方案.md 是否有 專案模式 欄位
i. 參考資料是否有未登記到資料地圖的孤兒檔
j. 文件產出是否有未登記到產出索引的孤兒檔
k. 人格/角色卡 furnace_version 是否與 Furnace 一致
l. SRI 版本號是否全系統一致
```

產出 bug 報告，嚴重程度分級：
- 🔴 阻斷級：placeholder 殘留、正式專案含 template_only、人格/角色完全空殼
- 🟡 應修：缺檔、命名不符、版本不一致、孤兒檔、缺 metadata
- 🟢 建議：排序可優化、索引可補欄位

⚠️ 此階段不得寫入任何檔案。報告只在對話中呈現。

### 階段 2：經確認後修復

使用者看完 bug 報告後，下第二道口令：

```
根據 bug 修復報告執行修復
```

使用者可選擇：
- `全部修` — 依建議順序全部執行
- `逐一確認` — 每個修復動作先問再做
- `只修 🔴` — 只處理阻斷級問題
- `先不動` — 只保留報告，不修

**修復規則：**

1. 人格/角色卡修復 = 重新從 Furnace 配發（同 Phase 2 步驟 3/4）
2. 已有專案累積區內容的卡，只覆蓋基底區，保留專案累積區
3. 缺檔修復 = 從模板生成空白結構檔（非人格/角色）
4. 命名修復 = rename 資料夾或檔案，不改內容
5. 每個修改動作都要報告

**修復後驗證：**

修復完成後，再跑一次階段 1 的完整掃描，確認所有 🔴 已清除。

若使用者要求，將最終修復摘要寫入 `_state/system-log.md`。

## 最終完成標準

Phase 3 完成的定義：
1. 所有專案 🔴 阻斷級問題已清除
2. 修復後驗證掃描無新增 🔴
3. 使用者已確認修復報告
4. （可選）修復摘要已寫入 system-log.md
