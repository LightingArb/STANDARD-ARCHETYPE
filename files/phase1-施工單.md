### 步驟 2：修 `sri-core.md`

版本號從現有版本改為 `4.2.0`。

**2a. 在 §2-1A「角色帶依賴自動配發」段落末尾，新增以下內容：**

```markdown
⚠️ 配發完成硬性自檢（不可省略）：

### A. 數量檢查
□ 人格卡數量 = 該模式應配數量
  一般模式：6 張（#101, #102, #103, #104, #211, #213）
  程式設計模式：7 張（上述 + #202）
□ 角色卡數量 = 該模式應配數量
  一般模式：4 張（01-04）
  程式設計模式：7 張（01-04 + 11-13）
□ 角色會議紀錄.md 已建立

### B. 空殼防呆檢查
□ 每張人格卡 > 1000 bytes
□ 每張角色卡 > 500 bytes
□ 不含 "Template placeholder"
□ 不含 "應從 Furnace 配發"
□ 角色卡不含空白 "角色定位："
□ 角色卡不含空白 "本輪最終建議："

### C. 必備章節檢查
每張正式人格卡必須包含：
□ 檔頭 metadata（furnace_source / furnace_version / last_synced）
□ 基底區分隔線
□ 專案累積區分隔線

每張正式角色卡必須包含：
□ 角色定位（非空白）
□ 職責 或等價職責區段
□ 預設人格
□ 本輪最終建議（非空白）

### D. 來源型別檢查
□ 正式專案檔不含 "template_only: true"
□ 正式專案檔不含 "demo_only: true"

### E. 合格判定
任一項失敗 → 建立流程視為失敗，必須立即回報使用者，不進入下一步。
禁止用 template、demo 摘要版或空殼替代正式內容。
```

**2b. 新增「系統更新」段落（在 §2 指令設計的適當位置）：**

新增指令 `系統更新`，定義 A/B/C 三類檔案：

A 類（可覆蓋）：
```
✅ 專案管理/指令.md
✅ 專案管理/大絕招.md
✅ 專案管理/00_AI啟動入口.md
✅ 專案管理/_系統/基地/README.md
✅ 專案管理/_系統/基地/sa/*.md
✅ 專案管理/_系統/基地/mcp/*
✅ 專案管理/_系統/templates/*
✅ 專案管理/_系統/基地/系統總覽.md 僅以下章節：
  - ## 系統版本
  - ## 基地結構總表
  - ## 系統級文件索引
  - ## 版本一致性檢查
```

B 類（禁止）：
```
❌ 任何專案下的 總方案.md
❌ 任何專案下的 _系統/10~15, 20~31
❌ 任何專案下的 02_參考資料/*、03_文件產出/*、04_主專案程式/*、05_沙盒區/*
❌ 任何 01_團隊/ 的「專案累積區」
❌ 系統總覽.md 的 ## 當前運行狀態、## 使用者維護備註、## 專案現況摘要
```

C 類（需確認）：`01_團隊/` 人格/角色卡的基底區。

系統更新流程：覆蓋 A 類 → 比對 C 類產出刷新清單 → 掃描產出升級建議 → 問使用者確認 → 確認後才動 B/C 類。

**2c. 新增 `sudo 系統修復` 指令段落：**

觸發口令：`sudo 系統修復`（必須完整匹配，不接受縮寫或自然語句等價）

第 1 階段：唯讀全量複盤
- 掃描所有專案逐容器檢查（人格/角色卡大小、必備章節、命名規則、必備檔案、孤兒檔、版本一致性）
- 產出 bug 報告（🔴 阻斷 / 🟡 應修 / 🟢 建議）
- ⚠️ 不得寫入任何檔案，報告只在對話中

第 2 階段：需使用者下 `根據 bug 修復報告執行修復` 才動手
- 可選全修 / 逐一 / 只修🔴 / 不動
- 每個動作都報告
- 修完再跑一次第 1 階段驗證

**2d. 新增排序規則段落：**

```
02_參考資料/：
- 資料地圖.md 必須包含：編號、日期、狀態、讀取優先序、是否主線引用
- 新增參考資料必須同步登記到資料地圖；未登記視為不合法

```

```markdown
# AI 啟動入口

> 本檔是 AI 的 bootloader，不是說明書。每次新對話必須先讀本檔。

## 規則 1：開始任何操作前，先讀

- `_系統/基地/sa/sri-core.md`
- `_系統/基地/sa/sri-ai-behavior.md`

## 規則 2：建立新專案時

## 規則 3：建立完成前必須自檢

（完整自檢清單見 sri-core.md §2-1A）

快速版：
- 所有 `人格_*.md` > 1000 bytes 且含必備章節

- 正式專案檔不得包含 `template_only` 或 `demo_only`

## 規則 4：禁止事項

- 禁止從 `_系統/templates/` 複製人格/角色卡內容到正式專案
- 禁止在正式專案使用 demo 摘要版人格
- 禁止不讀規則就建立資料夾或檔案
- 禁止以語義推定「這段看起來像系統區」就自行修改未列入白名單的內容
```
